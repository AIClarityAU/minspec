# PR #790 remediation — wire the harness-refresh commit recovery affordance

## Root cause

`44b1269` (feat #758) shipped the *library layer* for a recoverable
"harness uncommitted" affordance — `commitHarnessRefreshCommand`,
`collectDirtyScaffoldPaths`, and `MinSpecScaffoldCommitStatusBar` — but never
wired it into `extension.ts` or `package.json`. `MinSpecScaffoldCommitStatusBar`
was never instantiated, `minspec.commitHarnessRefresh` was never registered as
a VS Code command, and it had no `contributes.commands` entry, so the status
bar item could never appear and clicking it (or invoking it from the palette)
would resolve to a nonexistent command. All three independent reviewers
(dev/architect/skeptic) flagged this as blocking dead code.

## Fix

`packages/minspec/src/extension.ts`:
- Import `commitHarnessRefreshCommand`, `collectDirtyScaffoldPaths` from
  `./commands/init`, and `MinSpecScaffoldCommitStatusBar` from
  `./views/status-bar`.
- Instantiate `scaffoldCommitStatusBar` alongside the existing status bars in
  `activate()`, with a debounced `refreshScaffoldCommitStatusBar()` helper
  (same 300ms-debounce shape as the existing next-task signpost) that polls
  `collectDirtyScaffoldPaths(workspaceRoot)` and calls `update()`.
- Register `minspec.commitHarnessRefresh`, wrapping
  `commitHarnessRefreshCommand` and refreshing the status bar afterward so it
  hides once the offer is accepted (or reports nothing to commit).
- Wrap the existing `minspec.init` / `minspec.initRefresh` registrations so
  they also refresh the status bar right after scaffolding/refreshing — the
  moment the managed files can actually go dirty — instead of only picking up
  the new state on next activation.
- Push `scaffoldCommitStatusBar` into `context.subscriptions` for disposal.

`packages/minspec/package.json`: added the `minspec.commitHarnessRefresh`
`contributes.commands` entry ("MinSpec: Commit Harness Refresh"), making it
palette-invokable as the (now-true) doc comments already claimed.

Left out (deliberately, to keep the change minimal): an always-on
`.git/HEAD` watcher to catch commits made outside VS Code. That would also
have moved the recovery status bar out of sync with the rest of the file —
`extension.test.ts` has an exact-count assertion
(`createFileSystemWatcher` called 5 times) that a new unconditional watcher
would break, and every other git-plumbing watcher in this file is opt-in
(`autoClassifyOnCommit`). The command/init/refresh hooks already cover the
actual #758 scenario (a missed toast after scaffold/refresh).

## Tests

- `packages/minspec/tests/extension.test.ts` /
  `extension-extra.test.ts`: added `MinSpecScaffoldCommitStatusBar` +
  `commitHarnessRefreshCommand` / `collectDirtyScaffoldPaths` to the existing
  mock scaffolding (both files instantiate the real `extension.ts`, and the
  real class would otherwise call the unmocked `vscode.window.createStatusBarItem`
  and throw). Added assertions that the command is registered, wired to
  `commitHarnessRefreshCommand`, and that the status bar is constructed and
  pushed for disposal.
