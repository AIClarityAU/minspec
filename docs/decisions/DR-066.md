# DR-066: Audience-partitioned approvables (role-based review split)

> **Born `proposed` per DR-029** — acceptance is a separate human act (*MinSpec: Accept ADR*).
> This DR records context, the decision, and its follow-ups; no code ships with it.

## Context

Specs interleave audiences in one project. The **what/why** — functional requirements,
acceptance criteria, goals, open questions — is product-owner (PO) territory. The **how** —
contracts, invariants, resolved design decisions, test/verification items, non-functional
constraints — is engineering territory. A non-technical PO who must review and sign off on
the business intent currently has to read past engineering content to find the parts that
are theirs, and cannot approve at all without the VS Code extension.

Current state (verified 2026-07-21):

- **No audience/owner/assignee/role primitive exists** anywhere — not on a spec, not on a
  section, not on a paragraph. Human identity enters only as `approvedBy` (a
  `git config user.email` on an `ApprovalRecord`), used purely to prove *a human, not a
  bot,* approved (DR-056). It is provenance, not assignment.
- **Approval is per-spec-file**, path-keyed: one committed sidecar per artifact at
  `.minspec/approvals/<repo-relative-spec-path>.json`, binding the file's canonical content
  hash (SPEC-022 FR-1..FR-4, `packages/minspec/src/lib/approval.ts:55-64`).
  "Single-approver-suffices" clears the file for the team (SPEC-022 FR-2). There is **no
  sub-file unit** an approval can attach to.
- **The split-file layout already exists**: specs may be authored as `requirements.md` /
  `design.md` / `tasks.md`. `requirements.md` is, by construction, the PO-facing artifact.
- **A jargon audit (2026-07-21)** of ~379 PO-facing paragraphs across 15 specs found ≈96%
  contain developer jargon (see R1) — so *which file* the PO reads is only half the problem;
  *how it is written* is the other half.
- **Deterministic machine gates already exist and are approval-independent**: frontmatter
  `validate`, the consequence/blast-radius screen (SPEC-023), and auto-merge eligibility
  (SPEC-024). These run regardless of who (or whether a human) approves.
- **The review webview is specced but not built**: SPEC-014 ("Prettified Spec-Review
  Webview", `status: implementing`) designs a rendered-markdown review surface; the only
  `WebviewPanel` shipping today is the lightweight `SpecPanel` stepper
  (`packages/minspec/src/views/spec-panel.ts`), which renders no prose and has no approve
  action. A VS Code webview is desktop-bound — it cannot run on a phone.

The request and same-session decisions: make the audience split a **universal** capability
(all projects; existing minspec specs need **not** be backfilled); let a non-technical PO
review and approve **just their subset** from a browser; **keep PO and engineering content
in separate files** (so no sub-file projection is needed); serve **all committers** on
mobile (a dev must be able to approve from their phone); design the model to **extend to
further roles** in the teams version (UX designer, marketing copy/proofreader, BA/producer);
and provide a **PO-only mode** that auto-approves technical approvables for solo devs who
want to own only the what/why.

## Decision

Partition approvables by **audience**, where an audience is a class of *content* mapped to
whole *files* and to a *role* that approves them. Four decisions:

### 1. Audience = file. Separate files per audience; no sub-file projection.

Each audience owns whole files. PO and engineering content live in **separate files** by
authoring discipline — `requirements.md` = PO; `design.md` / `tasks.md` = engineering — and
a spec MUST NOT mix audiences within one file. This is the whole mechanism: because
per-file approval sidecars already make each file its own hash-bound approval unit
(SPEC-022), a PO approving `requirements.md` is already a distinct approval from the
engineering files, with **zero new parsing, no new frontmatter field, and no corpus
migration**.

**Sub-file projection ("Phase C" — deriving audience from DR-053 paragraph-type codes
inside one mixed file) is dropped, not deferred.** Mandating separate files is simpler,
removes the hard dependency on DR-053 + #679, keeps approval at the file granularity that
already exists, and — decisively — matches the granularity of every downstream surface:
GitHub review, CODEOWNERS, and the approval sidecar are all **per-file**. The bespoke
"projection webview" that existed only to serve Phase C is therefore **dropped** as well.

### 2. Role-extensible from the start (teams-ready).

The audience→file→role map is an **N-role table**, not a hardcoded PO/engineering binary.
Additional team roles slot in as more audience files + CODEOWNERS globs — e.g. UX designer
→ `ux.md`, marketing copywriter/proofreader → `copy.md`, BA/producer → `research.md` —
with no new mechanism. File-separation (decision 1) is precisely what makes this tractable:
a role owns file globs. The teams version consumes this table; solo/OSS projects use the
default PO/engineering rows.

### 3. Approval ingress — multi-surface and audience-agnostic.

Approving is not PO-only, and the *surface* is separate from the *partition*: the partition
decides who is routed to and responsible for which files; the ingress is how any approver —
dev or PO — records an approval. Two ingresses land the **same** on-disk `ApprovalRecord`:

- **Local (offline, Tier-0):** the existing `MinSpec: Approve` command / SPEC-014 webview,
  inside VS Code. Rich and offline — but **desktop-bound**; it does not solve mobile for
  anyone.
- **Networked (anywhere, incl. mobile) — GitHub-native:** GitHub's own app / mobile-web is
  already a review-and-approve surface, closing the mobile gap for **all committers**. A dev
  approves `design.md`/`tasks.md` from their phone exactly as a PO approves
  `requirements.md`:
  - **Route by role with `CODEOWNERS`** — `**/requirements.md` → PO team, `**/design.md` /
    `**/tasks.md` → engineering (and future role globs). Branch protection's *require review
    from Code Owners* makes the right role a **required reviewer per file**.
  - **Read** = GitHub's native markdown render (the "pretty page" already exists);
    **approve** = the native *Approve* button (or a `/approve` PR comment), desktop **or
    mobile**.
  - **Materialise the record.** A GitHub Action on `pull_request_review` (approved) verifies
    the reviewer's team, recomputes the canonical hash (SPEC-022 FR-3), and writes the
    committed `ApprovalRecord` sidecar attributing the **human** approver (verified GitHub
    identity — the Action is a scribe, so DR-056's human-not-bot gate still passes on
    `approvedBy`).

GitHub is an **optional, consent-gated ingress, never the only path** — the offline
`MinSpec: Approve` still writes the identical record with no network, so the invariant
holds. No bespoke *networked* webview is needed: GitHub already is the anywhere/mobile
surface for every role.

**PRs are not split by audience by default.** Because approval is already per-file (sidecar
+ CODEOWNERS required review), one PR may touch several audiences' files and each role
approves only its own — splitting PRs would buy only *visual* hiding at the cost of change
atomicity. Audience-split PRs (a PO PR touching only `requirements.md`) are an **opt-in**
for the strict case where the PO must not even see the engineering diff. (Evaluation in
*Rationale* / R6.)

### 4. PO-only mode — auto-approve technical approvables, honestly.

A per-project setting declares, per audience, whether its approvables need a **human**
approver or are **auto-approved by policy**. Primary use: a solo dev upgrading from
vibe-coding who wants to be *only* the PO — engineering approvables auto-pass so they are
not forced to review the *how*. Config lives in `.minspec/config.json` (offline, Tier-0),
per audience: `approval: "human" | "auto"`. This generalises to the teams version (each
role independently human/auto/delegated).

Three hard constraints, because this touches the never-wrong signpost and DR-056:

- **Honesty.** An auto-approved approvable is recorded and displayed as **auto-approved by
  policy** — a distinct, explicitly-typed record — **never** as a human approval and
  **never** via a forged `approvedBy`. A false "a human reviewed this" is the worst defect
  in a never-wrong product (DR-003 / EPIC-002).
- **DR-056 is not laundered.** Auto-approval does **not** route through the human approver
  path (which would throw on a non-human identity, or worse, disguise a machine approval as
  human). The gate admits an auto record only under an explicit, first-class policy branch;
  provenance names the policy and the human who enabled it — who **owns the outcome**.
- **Waives review, not the gates.** Auto-approve waives *human review*, not the
  deterministic machine gates. `validate`, the SPEC-023 consequence screen, and SPEC-024
  auto-merge eligibility still run on auto-approved approvables — the solo PO-dev keeps the
  machine safety net; they have only opted out of eyeballing the engineering content.

## Rationale

- **Reuse over invention.** The mechanism rides structure that already exists — the split
  layout and per-file approval sidecars — so a PO owns a real, hash-bound approval unit
  *today* with near-zero new machinery.
- **Structure beats a declared field.** Audience is *where the content lives* (the file),
  which the author already decides — not a separate metadata field that rots and needs a
  gate to stay honest (DR-003 sibling rule).
- **Match the downstream granularity.** GitHub, CODEOWNERS, and the sidecar are all
  per-file; file-separation is what lets native GitHub approval work at all, and it is why
  sub-file projection (Phase C) is not worth its DR-053 dependency and complexity.
- **Split-PRs buy only visual hiding.** Since the PO's approval is already scoped to
  `requirements.md`, a single PR with per-file required reviews gives correct per-role
  gating without breaking the atomicity of a change that naturally spans requirements +
  design + tasks. Reserve splitting for when *seeing* the eng diff is itself the problem.
- **Auto-approve is an on-ramp, not a bypass.** Letting a solo dev be PO-only lowers the
  barrier to adopting SDD from vibe-coding, while the machine gates keep correctness — so
  the ceremony they skip is human review of the *how*, not the safety net.

## Consequences

- **+** A non-technical PO owns and signs the business subset (their file) without reading
  engineering content, and can do it from a browser or phone.
- **+** Ships on existing plumbing — per-file sidecars + CODEOWNERS + a thin Action; no
  corpus migration, no DR-053 dependency.
- **+** The networked ingress serves **all committers**: **devs gain mobile/web approval
  they lack today** (a VS Code webview can't run on a phone), not just POs.
- **+** Role-extensible: UX/marketing/BA roles slot in as files + CODEOWNERS globs with no
  new mechanism (teams version).
- **+** PO-only mode is a clean adoption on-ramp for solo devs upgrading from vibe-coding.
- **±** Routing adopts a **GitHub-team role gate** (CODEOWNERS), deliberately **reversing**
  SPEC-022 FR-2's "any committer may approve," scoped to the GitHub ingress. Provenance
  strengthens (verified identity vs self-asserted git email) but the trust root now includes
  a networked writer Action (EPIC-006 supply-chain surface).
- **−** Mandating separate files per audience makes **single-file mixed specs
  non-conformant** for new work; needs a warn-first authoring gate (a T1 one-liner should
  not be forced into three files — small specs need an exemption).
- **−** Optional audience-split PRs trade change **atomicity** for visual hiding (R6).
- **−** Auto-approve adds a **new record type and a new gate branch** that must be got right
  (honesty + DR-056), or the signpost lies (R2).
- **−** The PO view is only as accessible as the authoring — ≈96% of today's PO-facing
  paragraphs carry jargon (R1); file-separation does not fix wording.

## Risks & Mitigations

- **R1 — Jargon leak makes the PO's file unreadable even when correctly separated — the
  dominant risk.** A 2026-07-21 audit of ~379 PO-facing paragraphs (FR/AC/OQ/G/M/AS/R)
  across 15 specs found **≈96% (364/379) contain developer/CS jargon**; 7 specs scored 100%.
  Density is worst in exactly the PO types — FR and AC bullets inline function signatures,
  `file:line` citations, CLI flags, git/CI mechanics. **Separating files does not fix
  authoring.** **Mitigation:** the authoring-accessibility guideline + jargon lint (flagging
  jargon in PO-facing files at author time) is a **precondition** for value, not a nicety.
- **R2 — Auto-approve masquerade.** If an auto-approval were recorded as a human approval,
  the signpost would lie and DR-056 would be laundered. **Mitigation:** a distinct,
  explicitly-typed "auto-approved by policy" record; the gate admits it only under an
  explicit policy branch, never the human path; provenance names the policy + the human who
  enabled it; the UI shows "auto-approved (no <role> approver)."
- **R3 — Network invariant.** "Core functionality works offline — no network calls without
  explicit user consent." **Mitigation:** GitHub is an *optional, consent-gated ingress*;
  the offline `MinSpec: Approve` writes the identical sidecar with no network, and
  auto-approve is a local policy — the core partition + approval mechanics contain **no**
  network calls.
- **R4 — Map/policy versioning.** If the audience→file→role map or a per-audience approval
  policy changes after approvals exist, prior approvals could silently become wrong.
  **Mitigation:** version the map and the policy, and pin the version into the approval
  record (see *Costly to Refactor*).
- **R5 — Over-invalidation of approvals.** GitHub dismisses stale reviews on *any* push;
  SPEC-022's canonical hash ignores lifecycle-field-only edits. **Mitigation:** the committed
  sidecar hash stays the arbiter of freshness — the GitHub review is the *trigger*, the
  canonical file hash *decides* whether the approval holds.
- **R6 — Split-PR atomicity/drift.** Opt-in audience-split PRs can land requirements/design/
  tasks out of sync on `main` (a merged FR with no design; design.md referencing an FR id
  not yet merged). **Mitigation:** default to one PR + per-file required reviews (atomic);
  when a project opts into splitting, gate cross-file coherence so an orphaned half is
  flagged.
- **R7 — Auto-approve hides real regressions.** Auto-approving all technical approvables
  means no human reviews the *how* — for a solo dev that is their explicit choice, but it
  removes the engineering review gate. **Mitigation:** keep the deterministic machine gates
  (`validate`, SPEC-023, SPEC-024) running on auto-approved approvables; auto-approve waives
  *human review*, not the machine safety net.

## Alternatives considered

- **Sub-file projection (former "Phase C") — derive audience from DR-053 paragraph-type
  codes within one mixed file, and project a filtered PO view.** Rejected in favour of
  file-separation: it adds a hard DR-053 + #679 dependency, needs a new paragraph-projection
  hash unit, and fights the per-file granularity of GitHub/CODEOWNERS/sidecars. Elegant, but
  more machinery for a split that separate files give for free.
- **Mandatory audience-split PRs.** Rejected in favour of opt-in: because approval is
  already file-scoped, mandating splits pays an atomicity + coordination cost for a purely
  presentational gain (R6).
- **Auto-approve by writing a normal `ApprovalRecord` with a synthetic/service identity.**
  Rejected: it launders DR-056 and makes the signpost claim a human reviewed what none did.
  Auto-approval must be its own honest record type (decision 4, R2).
- **Declared whole-spec `audience:` frontmatter (option A) / per-paragraph declared
  `audience` (option D).** Coarse or rot-prone; a declared field needs a gate to stay
  honest. Superseded by file-separation, which encodes audience in structure the author
  already chooses. Option D retained only as a far-future escape hatch, not part of this DR.
- **Approve the whole spec but hide technical files in the UI only (no approval change).**
  The PO's signature would attest to engineering content they never saw — a signpost that
  lies. Rejected; the approval unit must match what was shown (which per-file approval gives).

## Costly to Refactor

- **The audience→file→role map and the per-audience approval policy are load-bearing once
  approvals (human or auto) exist against them.** Changing them retroactively changes what a
  past approval meant. Both are **versioned**, and the version is pinned into the
  `ApprovalRecord` (extending SPEC-022's record); a change invalidates affected approvals and
  re-requests them rather than mutating meaning in place.
- **The auto-approved record type is a public contract** the moment specs are auto-approved
  under it; its shape and provenance semantics must be fixed before first use, like SPEC-022
  FR-3's whole-file hash.
