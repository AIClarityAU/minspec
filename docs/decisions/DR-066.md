# DR-066: No silent gate

> **Born / stays `proposed` per DR-029** — acceptance is a separate human act
> (*MinSpec: Accept ADR*). The **constitution amendment in §5 is a proposal to
> redline at Accept-ADR**; it is NOT applied to `.minspec/constitution.md` until
> this DR is accepted. (Note: the ext's Accept does not yet auto-apply §5 — #768 —
> so the redline is applied by hand at Accept.)

## Context

Three times in this repo, a merge gate that *looked* present enforced **nothing**, and the
symptom each time was identical — "every merge needs `--admin`", i.e. the required gate was
being bypassed on every landing:

1. **#560** — the `ai-review` required-check context was pinned to the **wrong GitHub App id**,
   so the ruleset waited on a check that could never post → unsatisfiable → every merge a bypass.
2. **#810** — `ai-review.yml` posted the load-bearing `ai-review/pass` commit status
   **best-effort** (`gh api … || true`). The POST `403`'d because the App lacked
   `statuses: write`; `|| true` swallowed it, so the `ai-review:pass` **label** was applied
   while the SHA-bound **witness** never existed. `ready-to-merge` (single-witness) then read
   `failure` on every genuinely-passing PR → unsatisfiable → every merge a bypass.
3. **#857** — the `bumblebee` supply-chain scan (DR-005, the pre-publish gate) has been **red
   on `main` for days** (catalog schema `0.2.0` vs a binary supporting `0.1.0`). Because it is
   **not** a required context, it failed *silently and non-blockingly* — the gate appears
   present but performs zero scanning.

The shared mechanism across all three: **a load-bearing signal that could fail without the
failure being visible at the gate.** Best-effort writes (`|| true`) hide producer errors; a
single producer makes one permission/config gap fatal instead of degrading; a non-required or
mis-pinned context lets a red (or absent) gate read as "fine". In a **never-wrong** product,
a gate that can silently stop enforcing is the worst possible defect — it makes the signpost
lie by omission. This is the same family as the *artifact-existence ≠ feature-existence* rule
(DR-003 evidence discipline), applied to gates: **gate-presence ≠ gate-enforcement.**

## Decision

Adopt a new constitution **Invariant** (§5): **No silent gate.** A required-check or
merge-gating signal must fail **visibly** and never be produced best-effort. Concretely, every
gate signal must satisfy all three:

1. **No best-effort load-bearing write.** A signal a gate depends on is NEVER written with a
   swallowed error (`|| true`, ignored non-zero exit, fire-and-forget) on its load-bearing
   path. If the write can fail, the failure must surface.
2. **Absence fails closed AND visible.** When a gate's witness is missing or its producer
   errors, the gate must read red / blocked — never silently pass, and never silently *stop
   evaluating*. A gate that is "green because it didn't run" is prohibited.
3. **No single disableable producer.** If a required check's sole witness can be disabled by
   one permission, id-pin, or config gap, it must accept an **independent second witness** (or
   otherwise degrade rather than become unsatisfiable), so one gap *degrades* the gate instead
   of silently killing it.

This is the generalisation of the fixes already shipped: DR-063/#854 gave `ready-to-merge` a
second, independent witness (the `ai-review` check-run OR the `ai-review/pass` status), which
is clause 3 applied to that gate. This DR makes the rule general so the *next* gate is built
right by construction, not patched after the third incident.

**Enforcement (how the Invariant becomes deterministic, not a hope — Principle
"Don't hope an LLM will follow rules, enforce it via code"):** a CI/lint check that (a) flags
`|| true` / swallowed errors on any line that posts a commit status, check-run, or gate label,
and (b) asserts every **required** ruleset context is backed by ≥2 independent producers or an
explicit fail-closed-and-visible annotation. Tracked as a follow-up (below) — the Invariant is
the contract; the linter is its write-time teeth.

## §5 — Proposed constitution amendment (redline at Accept-ADR)

Add one **Invariant** (an absolute never-break rule — unlike DR-055's Principle, this has no
recorded exceptions: a silent gate is never acceptable in a never-wrong product).

Proposed addition to `## Invariants`:

```
- No silent gate — a required or merge-gating check fails visibly, never best-effort:
  no load-bearing gate signal is written with a swallowed error (`|| true`), a missing or
  errored witness fails the gate closed and visibly (never silently passes or stops
  evaluating), and no required check hinges on a single producer that one permission/config
  gap can disable (provide an independent second witness). (DR-066; instances #560/#810/#857.)
```

**Open call for the human at Accept — Invariant vs Principle.** Recorded recommendation:
**Invariant.** A Principle is directional and admits exceptions (DR-055 chose Principle
*because* conformity has recorded §3 exceptions); a silent gate has none — it is always a
defect. If on reflection you want to allow scoped exceptions (e.g. a genuinely advisory,
clearly-labelled non-gating check), demote to a Principle at Accept and this DR's §5 moves to
`## Principles` unchanged in wording.

## Consequences

- New gates are built with a verifiable, fail-visible, multi-witness signal from day one; the
  "third time's the pattern" class is closed by construction, not by a fourth incident.
- Existing gates get audited against the three clauses. Known debt to reconcile: #811
  (`MinSpec SDD validation` required check is a fake-green stub — violates clause 2), #857
  (bumblebee silently red + non-required — clauses 2/3), and any remaining `|| true` on a
  status/label/check post.
- Slightly more ceremony per gate (a second witness or an explicit fail-closed annotation).
  Accepted: the cost of a silent gate is "every merge is an unreviewed bypass", which dwarfs it.
- No change to what the gates *decide* — only to how their signals are produced and read.

## Alternatives considered

- **Leave it as a per-incident fix (status quo).** Rejected — three incidents of the identical
  class is the definition of a missing invariant; patching the fourth after the fact is the
  anti-pattern this repo exists to kill.
- **Encode as a Principle, not an Invariant.** Viable (see §5 open call), but a Principle reads
  as "directional, exceptions allowed"; a silent gate is never acceptable, so Invariant is the
  honest register. Left as the human's call at Accept.
- **Ban `|| true` outright, everywhere.** Too broad — `|| true` is legitimate on genuinely
  non-load-bearing best-effort side-effects (audit comments, label cosmetics). The Invariant is
  scoped to signals a gate *depends on*, which is the actual failure surface.

## Costly to Refactor

**Moderate.** The Invariant itself is cheap to state; the enforcing linter + the audit/repair of
existing gates (#811, #857) is real work, but each piece is independently shippable and
reversible. Recorded as a DR because it changes the constitution (a hard-to-reverse governance
act), not because the code is hard to unwind.
