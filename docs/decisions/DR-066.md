# DR-066: Audience-partitioned approvables (PO-facing review split)

> **Born `proposed` per DR-029** — acceptance is a separate human act (*MinSpec: Accept ADR*).
> This DR records context, the decision, and its follow-ups; no code ships with it.

## Context

Specs interleave two audiences in one document. The **what/why** — functional
requirements, acceptance criteria, goals, open questions — is product-owner (PO)
territory. The **how** — contracts, invariants, resolved design decisions,
test/verification items, non-functional constraints — is engineering territory. A
non-technical PO who needs to review and sign off on the business intent currently has
to read past engineering paragraphs to find the parts that are theirs.

Current state (verified 2026-07-21):

- **No audience/owner/assignee/role primitive exists** anywhere — not on a spec, not on
  a section, not on a paragraph. Human identity enters only as `approvedBy` (a
  `git config user.email` on an `ApprovalRecord`), used purely to prove *a human, not a
  bot,* approved (DR-056). It is provenance, not assignment.
- **Approval is per-spec-file**, path-keyed: one committed sidecar per artifact at
  `.minspec/approvals/<repo-relative-spec-path>.json`, binding the file's canonical
  content hash (SPEC-022 FR-1..FR-4, `packages/minspec/src/lib/approval.ts:55-64`).
  "Single-approver-suffices" clears the file for the team (SPEC-022 FR-2). There is **no
  sub-file unit** an approval can attach to today.
- **The split-file layout already exists**: specs may be authored as
  `requirements.md` / `design.md` / `tasks.md`. `requirements.md` is, by construction,
  the PO-facing artifact.
- **Paragraph type codes exist only on paper**: DR-053 v2 defines a typed-paragraph
  grammar (`FR OQ AC G M AS R` = business-facing; `NFR CT RD INV CR TV FM AL DP` =
  engineering-facing) but it is `proposed`, not parsed, and the shared `project-prefix`
  resolver does not yet understand paragraph segments (blocked on #679).
- **The review webview is specced but not built**: SPEC-014 ("Prettified Spec-Review
  Webview", `status: implementing`) designs a rendered-markdown review surface with a
  gate-state header, comment pins, and a scroll-bottom Approve that runs the DR-012 gate
  verbatim. The only `WebviewPanel` that ships today is the lightweight `SpecPanel`
  stepper (`packages/minspec/src/views/spec-panel.ts`), which renders no prose and has no
  approve action. The SPEC-014 design is highly reusable for a PO surface, but it is not
  yet implemented.

The request: make the audience split a **universal** capability (all projects, not an
opt-in) — with the explicit clarification that existing minspec specs need **not** be
backfilled — and give a non-technical PO a web page to review and approve **just their
subset**, directly from and to GitHub, hiding the technical detail, reusing the SPEC-014
webview design where possible.

## Decision

Introduce **audience** as a partition over approvables, in two phases. Audience is a
property of *content type*, never of a person — role/type-based, not a CODEOWNERS
assignment.

**Phase B — now (file-boundary audience, zero new parsing).**
The audience boundary is the split-file boundary. `requirements.md` is the **PO-facing
approvable**; `design.md` and `tasks.md` are **engineering-facing**. This ships today
because per-file approval sidecars already give each split file its own approval unit
(SPEC-022) — a PO approving `requirements.md` is already a distinct, hash-bound record
from the design/tasks approval. No new frontmatter field, no corpus migration.

**Phase C — later (audience derived from paragraph type).**
Once DR-053 makes paragraph types machine-readable (hard predecessor: #679), audience
becomes a **derived** function of the type code via a default map:

| Audience | Paragraph type codes |
|---|---|
| **product-owner** | `FR` `AC` `OQ` `G` `M` `AS` `R` |
| **engineering** | `NFR` `CT` `RD` `INV` `CR` `TV` `FM` `AL` `DP` |

Derived — not a declared, hand-maintained field — so it cannot drift and needs no gate
to stay honest (aligned with RCDD / DR-003, and with the classifier's signal-derived
philosophy). A single mixed document then projects a **filtered PO view** containing only
PO-facing paragraphs. A per-instance `audience:` override on an individual paragraph is
allowed only as an escape hatch over the derived default (option D as a lever, never the
primary mechanism).

**Universal, not opt-in.** The capability is on in every project. The audience→type map
ships with a sensible default and is *overridable* per project (a project may reclassify
a type), but there is no enable flag. "No backfill" scopes only the corpus: the split
applies to all **new** work everywhere; existing minspec specs are not retro-fitted.

**Approval partition.** A PO's approval attaches to the PO-facing unit:
- Phase B: the existing `requirements.md` sidecar hash.
- Phase C: a **paragraph-projection hash** — SPEC-022 FR-3's canonical hash, computed
  over only the PO-facing paragraphs (the PO view). Engineering approval covers the
  remainder. "Single-approver-suffices" generalises to **one approver per audience**.

**PO approval ingress — GitHub-native first (Phase B).** Do not build a bespoke surface
for the PO. Use GitHub itself as the review-and-approve page, because it maps onto Phase
B's file boundary with (almost) no custom code:

- **Route by audience with `CODEOWNERS`.** Map `**/requirements.md` → the product-owner
  team, `**/design.md` / `**/tasks.md` → engineering. Branch protection's *require review
  from Code Owners* then makes a PO a **required reviewer** on any PR touching a
  requirements file — GitHub-native audience routing that matches Phase B exactly.
- **Read** = GitHub's native markdown rendering of `requirements.md` (the "pretty page"
  already exists); **approve** = the PO clicks GitHub's native *Approve* (or a `/approve`
  PR comment).
- **Materialise the record.** A GitHub Action on the `pull_request_review` (approved)
  event verifies PO-team membership, recomputes the canonical `requirements.md` hash
  (SPEC-022 FR-3), and writes the committed `ApprovalRecord` sidecar attributing the
  **human** PO (their verified GitHub identity — the Action is a scribe, not the
  approver, so DR-056's human-not-bot gate still passes on `approvedBy`). The on-disk
  truth is unchanged; the offline `MinSpec: Approve` command still writes the identical
  record. GitHub is therefore an **optional, consent-gated ingress, never the only path**
  — the offline invariant holds (core approval works with no network; GitHub is the
  consented convenience).

This replaces the bespoke networked webview for the PO in Phase B: a surface the PO
already knows, OAuth-verified identity, notifications for free, nothing to build. A
bespoke **projection webview is retained only as the Phase-C fallback**, because GitHub's
granularity is per-file — it cannot route to or hide a paragraph *subset within one mixed
file*. That limit is a signal, not a defect: keeping PO and engineering paragraphs in
**separate files** (the Phase-B discipline) is what makes the whole approach work
natively, and to truly hide technical detail a spec change should split into an
audience-scoped PR per file (a PO PR touching only `requirements.md`).

## Rationale

- **Reuse over invention.** Phase B rides structure that already exists — the split
  layout and per-file approval sidecars — so the PO can own a real approval unit *today*
  with near-zero new machinery.
- **Derived beats declared.** A hand-maintained `audience:` field on every paragraph
  would rot, and per DR-003's sibling rule an un-gated field lies. Deriving audience from
  a type the author already writes keeps it truthful for free.
- **Type already encodes audience.** The DR-053 paragraph taxonomy sorts almost perfectly
  by audience; we are surfacing a partition that is latent, not inventing one.
- **Role, not person.** SPEC-022 FR-2 deliberately deferred authority gating; keeping
  audience type-based avoids re-opening the CODEOWNERS question while still routing the
  right subset to a PO.

## Consequences

- **+** A non-technical PO can own and sign the business subset without reading
  engineering paragraphs.
- **+** Phase B ships on existing plumbing; no migration of the current corpus.
- **+** Derived audience (Phase C) cannot drift; no new gate needed to keep it honest.
- **−** Phase C is blocked on DR-053 acceptance and #679 (paragraph-addressable resolver).
- **−** Approving a *view* (a paragraph subset) requires a **new hash unit** — a
  canonical projection over PO-facing paragraphs — extending SPEC-022 FR-3.
- **−** The PO web-approval page reads/writes GitHub: it is networked, so it sits outside
  the Tier-0 offline core and must be explicit-consent-gated.
- **−** The PO view is only as accessible as the authoring. Today's PO-facing paragraphs
  already carry engineering jargon (≈96% — see R1), so a filtered view alone does not
  guarantee a readable one.
- **+** For Phase B the PO's "web page" is **GitHub itself** — nothing to build,
  OAuth-verified identity, native per-file review, notifications for free.
- **±** GitHub-native PO approval adopts a **GitHub-team role gate** (CODEOWNERS), which
  deliberately **reverses** SPEC-022 FR-2's "any committer may approve", scoped to the
  GitHub ingress only. Provenance strengthens (verified identity vs self-asserted git
  email) but the trust root now includes a networked writer Action (EPIC-006
  supply-chain surface).

## Risks & Mitigations

- **R1 — Jargon leak makes the PO view unreadable even when correctly filtered — the
  dominant risk.** A 2026-07-21 audit of ~379 PO-facing paragraphs (FR/AC/OQ/G/M/AS/R)
  across 15 specs found **≈96% (364/379) contain developer/CS jargon** a non-technical PO
  could not parse; 7 specs scored 100% (zero clean PO-facing paragraphs). Density is worst
  in exactly the PO types — FR and AC bullets routinely inline function signatures,
  `file:line` citations, CLI flags, and git/CI mechanics. Most-pervasive terms: tier
  notation (`T0`–`T4`), bare file paths / script names, `code identifiers()`,
  `frontmatter`, `gate`/`validator`, `CI`, git internals, content-`hash`. **So filtering
  by audience does not, on its own, produce a readable PO view — the authoring is the
  bottleneck.** **Mitigation:** the follow-up authoring-accessibility guideline + jargon
  lint is a **precondition** for this DR to deliver value, not an optional nicety; the
  lint flags jargon in PO-facing paragraph types at author time.
- **R2 — Partial-approval hash unit does not exist yet.** Phase C's PO-view approval needs
  a canonical hash over a paragraph subset, which presumes paragraph-addressable ids.
  **Mitigation:** Phase B uses whole-file (`requirements.md`) granularity until DR-053 +
  #679 land; Phase C's projection hash is specced as an explicit extension of SPEC-022
  FR-3, versioned so a change to the audience map invalidates and re-requests approval
  rather than silently drifting.
- **R3 — Network invariant.** The one hard invariant is "core functionality works offline
  — no network calls without explicit user consent." A GitHub-connected PO approval path
  violates that if made mandatory in the offline core. **Mitigation:** GitHub is an
  *optional, consent-gated ingress*; the local `MinSpec: Approve` command still writes the
  identical sidecar with no network, so core approval works offline and the
  audience-partition mechanics (Phases B/C) contain **no** network calls in the Tier-0
  core.
- **R5 — Over-invalidation of PO approvals.** GitHub dismisses stale reviews on *any*
  push; SPEC-022's canonical hash deliberately ignores lifecycle-field-only edits. A naïve
  mapping would revoke a PO's approval on cosmetic changes. **Mitigation:** the committed
  sidecar hash stays the arbiter of freshness — the GitHub review is the *trigger*, the
  canonical `requirements.md` hash *decides* whether the approval still holds.
- **R4 — Audience map versioning.** If "which types are PO-facing" changes after POs have
  approved projections, prior approvals could silently become wrong. **Mitigation:** pin
  the audience-map version into the approval record (see *Costly to Refactor*).

## Alternatives considered

- **A — Declared whole-spec `audience:` frontmatter.** Coarsest; a single spec that mixes
  business and technical content cannot be split; the field rots. Rejected as the primary
  mechanism (retained conceptually only as the file-level truth Phase B already gives us).
- **D — Per-paragraph declared `audience` on every item.** Hand-maintained, drifts, needs
  a gate. Kept only as a narrow escape-hatch override over the derived default, never the
  base mechanism.
- **Person-level assignment (CODEOWNERS-style PO).** Re-opens the authority-gate question
  SPEC-022 FR-2 deliberately deferred, and ties audience to individuals rather than
  content. Rejected: audience is a property of the paragraph, not a name.
- **Approve the whole spec but hide technical sections in the UI only (presentation, no
  approval change).** Simplest, but the PO's signature would then attest to engineering
  paragraphs they never saw — a signpost that lies. Rejected as insufficient; the
  approval unit must match what was shown.
- **Bespoke networked PO webview as the primary surface.** Reusing SPEC-014 to build a
  custom PO review-and-approve page. Rejected as the *primary* Phase-B surface — it
  rebuilds what GitHub already gives us (rendering, verified identity, per-file routing,
  notifications) — but **retained as the Phase-C fallback**, since only a custom surface
  can project a paragraph *subset within a single mixed file*, which GitHub's per-file
  granularity cannot express.

## Costly to Refactor

- **The audience→type map is load-bearing once POs approve projections.** Changing which
  types count as PO-facing retroactively changes *what a past PO approval meant*.
  Therefore the map is **versioned**, and the version is pinned into the `ApprovalRecord`
  (extending SPEC-022's record); a map change invalidates affected approvals and
  re-requests them rather than mutating their meaning in place.
- **The paragraph-projection hash** (Phase C) becomes a public contract the moment a PO
  approves against it; its canonicalisation rules must be fixed before first use, like
  SPEC-022 FR-3's whole-file hash.
