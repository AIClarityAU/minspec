# DR-056: Agent-proof approver identity; separate agent commit identity from human approver identity

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records context, decision, follow-ups; no code ships with this DR.

## Context

MinSpec captures the approver of a spec/DR as `approvedBy = git config user.email`, read
offline at approval time (`approval.ts:12-13`, `:57`, `:284-296` — `execFileSync('git',
['config','user.email'])`, Tier-0, degrades to `'unknown'`). [DR-012](DR-012.md) defines
approval as **an explicit human act**; [DR-033](DR-033.md) §6 made the *reviewer* identity
trustworthy via the `AI_REVIEW_BOT_LOGINS` allowlist (only the bot may apply `ai-review:*`).
The *approver* identity has no equivalent guard.

**The defect, found this session.** This checkout carries a **MinSpecPro-only** repo-local
override in `.git/config`: `user.email = claude@harvest316.com` (the maintainer's
Claude-account email), shadowing the correct global `~/.gitconfig` (`github@harvest316.com`,
the human's real GitHub identity). None of the sibling repos (sealbox, scroogellm, ColdForge,
LeadForge) have this override — it was set by a `git config --local` in this repo alone, the
value being the Claude-account email (a Claude session set its own account email as the repo
identity). Consequences, both from this one line:

1. **Commit masquerade** — every container/agent commit is authored `Paul Harvey
   <claude@harvest316.com>`: the human's *name* with the container's email.
2. **Ambiguous approval provenance** — because `approvedBy` reads the *same* git-config value,
   ~17 specs on `origin` record `approvedBy: claude@harvest316.com` (vs ~8 `github@…`). The
   record **cannot distinguish** "the human deliberately approved" from "an agent/container ran
   *Approve Spec*" — one blurred identity authors both agent work and approvals. The
   independent AI reviewer flagged exactly this on PR #677.

**Root class.** Deriving the *human-approver* identity from `git config user.email` — a
trivially-settable value **shared with agent commit authoring** — is structurally unable to
enforce DR-012's "explicit human act." This is the HITL-provenance analogue of the reviewer
allowlist DR-033 already built: an approval, like an `ai-review:pass`, is only trustworthy if
its applier is provably **not** the agent.

## Decision

**1 — Separate the agent/container commit identity from the human.** Agent/container commits
must be authored as **`minspec-sdd[bot]`** (the existing App identity, id 299695933), never as
`Paul Harvey <claude@…>`. The maintainer's stated intent: *automation → `minspec-sdd[bot]`;
the human's explicit acts (approve, merge) → the human.* Mechanism (settle in Specify): the
container's git identity points at the bot's noreply, or agent commits are made under the bot
— so a reader can tell agent work from human work by author alone.

**2 — Approver identity is agent-proof (deny-by-default).** MinSpec must **refuse** to record
an approval when `git config user.email` is a known agent/bot identity. Mirror
[DR-033](DR-033.md)'s `AI_REVIEW_BOT_LOGINS` as an inverted **denylist** (e.g.
`MINSPEC_AGENT_IDENTITIES`, incl. `claude@harvest316.com` and the bot noreply): if the capture
resolves to a denied identity → *Approve Spec* aborts with "an agent can't self-approve —
approve under your human identity." `'unknown'` (empty git email) is likewise not a valid
approver. This keeps the Tier-0/offline capture (git config, no network) while closing the
"forgeable-by-config" hole.

**3 — Fix the repo-local override as part of the change.** The MinSpecPro `.git/config`
`user.email = claude@harvest316.com` override is removed/repointed so this repo stops
authoring under the human's name with the container email. (Do **not** simply inherit
`github@` — that would make agent commits masquerade as the human's *real* identity; Decision 1
governs what agent commits use.)

**4 — Reconcile the ~17 `claude@`-attributed approvals.** Their provenance is unverifiable
(Decision 2's gate did not exist when they were recorded), so they are treated as **not proven
human**. The maintainer either (a) re-ratifies them under the human identity, or (b) confirms
he personally drove them and they are accepted-and-relabelled. Not swept silently; not
fabricated by an agent ([no self-approval under borrowed identity](../../packages/minspec) —
prior guidance, #517).

## Rationale

The whole product is HITL / never-wrong: an approval that a settable git-config value can
mint under the human's name is a signpost that can lie. DR-033 already established the pattern
(trustworthy applier via allowlist) for the reviewer; approval is the *more* critical gate and
had *no* such guard. Separating agent identity from human identity (Decision 1) is what makes
the denylist (Decision 2) enforceable — you cannot deny the agent as approver if the agent
commits under the human's identity.

## Consequences

- Approval provenance becomes trustworthy: `approvedBy` can no longer be the agent, so a
  recorded human approval means a human (or the gate refused).
- Agent vs human work is legible from commit author (bot vs human), matching the maintainer's
  intent and un-masking the ~everything currently authored "Paul Harvey <claude@…>".
- ~17 existing approvals need re-ratification (Decision 4) — a one-time cleanup cost, the
  honest price of the prior ambiguity.
- `approval.ts` gains a denylist check at capture; the git-config source stays (Tier-0),
  so no new network/credential surface.

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| Denylist incomplete → an agent identity not listed still self-approves | New agent email/CI identity not in `MINSPEC_AGENT_IDENTITIES` | Deny-by-default posture + document that every agent/CI identity MUST be added; treat `'unknown'`/empty as invalid too. |
| Fixing the git identity breaks the human's own local approve flow | If repo-local is set to the bot, a human approving *in this container* would be denied (correctly) but have no human identity to use | Specify how the human approves from the container (e.g. approve runs under an explicit human-identity prompt/setting, not the ambient git email); or the human approves from their desktop (`github@`). |
| Bot-authored local commits don't attribute to the bot on GitHub | Author-email attribution is finicky for App bots | Settle the exact mechanism (bot noreply `<id>+minspec-sdd[bot]@users.noreply.github.com`, or commits made via the App token) in Specify. |
| Re-ratification churn misread as new approvals | 17 re-approvals land at once | Do them in one clearly-labelled reconcile change, not bundled with feature work (the #677 scope lesson). |

## Alternatives considered

- **Just remove the repo-local override (inherit `github@`).** Rejected: makes agent commits
  masquerade as the human's *real* GitHub identity — worse, not better; and still derives
  `approvedBy` from a settable value with no agent guard.
- **Trust `git config user.email` as-is.** Rejected: it is the exact mechanism that produced
  the ambiguity; DR-012's "explicit human act" is unenforceable without a guard.
- **Full cryptographic/verified human identity for approval.** Deferred: heavier, and likely a
  network/credential surface that conflicts with Tier-0. The offline denylist closes the
  immediate hole; a stronger identity can build on it later.

## Costly to Refactor

Low-medium. The denylist check is a small addition to `approval.ts` capture; the git-identity
separation is config + a documented convention. The genuinely costly part is **reversing a
wrong human-identity model once approvals are recorded against it** — which is why this is a
recorded decision, not a silent config edit.
