# DR-062: Approval validity is graph-aware — cross-artifact staleness, ADR/epic records, actor-agnostic implement-gate

## Context

A read-only audit of the approvables system established three facts, each with `file:line`
evidence:

1. **Approval validity is per-artifact-content only.** A spec's approval binds a canonical
   hash of *its own* content ([approval.ts:306-313](../../packages/minspec/src/lib/approval.ts#L306)).
   `depends_on` / `relates_to` / `supersedes` edges are parsed and walked, but **only** for
   corruption detection and priority ordering ([next-task.ts:262-395](../../packages/shared/src/next-task.ts#L262))
   — never as an input to approval validity. So when an upstream approvable a spec depends
   on changes, the dependent stays `approved`, signed off against content that no longer
   exists. **There is no cross-artifact change propagation.** The one incidental effect —
   editing A flips A's own state to `stale`, which re-floors an *advancing* dependent B in
   the queue — leaves B's approval record untouched and lets B silently re-clear the moment
   A is re-approved, no matter how materially A changed.

2. **Two divergent approval mechanisms.** Specs get a real, committed, attributed,
   hash-locked sidecar record ([DR-034](DR-034.md)). ADRs and epics get a **one-way
   frontmatter status flip** with no record, no hash, no staleness
   ([adr-manager.ts:381-411](../../packages/minspec/src/lib/adr-manager.ts#L381),
   [epic-manager.ts:305-323](../../packages/minspec/src/lib/epic-manager.ts#L305)). Editing an
   accepted DR or active epic never un-accepts it. This is the project's recurring
   validator-asymmetry class, applied to decisions and epics — and it matters here because
   **most `depends_on` targets in the corpus are DRs**, which today carry nothing to
   fingerprint.

3. **"No implementing an unapproved approvable" is enforced session-locally only, for
   specs only.** The spec-gate ([DR-012](DR-012.md), [DR-031](DR-031.md)) is a Claude Code
   `PreToolUse` hook that denies an agent's Edit/Write to an unapproved T3/T4 spec's declared
   impl code. It does **not** fire for a human editing directly, for CI, or for any non-CC
   script; it gates **specs only** (not DRs/epics); and it cannot block creation of files no
   spec declares (greenfield, [AIClarityAU/minspec#460](https://github.com/AIClarityAU/minspec/issues/460)).

The foundations these build on are strong and are kept unchanged: committed attributed
path-keyed sidecars, a canonical hash that ignores tool-driven status flips but voids on
content edits, and authoritatively *derived* status. The gap is that **validity stops at the
artifact boundary.**

## Decision

**Approval validity becomes graph-aware.** Three coupled changes, all obeying one spine:
*staleness is DERIVED on read, never an event that mutates a committed sign-off.*

### 1. Cross-artifact staleness via committed upstream fingerprints

When a human approves B, the approval record snapshots the current canonical hash of each
thing B depends on — the cross-artifact analog of the self `specHash`:

```jsonc
// .minspec/approvals/<path>.json  (existing committed sidecar, extended)
{
  "specPath": "...", "specHash": "...", "approvedBy": "...", "tier": "T2",
  "upstreamDeps": [
    { "ref": "DR-019", "hash": "sha256:…" },   // what B relied on, at approve time
    { "ref": "SPEC-012", "hash": "sha256:…" }
  ]
}
```

`resolveStatus` gains a second dimension → a new derived state **`upstream-stale`**: B was
signed off against upstream content that has since changed. Both `stale` (self-content
drift, today's behaviour) and `upstream-stale` mean "needs re-approval" but are
UI-distinguishable. This is a pure function of `(B's record, current graph)` — deterministic,
no LLM. **Hash drift is a fact; whether it *matters* stays a human judgment** (§4).

`upstream-stale` is added to the resolver's `gateCleared`-false set, so **propagation
cascades through the existing flooring machinery for free** — an upstream-stale B floors its
own dependents, and an `implementing` B that is upstream-stale becomes a gate-violation
exactly like `implementing-unapproved`. No new DAG edge type. A new pure reverse-index
(keyed by `edge.to`) answers the "who depends on this" question the graph cannot answer
today, powering the HITL surface in §4.

### 2. ADR/epic get the same hash-locked committed record

Give ADRs and epics the same committed, attributed, canonical-hash-locked record specs
have (extend [DR-034](DR-034.md)'s store to all three approvable kinds), with the same
`upstreamDeps`. Without this, propagation is blind to the majority of real dependencies
(DR targets dominate), so **this is a prerequisite of §1, not a later slice.** Acceptance
stops being a bare status flip and becomes a hash-bound record; editing an accepted
DR/epic derives it `stale`, symmetric with specs.

### 3. "No implementing an unapproved approvable" moves to an actor-agnostic gate

The session-local `PreToolUse` spec-gate is retained but is no longer the only enforcement.
Add a **commit-time / CI gate** (the actor-agnostic tier, per [DR-031](DR-031.md)/DR-037's
reasoning) that reads the committed approval records + each approvable's owned-file set and
**blocks a commit/PR that touches the impl code of an unapproved (or `stale`/`upstream-stale`)
approvable** — now covering DRs and epics too, because §2 gives them records to read. This
closes the "human editor / CI / script is unconstrained" and "specs only" holes. The
greenfield hole (#460) is closed only once approvables declare `implements:`/`affects:`
(the code side, §6 leg B).

### 4. Never-wrong HITL flow (keyboard-first, non-modal)

On debounced recompute after a save, the deterministic pass re-hashes, walks the reverse
index, and finds affected dependents. A **non-modal toast** over the visible artifact (never
focus-stealing, per the HITL-UX rule): *"DR-019 changed — 3 approved artifacts depend on it.
**Alt+R** to review."* The human verifies the *signal*, then one keystroke: **Alt+A** on a
dependent = "immaterial" → re-snapshot that upstream hash to current, clearing
`upstream-stale` (an attributed human ack, committed via the existing commit-on-approve
path); or edit to accommodate, then approve normally. **The tool never LLM-judges materiality
and never auto-mutates a sign-off record** — the `upstreamDeps` hashes are written once, by
the human approving that dependent. A background rewrite of an attributed sign-off would
forge a human decision (the never-wrong invariant, and the no-self-approval-under-borrowed-identity
rule).

### 5. Precondition — edges must be trustworthy

§1's propagation is only as good as the `depends_on` edges. The intended authoring flow is
opt-in-LLM-suggests → human bulk-approves ([SPEC-012] R2, [DR-019] — LLM never auto-writes,
never in the ranking path). Bulk-approve can admit a plausible-but-false edge. Structurally-bad
edges (dangling/cyclic) are already caught as corruption; the residual —
**a plausible false edge** — is the asymmetric half the current one-directional prose-link
linter misses. Shipping §1 therefore **requires** the symmetric linter (flag an edge whose
target has no textual basis in the body) + edge provenance
([AIClarityAU/minspec#803](https://github.com/AIClarityAU/minspec/issues/803)). A wrong edge's
blast radius is small — ordering-only, advisory, human-overridable, one-line fix — so
bulk-approve is safe *once #803 lands*.

### 6. This is leg A of three; the other two are recorded elsewhere

The full picture is one primitive — *fingerprint what X was approved against, derive
staleness on read, deterministic facts gate / semantic judgment advisory* — in three
directions:

- **Leg A — approvable ↔ approvable (this DR).**
- **Leg B — approvable ↔ code (false-built / drift).** Owned by
  [AIClarityAU/minspec#643](https://github.com/AIClarityAU/minspec/issues/643) — a
  deterministic, **symbol/test-anchored** (not line/paragraph — line refs rot) trace graph
  over the CodeGraph index. #643's "target symbol hash changed since the FR was approved ⇒
  drift-flag" is exactly this DR's `upstreamDeps` primitive pointed at code. It supersedes
  the file:line-citation approach and is the natural code-side first build.
- **Leg C — corpus semantic consistency (spec vs spec).** Owned by
  [AIClarityAU/minspec#804](https://github.com/AIClarityAU/minspec/issues/804) (standing-set
  sweep, report-only) + an amendment to [DR-047](DR-047.md); **not** a fresh DR.

Section/paragraph-level edges are explicitly **out of scope of the first slices**: hand-authored
anchors rot like every other optional edge (`supersedes:`/`goal:` have zero corpus uses today).
Whole-doc `upstreamDeps` ships first; section precision is a later, mostly-auto-derived
refinement anchored on [SPEC-012] R2, not a new convention. Edges stay in **frontmatter**
(`depends_on: [SPEC-012]`); moving edge source-of-truth into prose would need its own DR.

## Consequences

**Positive:**
- Closes the silent-lie: an approved artifact can no longer read `approved` while the upstream
  it was signed off against has drifted. Cross-artifact change propagation exists,
  deterministically, with no LLM in the decision path.
- Ends the ADR/epic approval asymmetry — decisions and epics become first-class hash-locked
  approvables, so the dependency graph's majority (DR targets) participates in staleness.
- "No implementing an unapproved approvable" becomes actor-agnostic (commit/CI), covering
  human/CI/script edits and DRs/epics, not just CC-session spec edits.

**Negative / costs:**
- Schema change to the approval record (`upstreamDeps`) — additive and backward-compatible
  (absent ⇒ behaves as today), but a stored-shape commitment.
- Approving an artifact now also snapshots upstream hashes — a slightly heavier approve path.
- ADR/epic acceptance gains a record + hash where it had none — a migration for existing
  accepted DRs/epics (backfill `migrated: true`, mirroring the FR-5 spec backfill).
- Depends on #803 (trustworthy edges) landing first, or bulk-approved false edges produce
  spurious `upstream-stale` noise (bounded: ordering + a re-ack keystroke, never data loss).

## Alternatives considered

- **Event-driven revocation (watcher deletes/rewrites a dependent's sidecar on upstream
  edit).** Rejected — mutating an attributed sign-off in the background forges a human
  decision (never-wrong). Derived-on-read is the only truthful model.
- **LLM judges whether an upstream change is material.** Rejected — puts an LLM in the
  approval decision path. The tool asserts the *fact* of drift; materiality stays the
  human's Alt-A call.
- **Leave ADR/epic as status-flips; fingerprint only spec→spec deps.** Rejected — DR targets
  dominate `depends_on`, so a spec-only first slice fires on almost nothing (internal
  contradiction with the "blind to the majority" finding). ADR/epic records are a
  prerequisite, not a follow-up.
- **Section/paragraph-level edges from day one.** Rejected — hand-authored precision rots
  (proven by zero corpus adoption of existing optional edges); ship whole-doc first,
  auto-derive precision later.

## Costly to Refactor

Not cheap to reverse — this is why it is a DR. It changes the stored approval-record shape,
adds a derived state consumed by the resolver and the gate, converts ADR/epic acceptance from
a status flip to a hash-bound record (a data migration), and reverses the standing
"approval is per-artifact-content only" stance into "depends_on is an input to validity."
Undoing it after adoption means unwinding a schema + a migration + gate behaviour teammates
rely on — well over a day. Recorded accordingly.
