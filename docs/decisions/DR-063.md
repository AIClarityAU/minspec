# DR-063: `awaiting-approval` — one positive "your turn" queue signal

## Context

The independent AI reviewer (DR-033 §6) records a PR verdict as `ai-review:{pass,changes,blocked,pending}`.
On any non-pass, `.github/workflows/ai-review.yml` **also** applies `needs-human-review`
**unconditionally at t=0** (`ai-review.yml` L568-574 and L606-608). Two problems compound:

1. **`ai-review:changes` is overloaded.** It means both *"the reviewer read the code and
   wants specific fixes"* (substantive, AI-remediable) and *"the review could not produce a
   trustworthy green"* (procedural fail-closed — garbled/injected/truncated verdict,
   `ESCALATE`, workflow error). Different meanings, one label.

2. **`needs-human-review` fires on the wrong polarity.** It rides along with every
   `ai-review:changes` — i.e. on AI **failure**. But the auto-remediation loop
   ([remediate-pr.sh](../../scripts/remediate-pr.sh), DR-060/061) then gives a **robot** up to
   two attempts before a human is genuinely needed. So `needs-human-review` is present during
   the window where a robot — not a human — is the next actor. The two subsystems contradict:
   the eager `needs-human-review` **predates** the remediation loop and was never reconciled
   when DR-060/061 landed.

The concrete pain: a human doing review cannot tell, at a glance, whether a PR **passed** AI
review. There is **no positive "AI passed → your turn" signal**. PR
[#782](https://github.com/AIClarityAU/minspec/pull/782) is a live specimen — a machinery PR
forced to `ai-review:changes` by the self-edit guard (the code was fine; 8 findings fixed)
carried `needs-human-review` and merged via `--admin`. Filtering the human queue on
`needs-human-review` mixes AI-rejected PRs, machinery-forced PRs, and genuine
awaiting-approval PRs indistinguishably.

SPEC-031 already frames the target end-state — INV-2 (`ai-review:*` advises, never approves;
"AI-greenlit ≠ approved"), INV-3 (AI-review and human-approval are **decoupled** axes), INV-4
(the human queue = *greenlit AND prior-gates-clear AND human-gate-open*). What was missing is
a **concrete, positive artifact** for that predicate, and the reconciliation of the eager
`needs-human-review`.

## Decision

Introduce **`awaiting-approval`** — a single positive label meaning *"the independent AI
review has PASSED and the only remaining gate is a human keystroke."* It is the canonical
"my turn" filter, replacing the ambiguous read of `ai-review:pass` plus a negative label.

### 1. One authoritative owner, no scattered mirrors

`.github/workflows/ready-to-merge.yml` is already the **sole** authority that decides, on
every PR event, whether a PR carries a **verified, fresh, un-reverted** `ai-review:pass` with
no outstanding `ai-review:changes` (`ai-review-guard.js` `decideStatus` → `status.state`). It
now applies/removes `awaiting-approval` **in lock-step with that same decision**:

- `status.state === 'success'` **and** native auto-merge is **not** armed → add the label.
- otherwise → remove it (best-effort; `404` = already gone is benign).

Because every stale-strip (#359), forged-revert (#397), and `changes`-flip already drives
`status.state` to `failure`, the label is cleared **with no extra mirror site** — the durable
failure mode of a hand-mirrored label ("added on pass in one place, forgotten on a strip in
another") is structurally impossible here.

### 2. Never falsely claim a robot-merged PR is the human's turn

When native auto-merge (DR-061) is armed, the PR merges itself the instant the gate greens —
it is the **robot's** turn, never a human's. `shouldAwaitApproval({statusState,
autoMergeArmed})` returns `false` in that case, so an auto-merging PR never enters the queue.

### 3. Pure, tested seam; best-effort side-effect

The decision is a pure function `shouldAwaitApproval()` in `ai-review-guard.js`
(unit-tested). The label write itself is **best-effort** and wrapped so it can never fail the
load-bearing `ready-to-merge` commit status — the label is a convenience filter *on top of*
the authoritative status, never a gate.

### 4. Boundary with the SPEC-012 in-editor resolver (no second owner)

`awaiting-approval` is a **GitHub-surface projection** of the human-queue predicate for PR
review. It does **not** duplicate or compete with the Tier-0 in-editor signpost resolver
(SPEC-012 FR-4), which computes the next-human-task **inside the editor** from the same
inputs. They are complementary views of one predicate (greenlit + gate-open), derived
independently on their own surfaces, and agree by construction — the CI label is not a
resolver, and the resolver does not read the label. No cross-boundary ownership is created.

### 5. Supersession of the eager `needs-human-review`

`awaiting-approval` is the canonical "human's turn" signal. The eager
`needs-human-review`-on-`ai-review:changes` is **superseded**: `needs-human-review` should
mean only *"automation is exhausted — a human must fix"* (applied by `remediate-pr.sh` at
cap/escalate/quarantine/conflict, where it already is). Retiring the eager application is the
**subtractive** follow-up ([#816](https://github.com/AIClarityAU/minspec/issues/816)),
deliberately **not** bundled with this additive label because it touches fail-closed
semantics. Until then the two are mutually exclusive by construction (a failing gate removes
`awaiting-approval`; `needs-human-review` only appears with a non-pass), so shipping the
positive label first is safe.

### 6. Status & precedent

Recorded `proposed` (not yet human-accepted). The implementation is additive, reversible, and
best-effort; it is dogfooded ahead of formal acceptance exactly as the DR-060/061 autonomous
pipeline machinery was — a small, low-blast machinery increment under a DR, with the human
keystroke still holding the actual merge. Materialised at the requirement level as **SPEC-031
INV-8 / FR-9** (the spec is the requirement pointer; **this DR is the rationale authority**
the code cites).

## Consequences

- A human reviewing the PR queue gets a truthful **single-label** filter: `awaiting-approval`
  present → AI passed, your keystroke gates it; absent (with `ai-review:changes`) → not your
  turn (robots remediate / AI rejected). The two never coexist.
- The label tracks the gate on **every** PR event — a new push that strips a stale pass also
  clears `awaiting-approval` the same run, so the queue never shows a stale "your turn."
- No new merge power: `awaiting-approval` advises; `ready-to-merge` remains the gate and the
  human still holds the keystroke (never-wrong / INV-2).
- Machinery PRs (self-edit guard forces `ai-review:changes`) never get `ai-review:pass`, so
  they never get `awaiting-approval` — their `--admin` flow is unchanged (a known, separate
  path; the doc-approvable + machinery surfaces are out of scope here — see Follow-ups).

## Alternatives considered

- **Reuse `ai-review:pass` + `needs-human-review` as a two-label AND filter** (the reviewer's
  first instinct). Rejected: fragile (an AND of a positive and a negative label), and it keeps
  `needs-human-review` overloaded across "AI failed → fix" and "AI passed → approve".
- **Put the signal only in the SPEC-012 in-editor resolver.** Rejected: leaves the GitHub PR
  surface — where the daily "is it my turn to merge?" question actually lives — with no filter.
- **Apply the label from `ai-review.yml` on the pass branch, mirror-removing at every strip
  site.** Rejected: multiple owners across ai-review.yml + the fail-closed backstop +
  ready-to-merge's strip → the classic drift bug. `ready-to-merge.yml` is the single place
  that already recomputes the authoritative state on every event, so it is the correct owner.

## Costly to Refactor

**No.** The whole mechanism is one pure predicate + one best-effort label write in a single
workflow, plus a label. Reverting is deleting ~40 lines and the label. This is why it ships as
a dogfooded `proposed` decision rather than waiting on a full spec-implement cycle.
