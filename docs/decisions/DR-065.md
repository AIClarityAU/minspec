# DR-065: Presence-gated fast-forward — the one sanctioned exception to "never move a shared HEAD"

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> Architecture Decision Record*). Carves a **narrow, gated exception** into
> [DR-051](DR-051.md) §4a and rule #8 ([#168](https://github.com/AIClarityAU/minspec/issues/168)),
> and is the sync-side sibling of [DR-046](DR-046.md)'s base-freshness invariants.
> Materialized in [SPEC-026](../../specs/minspec/SPEC-026-session-presence/requirements.md).

## Context

Two rules the project already holds are, as written, in direct contradiction.

**Rule #8 / DR-051 §4a — never move a shared HEAD.** An agent never moves the HEAD of a
primary/shared checkout (`switch`/`checkout`/`reset --hard`/`rebase`/`merge`); all branch
work happens in a worktree. The rule exists because the failure it prevents is
*unrecoverable*: moving HEAD under a live session rewrites the working tree beneath it and
strands uncommitted work on the wrong branch. This has bitten repeatedly — it is the
originating incident behind #168, and DR-046 promoted the same concern to a T0 invariant
for SealBox dispatch.

**DR-051 §4c — on `main` is not the same as current with `origin/main`.** A shared checkout
can sit on `main` and be dozens of commits behind. Then the docs-on-`main` visibility win
*inverts*: the approvable "just there in the IDE" is a stale copy, and any gate that reads
the working tree (`validate`, the approve-hash) judges outdated content — a wrong verdict,
which in a never-wrong product is the worst defect class. Not hypothetical: §4c records a
SPEC-024 approval stranded because the acceptance criteria had merged to `origin` while the
35-behind local `main` still showed the pre-AC file.

So: staleness causes wrong verdicts, and the obvious remedy (advance the checkout) is
exactly what rule #8 forbids. The founder's 2026-07-16/17 ask ("keep the shared checkouts
current") is a request to resolve that tension, and ai-review
[#813](https://github.com/AIClarityAU/minspec/issues/813) flagged that a drain doing this
silently would be an **undocumented carve-out of a T0-grade rule** — the kind of exception
that, unrecorded, gets copied by the next author who sees the precedent but not the
reasoning.

What makes a resolution possible now is that **SPEC-026's presence layer makes occupancy
observable**. Each live session heartbeats a record every 30s carrying `worktreeRoot`
(`git rev-parse --show-toplevel` — the discriminator that differs per worktree even though
`.git` is shared) and `pid`; a record is live iff `lastSeen` is within 120s **and**
`process.kill(pid, 0)` does not throw (FR-2/FR-3/FR-4). "Is anyone in this tree?" stopped
being a guess and became a **query**.

The critical asymmetry — and the reason this DR exists rather than a code comment — is that
the two errors are not symmetric. A **skipped** fast-forward leaves a checkout stale:
harmless, self-correcting, retried on the next drain. A **wrongful** fast-forward mutates a
live tree: unrecoverable. The gate must therefore be biased, and its bias must be recorded
as a decision, because a future maintainer optimizing for "why is my checkout still stale?"
would otherwise loosen it without knowing which direction is the cliff.

## Decision

**A drain/loop MAY fast-forward a shared checkout to `origin/main` ONLY when the SPEC-026
presence layer positively confirms that NO live session claims that checkout's
`worktreeRoot` (heartbeat within 120s AND PID alive). Absence of proof means fetch-only.**

This is the **sole sanctioned exception** to DR-051 §4a / rule #8 / #168 "never move a
shared HEAD." It is justified because an **unoccupied checkout has no live tree to
disturb** — the rule protects live sessions, and a provably dormant checkout has none, so
advancing it violates the rule's *letter* while serving its *purpose*. Fail-safe toward
fetch-only preserves the invariant under any doubt.

### 1. Positive proof, not absence of evidence — the gate's fail direction

`isCheckoutOccupied` (`packages/minspec/src/lib/presence.ts:170`) returns `false`
(**dormant — ff permitted**) **only** on a conjunction of two positive facts:

- **(a) presence is demonstrably running** — at least one LIVE record exists *anywhere*, AND
- **(b) no live record's `worktreeRoot` resolves to this checkout**.

**"Anywhere" means every worktree's OWN sessions dir, not just the primary's.** Each
`SessionPresenceManager` writes its heartbeat into *its own* worktree's
`.minspec/sessions/` (a working-tree path), so both readers enumerate every worktree via
`git worktree list --porcelain` — always including `PRIMARY_ROOT`, deduped by canonical
path, git-failure falling back to the primary alone — and aggregate each worktree's own
records before applying (a)/(b). Reading only `PRIMARY_ROOT`'s dir was a real hole: a live
session in a *linked* on-`main` worktree would be invisible, its clean tree ff'd under it —
the exact rule-#8 corruption this DR forbids (fixed #846, security panel).

Condition (a) is the load-bearing subtlety and the easiest to mistake for redundancy. An
**empty / all-stale / all-corrupt** sessions directory is *indistinguishable* from "the
heartbeat isn't running" — a live session with presence disabled, crashed, or never
started looks exactly like no session at all. Treating "no records" as "nobody home" would
make the gate open **widest** precisely when its information is **worst**. So an empty dir
⇒ **occupied**. Every error path — missing dir, unreadable dir, corrupt record, `kill`
throw — returns `true` (occupied) (`presence.ts:175,179,183,186`).

This fail direction is deliberately **opposite** to SPEC-026 FR-12's pre-commit contention
backstop, which fails **open** (allow). The inversion is not an inconsistency: FR-12's
false-positive blocks a legitimate commit (visible, annoying, recoverable), whereas this
gate's false-negative corrupts a live tree (silent, unrecoverable). **Each gate fails toward
its own cheap error.** Recorded explicitly so the next reader does not "fix" the apparent
inconsistency by aligning them.

### 2. The exception is a conjunction of four guards, not presence alone

Presence-dormancy is *necessary but not sufficient*. `sync_shared_checkouts`
(`scripts/drain-inbox.sh:189`) fast-forwards only when **all four** hold; any failure ⇒
fetch-only:

| Guard | Condition | Protects against |
|---|---|---|
| **G1** | HEAD is the default branch (`main`) | Touching a feature-branch WIP tree |
| **G2** | Content-clean (`git status --porcelain` empty) | Stomping uncommitted WIP left by a session that exited |
| **G3** | Not presence-occupied (§1) | Mutating a live session's tree |
| **G4** | True fast-forward (`merge-base HEAD origin/main == HEAD`) | Losing local-only commits on a diverged checkout |

G2 and G4 are what make the exception safe **even if presence is wrong**. A dormant-but-
dirty checkout is skipped; a dormant-but-diverged checkout is skipped. The operation is
`merge --ff-only` — **never** `reset`, `rebase`, or a merge commit — so **no commit and no
WIP can be lost by construction**, not merely by policy. Presence is the *newest* guard,
not the *only* one; the pre-existing structural guards remain the floor.

### 3. Fetch is unconditional; only HEAD movement is gated

`git fetch` never moves HEAD and never touches the working tree, so it runs on **every**
checkout regardless of occupancy (`drain-inbox.sh:197`). This matters: even a checkout that
is never eligible for ff still gets current `origin/*` refs, which is what the extension's
**offline** staleness detection (DR-051 §4c / [#512](https://github.com/AIClarityAU/minspec/issues/512))
reads to warn a human that their tree is behind. The gated ff removes the stale-checkout
problem where it can do so provably; the fetch improves the *detection* everywhere else.
The exception is scoped to the one operation that carries the risk — **moving HEAD** — and
nothing more.

### 4. Bash and TypeScript must agree byte-for-byte

The gate exists **twice**: `isCheckoutOccupied` in TS (the extension's future callers) and a
jq-free `checkout_occupied()` in `drain-inbox.sh:148`. Two implementations of one predicate
is a drift class this project has already been bitten by, so the constants
(`HEARTBEAT_SECS=30` / `STALE_SECS=120`, **paired** as `STALE = 4 × HEARTBEAT`) are named in
one place per language with a tie-back comment, and a **golden-fixture parity test**
(`packages/minspec/tests/presence-sync-parity.test.ts`, SPEC-026 FR-14 family) fails on
divergence. Per the constitution's *enforce, don't trust the model* principle: the agreement
is a **gate**, not a comment asking future authors to remember.

### 5. Scope of the exception — deliberately minimal

This DR sanctions **exactly one** operation: `merge --ff-only` to `origin/<default>`, by a
drain/loop script, on a checkout passing G1–G4. It does **not** sanction, and must not be
cited to justify: any `switch`/`checkout` of a shared HEAD; any `reset`/`rebase`/merge-commit
on a shared checkout; ff of a **feature** branch (G1); ff of a checkout whose dormancy is
merely *assumed*; or the **extension** (Tier-0, offline core) doing any of this — DR-051 §4c
is explicit that the extension half offers a **consent-gated** sync, never a silent one. The
scripts may `fetch` because they are Tier-1 dispatch tooling (DR-004); the extension may not
(INV-1, offline core). Kill-switch: `MINSPEC_DRAIN_GATED_FF=0`.

## Consequences

**Positive:**
- The stale-shared-checkout defect class (DR-051 §4c — wrong verdicts from outdated content,
  the SPEC-024 stranding) is eliminated **for dormant checkouts** without weakening rule #8
  for live ones.
- Rule #8's exception surface becomes **explicit, enumerated, and gated** rather than
  precedent-by-example. A future author reading `sync_shared_checkouts` finds the reasoning,
  not just the code.
- Presence (SPEC-026 Layer 1) earns a second consumer beyond the status bar, validating the
  `worktreeRoot` discriminator as the "same working tree?" primitive DR-051 §4a assumed.
- The fail-safe bias is recorded, so "why is my dormant-looking checkout still stale?" is
  answered by a decision rather than re-litigated as a bug.

**Negative:**
- **Presence-off sessions never benefit.** A user who disables the heartbeat sees condition
  (a) fail forever, so their checkouts are always fetch-only. This is correct-by-design (they
  are indistinguishable from a live unclaimed tree) but will read as "the sync doesn't work."
  Accepted; the alternative is the unrecoverable error.
- **Two implementations of one predicate** (TS + bash) is a standing drift risk, mitigated by
  the parity test (§4) but not eliminated.
- **Presence is single-machine.** `process.kill(pid, 0)` is meaningless for a PID from another
  host; a shared checkout on network storage could in principle be claimed by an unobservable
  remote session. SPEC-026 puts cross-machine presence out of scope and relies on the 120s
  threshold to evict foreign records — which makes such a record *stale* ⇒ occupied ⇒
  fetch-only, so the failure lands on the **safe** side.
- The exception, however narrow, is a precedent. §5 exists to bound it.

## Alternatives considered

- **Do nothing — keep rule #8 absolute, accept staleness.** Rejected: DR-051 §4c documents
  this producing *wrong verdicts* (the stranded SPEC-024 approval), which is the never-wrong
  violation the project treats as its worst defect. "Never move a shared HEAD" was written to
  protect live trees; applying it to a provably empty tree serves the letter against the
  purpose.
- **Fast-forward whenever the tree is clean and on-main (no presence check).** Rejected: a
  live session frequently has a clean tree — between commits, mid-read, thinking. Clean is
  *not* evidence of dormancy, and this is precisely the reasoning that caused the original
  #168 incident.
- **Treat an empty sessions dir as "nobody home."** Rejected — this is the whole decision in
  miniature. It inverts the fail-safe exactly where information is weakest (§1) and would
  open the gate widest on a machine where presence never started.
- **Lock files instead of presence.** Rejected: a second source of truth for "who is in this
  tree," with its own staleness and cleanup problem. SPEC-026 FR-2 is explicit that the
  heartbeat **is** the live claim — no separate `.minspec/locks/` store.
- **`reset --hard origin/main` instead of `--ff-only`.** Rejected outright: it discards local
  commits and WIP. `--ff-only` makes loss structurally impossible, so G4 is enforced by git
  itself rather than only by our check.
- **Let the extension do the sync.** Rejected: Tier-0 offline core cannot `fetch` (INV-1), and
  a silent HEAD move in the user's IDE is the opposite of DR-051 §4c's consent-gated posture.
  The drain is Tier-1 tooling (DR-004) and is the right home.
- **Prompt a human for each ff.** Rejected for the *drain* (unattended by definition — a
  prompt nobody sees is a hang). Retained for the extension path (#512), where a human is
  present. Different actor, different answer.

## Costly to Refactor

**Costly:** the **fail-safe direction** (§1) and the **scope of the exception** (§5). Once a
drain is trusted to advance checkouts, loosening the gate — treating an empty presence dir as
dormant, dropping G2/G4, or extending the exception to `reset`/feature branches — re-opens an
**unrecoverable, silent** corruption path that no test observes after the fact and that no
gate downstream can catch. The cost is not the code change; it is that the failure destroys
the evidence. That is the DR-359 ADR filter (parent register, mmo-platform) tripping.

**Cheaper to reverse:** the constants' values (30s/120s — paired and named, one edit each
side, parity-tested); the bash-vs-TS duplication strategy (a future single implementation
could replace it); the kill-switch name; and *which* callers consume the gate.
