# DR-051: Artifact-class branch policy — approvables on main, code in worktrees

> **Born `proposed` per DR-029** (acceptance is a separate human act). Amends
> [SPEC-026](../../specs/minspec/SPEC-026-session-presence/requirements.md)'s
> worktree-steer (FR-9) and intersects [DR-034](DR-034.md)/[SPEC-022] approval
> ground truth. Relates to [DR-004](DR-004.md) (tiers) and [DR-047](DR-047.md)
> (criticality classing).

## Context

[SPEC-026](../../specs/minspec/SPEC-026-session-presence/requirements.md) treats the
whole **corpus** — `specs/**`, `docs/decisions/**`, `docs/epics/**`, `docs/domain/**` —
uniformly: worktree-steer (FR-9) would push a second live session editing *any* corpus
file into its own worktree, and the pre-commit backstop (FR-12) guards all of it.

The founder, reviewing SPEC-026, observed that **approvables are not code**:

1. **They are review-in-place living documents**, closer to GitHub **issues** (which
   this project deliberately does *not* branch) than to code. Reviewing a spec means
   reading it in the editor — [review = thinking, one doc at a time] — not diffing a PR.
2. **Worktree/PR isolation hides them.** A spec authored on a branch is only visible by
   "digging around in the PR"; on `main` it is just *there* in the IDE.
3. **Branching multiplies numbering conflicts.** Two sessions each running "new SPEC"
   on separate branches both allocate `max+1` → the same `SPEC-NNN`, colliding at merge.
   This is the same cross-branch id-collision class SPEC-022 FR-1 sidestepped for
   *approvals* by keying on path, not id.
4. A clean candidate invariant: **commit the doc in the approve action, so an unstaged /
   dirty doc reads as unapproved** — a legible, at-a-glance status.

Code, by contrast, genuinely needs branch isolation: it must pass CI before landing, it
is the high-blast-radius artifact class (DR-047), and concurrent code edits want the full
worktree separation SPEC-026 designed.

## Decision

**Split the branch policy by artifact class.** Three coupled rulings:

### 1. Approvables edit on `main`; only code isolates in worktrees

- **Approvables** (`specs/**`, `docs/decisions/**`, `docs/epics/**`, `docs/domain/**`)
  are edited and committed **directly on `main`**, like issues — no branch, no worktree,
  no PR required. They stay visible in the IDE by construction.
- **Code** (everything else — `packages/**`, `scripts/**`, config, hooks) keeps the
  SPEC-026 worktree-steer + PR + CI flow.
- SPEC-026 **FR-9 (worktree-steer) is therefore scoped to code changes only.** The
  SPEC-026 **concurrent-edit guard (FR-12) still applies to approvables on `main`** — it
  serializes two live sessions clobbering the *same* doc, which is exactly the residual
  protection you want once the branch is gone. So approvables lose branch isolation but
  keep same-file collision protection.

### 2. Approval state is the committed hash-matched sidecar — NOT git staging

The "unstaged = unapproved" instinct is right in *feel* but wrong in *signal*: git
staging is **machine-local and ephemeral**. A fresh clone or CI checkout shows every doc
clean-committed regardless of approval — the exact failure mode SPEC-022 fixed by
rejecting the gitignored, per-checkout `approvals.json`. Hanging approval on staging
would re-introduce that bug.

- **Ground truth stays the [SPEC-022] committed, content-hash-matched approval sidecar
  under `.minspec/approvals/`.** ([DR-034](DR-034.md) — unchanged.)
- The **derived, IDE-visible rule** the founder wants is available *from that model*
  already: SPEC-022's hash **voids approval on any content edit**, so
  **"committed & sidecar-hash-matches = approved; modified/dirty = pending"** is true,
  durable, and renders as the ordinary VCS modified-marker. Same visual, grounded in the
  durable sidecar, not the transient index.
- The **approve action commits the doc + its sidecar atomically** ("approve = commit").
  This gives the clean "approved things are committed" reading without making *staging*
  the source of truth.

### 3. Numbering: main-serialized allocation + a duplicate-id gate

Editing approvables on one serialized `main` narrows the id-collision window but does not
close it (two sessions can still both read `max=NN` between pulls). The **durable** fix is
a **duplicate-id validator gate** — `validate` rejects two artifacts sharing an `id`
(FATAL), the symmetric-primitive complement to the existing dangling-ref check. Main-direct
editing is the friction-reducer; the gate is the guarantee.

### Ruleset reconciliation (the sharp constraint)

The branch ruleset activated via #365 requires a PR for `main`, which **fights** direct
approvable commits. GitHub rulesets are **ref-scoped, not path-scoped**, so "docs may push
direct, code may not" cannot be expressed as a path rule on one branch. Resolution:
**add a bypass actor** (the trusted local committer / a MinSpec commit identity) to the
ruleset so approvable commits reach `main` directly, while **CI-required-PR remains the
convention for code**. If a bypass actor is unacceptable, the fallback is approvables via
**auto-merge PRs** — which recovers CI/audit but partly loses the "visible without digging"
benefit; recorded as the rejected alternative.

## Rationale

Approvables and code have different blast radius (DR-047), different review modes (read-in-
place vs. CI+diff), and different collision failure modes (id-numbering vs. line-level
merge). Forcing one branch policy onto both optimizes for neither. The issues-don't-branch
precedent is the tell: the project already treats one class of living document as
main-direct and visible; approvables belong in that class. Meanwhile the durable
approval/numbering guarantees stay where they must — in committed, hashed, validated
artifacts — so nothing here weakens SPEC-022's integrity story; it only relocates *where
authoring happens*, not *what counts as truth*.

## Consequences

**Positive:**
- Approvables are always visible in the IDE; review is read-in-place, no PR spelunking.
- Removes the cross-branch SPEC/DR numbering collision for the common case.
- Simplifies SPEC-026: worktree-steer is a *code* concern; the guard covers docs on main.
- "Approve = commit" yields a legible committed-clean = approved reading, durably.

**Negative:**
- Requires a ruleset bypass actor (or accepting approvable auto-merge PRs) — a repo-config
  change with its own trust surface.
- Two sessions can still race to *create* the same `SPEC-NNN` on main between pulls until
  the duplicate-id gate lands — an interim window.
- A second mental model for contributors ("is this an approvable or code?") — mitigated by
  the fixed path-prefix definition of the corpus.

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| Someone re-hangs approval on staging state | Reads "unstaged = unapproved" as the literal rule | Decision §2 is explicit: sidecar+hash is truth; staging is only a derived hint. INV in SPEC-022 already asserts hash-match; no code keys on index state. |
| Numbering collision in the interim window | Two sessions create SPEC-NNN on main before the gate exists | Ship the duplicate-id `validate` gate (Follow-ups); until then, main-serialization + the existing DR-sequence warning narrow it. |
| Ruleset bypass over-broadens | Bypass actor can push *anything* direct to main, not just approvables | Pair the bypass with the pre-commit corpus check: a direct-to-main push whose diff touches non-approvable paths is warned/blocked by a hook (the guard already classifies corpus vs code). |
| Direct-to-main doc commit clobbers a peer | Two live sessions edit the same doc on main | SPEC-026 FR-12 guard is explicitly retained for approvables (Decision §1). |

## Alternatives Considered

- **Status quo — everything (incl. approvables) on worktrees/PRs.** Rejected: the
  numbering-collision and review-in-a-PR friction the founder raised; over-isolates
  low-blast-radius living documents.
- **"Unstaged = unapproved" as the literal approval signal.** Rejected: staging is
  machine-local/ephemeral; fails on a fresh clone/CI; re-introduces the per-checkout-state
  bug SPEC-022 eliminated. Kept only as a *derived* hint over the durable sidecar.
- **Approvables via auto-merge PRs.** Considered as the ruleset fallback; recovers
  CI/audit but partly loses the "visible in the IDE without digging" benefit that is the
  whole point. Held as the fallback if a bypass actor is refused.

## Costly to Refactor

Medium. The path-prefix definition of "approvable corpus" is referenced by the SPEC-026
guard, the worktree-steer scope, and (proposed) the ruleset bypass hook — changing the set
touches all three, so pin it as one named constant. The approval-truth model is
**unchanged** (SPEC-022 sidecar+hash), so no migration there. The ruleset bypass is a
GitHub-config change, trivially reversible. The duplicate-id gate is a <1-day `validate`
addition. Reversing the whole policy (back to docs-on-branches) is a convention +
scope-of-FR-9 change, not a data migration.
