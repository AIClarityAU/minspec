# DR-061: Native auto-merge on ai-review:pass (interim, until the blast gate is real)

## Context

Auto-merge has been **deny-by-default** since [DR-033](DR-033.md) §6: a PR holds for a
human unless the [SPEC-024](../specs/minspec/SPEC-024-auto-merge-eligibility/requirements.md)
**consequence-hybrid** gate certifies it low-blast. But that gate measures blast via the
[#88](https://github.com/AIClarityAU/minspec/issues/88) consequence analyzers on a real
cross-file index ([#195](https://github.com/AIClarityAU/minspec/issues/195)) — **all
still open**. With no analyzers, the gate scores every change `high` (INV-2: unmeasured
blast = high) → HOLD. So the *designed* auto-merge cannot function today, and won't for
weeks. [DR-058](DR-058.md) hardened that gate further (affirmative evidence required),
widening — not closing — the gap.

Meanwhile the **human merge keystroke is the throughput bottleneck**: PRs come back
`ai-review:pass`, all CI green, and sit waiting for a hand-merge. Measured 2026-07-15:
issue/PR inflow ran ~5× outflow, with the merge step gated on one human.

The repo already has the safety primitive: `ready-to-merge` is a **required status
check**, and it goes SUCCESS only on a **provenance-verified `ai-review:pass`** — the
independent 4-voter panel (reviewer + security + architect + skeptic), fail-closed,
bot-authored (#397/#410/#600). GitHub-native auto-merge keyed on that check needs only a
repo-setting flip.

## Decision

**Enable GitHub-native auto-merge keyed on the `ready-to-merge` required check. A PR
marked `--auto` merges the moment that check (= provenance-verified `ai-review:pass`)
goes green — no human keystroke.** This is an **interim** policy: it supersedes DR-033
§6's deny-by-default *for the ai-review-gated path* until the consequence-hybrid blast
gate is real, at which point blast measurement layers on top (it does not replace it).

### 1. Mechanism

- Repo `allow_auto_merge = true`.
- `dispatch-issue.sh` arms `gh pr merge --auto --squash` on every PR it opens, gated by
  project policy `autoMerge.native` (`.minspec/config.json`, env override
  `MINSPEC_AUTOMERGE_NATIVE`). Default **off** (deny-by-default preserved for projects
  that don't opt in); **on** for this repo.
- GitHub does the merge when every required check passes (`ready-to-merge`, `test`,
  `lint`, `MinSpec SDD validation`, `ai-review`).

### 2. Why the safety story holds without the blast gate

- **The gate is the ai-review panel, not a bare label.** Four independent fresh-context
  voters read the diff as untrusted data; ANY voter requesting changes fails the panel
  closed; the verdict is provenance-checked (bot-only). Auto-merge fires only on that
  panel's PASS.
- **Machinery PRs never auto-merge.** The self-edit guard forces `ai-review:changes` on
  any PR touching `.github/`/`scripts/`, so `ready-to-merge` never greens → those stay
  human/`--admin`. The most dangerous class is still human-gated.
- **Conflicts and red checks block by construction** — they fail required checks, so
  `--auto` never fires.
- **Reversible instantly** — flip `allow_auto_merge` / `autoMerge.native=false`.

The keystroke is *replaced by the independent panel*, not removed from the safety story.
This is a weaker bar than the designed blast gate (no blast measurement), accepted
**explicitly and temporarily** as the interim until #88/#195 land.

### 3. Enabling infra (not part of this policy, but required for it to matter)

Auto-merge makes `main` advance faster, which exposed [#773](https://github.com/AIClarityAU/minspec/issues/773):
the drain self-terminated on staleness, so it couldn't run the pipeline against a
fast-moving `main`. Fixed alongside (the drain now runs each cycle from a self-synced
worktree; staleness self-heals instead of killing the loop). Without that, enabling
auto-merge would starve the very automation that produces the PRs.

## Consequences

**Positive:** clears the merge bottleneck for green PRs; auto-fix→auto-review→auto-merge
becomes a closed loop; the human keeps the irreversible-decision surface (spec/DR
acceptance, machinery merges) while ceding the mechanical merge of independently-reviewed
code.

**Negative:** merges code no human eyed, on a bar (ai-review panel) weaker than the
designed blast gate. Mitigated: panel is adversarial + fail-closed; machinery excluded;
instantly reversible; interim only. A wrong auto-merge is revertable (not a deploy).

## Alternatives considered

- **Keep the human keystroke (status quo).** Rejected — it *is* the bottleneck this DR
  exists to remove; the backlog analysis showed merge-gating on one human doesn't scale.
- **Wait for the consequence-hybrid gate (#88/#195).** Rejected as the *now* answer —
  weeks out; holds every PR until then. This DR is explicitly the interim bridge to it.
- **Turn on consequence-hybrid mode today.** Rejected — non-functional without the blast
  analyzers (holds everything); enabling it changes nothing observable.

## Costly to Refactor

Cheap to reverse — a repo-setting flip plus one config key; undoable in minutes. No
schema, no stored state. Recorded because it changes the HITL/merge posture DR-033 §6
established, and a future reader must see it is an *interim* supersession, not a deletion
of the deny-by-default design.
