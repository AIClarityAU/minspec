# DR-052: Subscription-CLI-default billing, amended for the consumer-OAuth ToS

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). **Amends** the billing-mode *mechanism* of [DR-016](DR-016.md)
> (subscription-default) and [DR-017](DR-017.md) (agent-execute substrate); refines
> [SPEC-019](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md) FR-5 /
> FR-14 (Plan phase). No code — records the decision + mandates the SPEC-019 edits.

## Context

[DR-016](DR-016.md) / [DR-017](DR-017.md) set the product-line default: AI/agent features
authenticate on the dev's Claude **subscription** via `claude -p`, not pay-as-you-go API
tokens. [DR-017](DR-017.md)'s stated *ideal* was a **credential-free sandbox + a host
broker that injects the subscription token** and can Scrooge-route it.

Issue **#74** investigated whether that broker-inject-of-subscription pattern is viable, and
confirmed the answer against Anthropic's **official** documentation (not secondary reporting):

- **[Legal and compliance](https://code.claude.com/docs/en/legal-and-compliance) §
  Authentication and credential use** (verbatim):
  - *"OAuth authentication is intended exclusively for purchasers of Claude Free, Pro, Max,
    Team, and Enterprise subscription plans and is designed to support ordinary use of
    Claude Code and other native Anthropic applications."*
  - *"Developers building products or services that interact with Claude's capabilities,
    including those using the Agent SDK, should use API key authentication… Anthropic does
    not permit third-party developers to offer Claude.ai login or to route requests through
    Free, Pro, or Max plan credentials **on behalf of their users**."*
  - *"Anthropic reserves the right to take measures to enforce these restrictions and may do
    so without prior notice."*
- **Acceptable use:** *"Advertised usage limits for Pro and Max plans assume ordinary,
  **individual** usage of Claude Code and the Agent SDK."*
- **[Agent SDK plan article](https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan):**
  a separate per-user monthly Agent-SDK credit ($20 Pro / $100 Max-5x / $200 Max-20x) was
  **announced then PAUSED** — a top banner reads *"Update June 15: We're pausing the changes
  to Claude Agent SDK usage described below. For now, nothing has changed."* So Agent-SDK /
  `claude -p` usage **currently still draws the normal subscription quota** (5h-window /
  weekly), competing with interactive Claude Code.

**Reading of the ToS.** The prohibition is **narrow**: third-party developers routing
**other users'** requests through consumer credentials ("on behalf of their users"), and
**building a product/service** (→ use an API key). *Individual* use of the genuine `claude`
CLI on one's *own* subscription is explicitly permitted. But a **host broker that
rewrites/reroutes subscription traffic** — [DR-017](DR-017.md)'s ideal — sits squarely on
the "route requests through Pro/Max credentials" line, and **Scrooge-routing subscription
traffic** is exactly the prohibited pattern. So the *posture* (subscription-default) stands;
the *mechanism* must change.

## Decision

1. **Subscription mode = the genuine `claude` CLI, direct to Anthropic — no broker
   rewrite/reroute.** agent-execute (SealBox) spawns the real CLI binary with the dev's own
   subscription credential; it does **not** proxy or Scrooge-route that traffic. Attestation
   whitelists exactly the one injected token ([DR-017](DR-017.md)'s documented fallback
   becomes the **only** sanctioned subscription path; SPEC-019 FR-6 `spawn-token-whitelisted`
   mode).
2. **Default billing mode = subscription-CLI** (#407 D1). Each user runs on their **own**
   subscription — never a shared/multi-tenant account. API-key / Scrooge = explicit opt-in.
3. **Ship both modes** (#407 D2), subscription default.
4. **Broker + Scrooge value = API-key mode only.** The credential-free-sandbox + Scrooge
   routing legitimately applies to **API keys** (always proxyable) or Teams/Enterprise —
   **never** to consumer subscription traffic.
5. **Never the Agent SDK with a subscription token** — shell the `claude` **CLI binary** (the
   ToS names "the Agent SDK" explicitly as a developer/API-key surface).
6. **Multi-tenant / one-sub-many-users is out** (per-user credit, non-pooled).
7. **Quota caveat (paused credit):** because the separate Agent-SDK credit is paused,
   subscription dispatch currently draws the **interactive** 5h/weekly quota — so
   [SPEC-019](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md) **FR-14
   concurrency caps are mandatory, not optional**, and API-key mode is the real headroom for
   heavy dispatch. Re-verify on each ToS sweep (#431); if the pause lifts, subscription
   dispatch gets its own pool and this caveat relaxes.

## Consequences

- **Positive:** ToS-clean by construction; keeps the cheapest default for users (their own
  subscription, no PAYG surprise); preserves MinSpec's air-gap (no MinSpec backend holding
  user tokens — see #442); the broker/Scrooge investment still pays off in API-key mode.
- **Negative / cost:** [DR-017](DR-017.md)'s "credential-free sandbox for subscription mode"
  ideal is **lost** — subscription mode holds the token in the (genuine-CLI) exec context, so
  its attestation whitelists that one token rather than proving zero-cred. Subscription
  dispatch competes with interactive quota until/unless the paused credit lifts.
- **SPEC-019 edits (Plan phase):** FR-5 (billing modes) → subscription = genuine-CLI-direct,
  drop broker-reroute for subscription; FR-14 caps mandatory; the attestation manifest's
  `spawn-token-whitelisted` mode is the subscription path, `no-cred` the API-key/L2 path.

## Alternatives considered

- **API-key default** — rejected as the *default* (shipped as opt-in, decision 3): simplest
  and most ToS-safe, but bills users metered dollars from day one — the billing surprise
  [DR-016](DR-016.md) exists to avoid.
- **Broker-inject + Scrooge-route the subscription token** ([DR-017](DR-017.md)'s original
  ideal) — **rejected**: it is the prohibited "route requests through Pro/Max credentials"
  pattern for consumer plans; enforced by ban without notice.
- **Multi-tenant shared subscription** — **rejected**: consumer credit is per-user and
  non-pooled; explicitly out of the ToS.
