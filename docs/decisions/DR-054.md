# DR-054: Reframe the network posture; adopt one shared reviewer App + an OIDC token-broker

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records the context, decision, and follow-ups; no code ships with this DR.
> **Amends [DR-004](DR-004.md)** (Tiered Network Consent Model): it does not reverse the
> tiers — it re-axes them and demotes "air-gapped" from the product's *primary selling
> point* to a *Tier-0 capability*.
>
> **Reframe accepted in-session 2026-07-11 (Paul Harvey, founder).** The OQ answers are
> folded into SPEC-034 (GH-native review primary; any-repo broker access; Cloudflare Worker).
> Status stays `proposed` until the formal *MinSpec: Accept ADR* keystroke records the

## Context

Three properties have been conflated under one word — **"offline"** — since
[DR-004](DR-004.md), and treating them as one produces a positioning that is both weak and
false the moment a customer uses the product:

1. **Determinism / no-LLM-in-the-decision-path** — the never-wrong signpost, the tier
   classifier, status resolution: pure functions, reproducible across editor/commit/CI/agent
   (Goal **G-6**, "determinism as moat"; DR-039/DR-019). This is a **correctness** property.
   It has nothing to do with network.
2. **Data sovereignty / no MinSpec-initiated egress** — the extension opens no socket,
   holds no token, ships no telemetry, and never *autonomously* sends user artifacts
   anywhere. The load-bearing distinction [DR-050](DR-050.md) already drew: *who initiates
   the egress and whose data goes where.* This is a **privacy** property.
3. **Zero-network operability** — literally runs with no NIC. Only the deterministic Tier-0
   core has this. Every feature a customer would pay for — AI review, the reviewer bot,
   ScroogeLLM routing — needs *a* network hop to *some* LLM.

[DR-004](DR-004.md)'s Rationale leads with property 3 as marketing: *"Tier 0 air-gapped
operation is the free extension's default and primary selling point."* The problem the
founder named: **as a product-wide identity, property 3 is empty.** A customer who takes
"air-gapped" literally and unplugs finds only the deterministic SDD scaffolding — the AI
features that are the reason to adopt MinSpec are all off. That is crippleware, and worse,
it makes the lead claim read as a bait-and-switch.

Meanwhile the customers who actually reach MinSpec **already rely heavily on LLMs** (they
arrive from Claude Code / Cursor). The privacy-conscious subset does not want *no packets* —
they want *their code and prompts not to leave their control*, which they satisfy today by
running a **local LLM** (Ollama / LM Studio / vLLM / any OpenAI- or Anthropic-compatible
endpoint). MinSpec can serve that need directly. So property **2 (data sovereignty), not 3
(zero-network), is the real, sellable privacy story** — and it survives every AI feature,
because the network *destination* is the user's own choice.

**Why this DR now, and not a pure marketing note:** the trigger was concrete. The reviewer
identity (`minspec-sdd[bot]`) turns out to be a **product constant**, not per-deployment
config (#428/#464/#597): MinSpec ships as one published GitHub App that customers *install*,
not one they *create*. Posting a review verdict *as* that identity from customer CI needs
the App's private key to mint installation tokens — which forces a **vendor-operated
token-broker** (the safe pattern) or shipping a shared secret (catastrophic — out). That
broker is a deliberate, small, always-on network dependency for the *team* tier. It is only
defensible if we stop pretending the product is air-gapped and state plainly what is and
isn't sovereign. The invariant reframe and the broker adoption are therefore one decision.

## Decision

### 1 — Retire "air-gapped" as the product identity; lead with two claims that survive AI features

- **RETIRE (as lead marketing):** "MinSpec is air-gapped / offline." Keep zero-network
  operability only as a **Tier-0 capability statement** ("the SDD engine runs with no
  network"), never as the product's headline promise.
- **LEAD WITH, instead, two crisp and defensible claims:**
  - **Deterministic core — no LLM in the decision path.** (G-6 moat; correctness; the
    never-wrong signpost.) Network-orthogonal; unchanged by anything below.
  - **Your data goes only where you send it.** MinSpec never phones home; no
    telemetry; and **AI features work against a user-chosen LLM, including a local one.**
    (Data sovereignty; generalises [DR-050](DR-050.md)'s who-initiates-egress principle.)

### 2 — Re-axis the tiers from "network presence" to "data egress + who chooses the destination"

[DR-004](DR-004.md)'s three tiers stand, re-described by the axis that actually matters:

| Tier | What | Egress posture |
|---|---|---|
| **0 — Deterministic core** | classify, spec lifecycle, validate, signpost, harness | No egress. Runs with no network. |
| **1 — AI features** | Specify/Plan/reviewer drafting, `claude -p` | Needs *an* LLM; **user picks the destination**: Claude subscription / vendor API / **local model endpoint (first-class)**. Data goes where the user points it. |
| **2 — Collaboration** | GitHub-native review, the reviewer bot, ScroogeLLM routing, opt-in telemetry | GitHub metadata egress + an optional vendor broker (§4). **Code and prompts do not flow through the broker** — only an OIDC handshake and label/status writes. |

**Local-LLM support is first-class, not a footnote** — it is the mechanism that makes the
data-sovereignty claim honest and unlocks the regulated/enterprise segment (run entirely
against an internally-hosted / self-hosted model endpoint — on-prem or a private gateway,
*not* per-dev localhost — zero egress to AIClarity). Making the model endpoint
user-selectable via an **arbitrary base URL** (a shared internal endpoint, or localhost) is a
required follow-up ([#633](https://github.com/AIClarityAU/minspec/issues/633)), not aspirational.

### 3 — The Tier-0 lint gate is unchanged

No `http`/`https`/`fetch`/`net` import in `packages/minspec` or `packages/shared`. The
extension process still opens no socket. This DR changes **positioning and tier
description**, not the mechanical Tier-0 boundary [DR-004](DR-004.md)/[DR-050](DR-050.md)
enforce. Determinism (property 1) is explicitly **out of scope** of this reframe and must not
be weakened by it: reframing privacy must never license an LLM into the decision path.

### 4 — Reviewer identity: one shared published GitHub App + an OIDC token-broker; customer-own-app as the enterprise override

- **Default (covers ~95%): one shared, published GitHub App, `minspec-sdd`.** Its bot login
  `minspec-sdd[bot]` is a **product constant** — a public identifier, like an OAuth
  `client_id`. **Ship the name in the vsix; scaffold it as the default reviewer allowlist**
  into every repo's ai-review workflow. This eliminates the per-repo `AI_REVIEW_BOT_LOGINS`
  setup footgun (#597): a fresh repo's gate is configured out of the box.
- **The credential seam is a vendor-operated OIDC token-broker** (specified in **SPEC-034**),
  a sibling of [DR-017](DR-017.md)'s host-side model-access broker: a credential-free client
  (customer CI) proves its identity to the broker via GitHub Actions **OIDC**, and the broker
  — which alone holds the App's private key — returns a **short-lived, repo-scoped**
  installation token. **No code, diff, or prompt passes through the broker** — only the OIDC
  assertion and the scoped-token response; the review itself runs in the customer's CI
  against the customer's chosen LLM (Tier 1). Shipping the App private key into customer repos
  is rejected outright (a shared secret, extractable → forge `ai-review:pass` on every
  install → total provenance collapse — see [DR-033](DR-033.md) §6 / #428).
- **Enterprise override — customer-own-app / GHES.** A customer who will not grant a
  third-party SaaS App, or who runs GitHub Enterprise Server on-prem, registers **their own**
  App (or bot user), points the workflow at **their** broker/app-id, and sets
  **`AI_REVIEW_BOT_LOGINS`** to their bot login. This is the one segment where a
  per-deployment reviewer identity is real — and it is exactly where the variable this whole
  investigation started from **survives, as the documented override knob**, not the default.

**Answering "customer-own-app — for what benefit?":** for the common developer, none — the
shared App is strictly less setup. Its sole benefit is the enterprise/GHES/no-third-party-SaaS
escape hatch above. So it is not the default; it is the override.

## Rationale

- **Honesty is the stronger position.** "Air-gapped" is falsified by the first AI feature;
  "your data goes only where you send it, including nowhere but a local model" is true *with*
  AI features and is what the privacy-conscious buyer actually needs. A claim that survives
  the product's own feature set beats a slogan that collapses on contact.
- **The reframe is an enterprise unlock, not a retreat.** Regulated buyers cannot purchase
  tools that mandate egress to an unapproved vendor. "Runs fully against your organization's
  internally-hosted shared model endpoint, zero data leaves your network, no vendor in the
  decision path, auditable" is a
  procurement checkbox that wins top-down deals (the GTM's budget-owner motion). Air-gap
  *capability* is worth more than the air-gap *slogan*; we keep the capability and drop the
  slogan.
- **The broker is the least-bad way to get dual attribution.** The requirement — human
  activity under the human's name, MinSpec's under `minspec-sdd[bot]`, in a readable GitHub
  timeline — cannot be met without a distinct bot identity that posts under its own token
  ([#464](https://github.com/AIClarityAU/minspec/issues/464): a human-run reviewer can't even
  GH-natively approve its author's PR). One shared App gives every install the same
  zero-setup identity; the broker is what lets customer CI wield it without holding the key.
  It touches only the verdict-posting step, sends no user code, and is opt-in with the whole
  team tier.
- **It is a sibling of an already-accepted pattern.** [DR-017](DR-017.md) already blessed a
  broker as "the single … seam" where a credential-free client points at a fixed endpoint and
  the broker injects the real credential. This DR applies the same shape to GitHub App tokens.

## Consequences

- **[DR-004](DR-004.md) is amended:** its Rationale/Consequences framing of Tier-0 as the
  *"primary selling point / air-gapped"* is superseded by §1–§2 here. The tier *mechanism*
  and the Tier-0 lint are unchanged; the *positioning* and the *axis description* change.
- **Constitution + AGENTS.md + marketplace/site copy must be re-stated** off "air-gapped" and
  onto the two lead claims. Left stale, they contradict this DR while reading as live — the
  cross-doc drift [DR-004](DR-004.md)/[DR-029](DR-029.md) already warn about (§Follow-ups).
- **`AI_REVIEW_BOT_LOGINS` changes role:** from a mandatory per-repo variable to a shipped
  default (`minspec-sdd[bot]`) with the variable retained only as the enterprise override.
  This resolves #597 and re-scopes #428's "human setup step" to the enterprise path only.
- **A new always-on vendor service exists** (the broker, SPEC-034): its uptime gates
  team-tier merges and its compromise is a forge-everywhere event, so it carries its own
  security review and SLO. The solo/local path (Tier 0/1, `approvals.json`, review under the
  user's own `gh` per [DR-050](DR-050.md)) has **no** such dependency and stays fully
  sovereign.
- **Local-LLM support becomes a committed requirement**, not a maybe — it is load-bearing for
  the reframed privacy claim.

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| Reframe read as "privacy no longer matters" → an LLM creeps into the decision path | A contributor treats "we're not air-gapped" as licence to call a model in the classifier/resolver | §3 is explicit: determinism (property 1) is out of scope of this reframe and unweakened; the no-LLM-in-decision-path rule (G-6) stands, separately enforced. |
| Broker becomes a single point of failure / forge-everywhere target | It holds the shared App private key; a compromise mints tokens for every install | Dedicated security review + SLO for SPEC-034; short-lived repo-scoped tokens (blast radius per-request); the guard still fails **closed** on any unverifiable pass ([DR-033](DR-033.md) §6) so a broker outage blocks merges (safe) rather than green-lighting them. |
| "Air-gapped" copy left live somewhere after the reframe | Marketplace listing, README, site, constitution each paraphrase it | Land the copy re-statement in the change that adopts this DR; track each surface as a follow-up issue (§Follow-ups); RCDD cross-artifact gate (DR-003). |
| Enterprise override path rots (untested, undocumented) | Default is the shared App; the own-app path gets no dogfood | Keep `AI_REVIEW_BOT_LOGINS` + app-id as first-class documented config in SPEC-034; a contract test asserts the workflow reads an overridden identity, not a hardcoded one. |
| Local-LLM "support" is claimed but never built → the privacy story is a lie | The reframe leans on it; it is easy to defer | File it as a tracked, Phase-gated requirement now (§Follow-ups); the data-sovereignty claim may not ship in copy until the local-endpoint path is real (evidence discipline / DR-003). |

## Alternatives considered

- **Keep "air-gapped" as the lead; treat AI features as the exception.** Rejected: the
  exception *is* the product. The slogan is falsified on first use and reads as bait-and-switch
  — the founder's crippleware point.
- **Customer-own-app as the default (no shared App, no broker).** Rejected: it forces a
  GitHub-App setup on every user for a benefit only the enterprise/GHES segment needs, and
  reintroduces the exact per-deployment `AI_REVIEW_BOT_LOGINS` friction #597 flagged. Kept as
  the override, not the default.
- **Ship the shared App private key in the vsix / as a repo secret.** Rejected outright: a
  shared, extractable secret → anyone forges `minspec-sdd[bot]` `ai-review:pass` on every
  install → the provenance guarantee [DR-033](DR-033.md) §6 exists to provide collapses.
- **Post review verdicts under the user's own `gh` (no bot, no broker).** Rejected for the
  team tier: activity then lands under the human's name (no dual attribution) and cannot
  GH-natively approve its author's PR ([#464](https://github.com/AIClarityAU/minspec/issues/464)).
  Retained as the correct **solo/local** posture (Tier 0/1), which needs no broker.
- **Drop the privacy story entirely, just disclose network use.** Rejected: data sovereignty
  + local-LLM is a genuine differentiator and enterprise unlock; discarding it throws away the
  budget-owner motion to save a positioning paragraph.

## Costly to Refactor

**High for the positioning, moderate for the mechanism. ADR-filter: YES.** The *reframed
public claims* — once on the marketplace listing, the site, and enterprise procurement
answers — are a trust/compliance commitment, not a code edit; walking them back is a
trust event (same asymmetry [DR-004](DR-004.md) named for "air-gapped"). The *shared-App +
broker* is load-bearing once team-tier repos authenticate through it: the App name is
near-irreversible to rename after publish, and moving off a vendor broker later (e.g. to
per-customer apps) is a re-architecture of the auth seam, not an edit. Genuinely cheap to
change: which model endpoint Tier-1 points at (user config), and the enterprise override
wiring (documented config, isolated behind the same allowlist variable).
