# DR-020: Tier-gated Risks & Mitigations section

## Context

Specs and DRs were being written without an explicit **Risks & Mitigations**
section — SPEC-012 shipped its first draft without one. A blanket "every doc must
have risks" rule was proposed.

A blanket mandate collides with MinSpec's load-bearing principle: **ceremony
proportional to tier** (T1 = one-sentence spec; the whole "just enough spec"
pitch). Forcing a risks matrix onto a T1/T2 artifact is exactly the over-ceremony
the tool exists to remove. Meanwhile DRs already carry **Consequences** +
**Alternatives considered**, which is most of the risk-thinking already — they
lack only an explicit risk *header*, not the reasoning.

So the real decision is *where the section is required*, not *whether it exists*.

## Decision

**Risks & Mitigations is a tier-gated required section.**

| Artifact | Risks & Mitigations |
|---|---|
| **DRs (all)** | **Required.** A decision without stated risk is half-made. Sits alongside Consequences/Alternatives. |
| **Specs T3 / T4** | **Required.** Real surface, multi-file blast radius — risk analysis earns its ceremony. |
| **Specs T1 / T2** | **Optional.** Tier-proportionality wins; a one-sentence/spec+plan artifact is not worth a risk matrix. Author may add if a real risk exists. |
| **Epics** | **Optional / lightweight.** A one-line risk in the Goal is enough; no mandated table. |

### Shape

A table — `# · Risk · Likelihood · Impact · Mitigation` — with each mitigation
tracing to a concrete FR / decision section / invariant (not hand-waving). The
table in SPEC-012 (R1–R7) is the reference exemplar.

### Enforcement — soft, never blocking

The frontmatter/structure validator gains a **warning** (not a hard block) when a
required-tier artifact lacks the section, mirroring the existing soft `epic:`
unresolved-ref rule (DR-013 §4). Ceremony stays proportional: a missing risks
section nags, it does not stop work. Consistent with the advisory-not-blocking
posture (the only hard gate is DR-012 approval).

## Consequences

- **Templates updated** to emit a Risks & Mitigations stub at the required tiers:
  the **Create ADR** command template, and the `specify` / `plan` skill scaffolds
  for T3/T4. T1/T2 spec scaffolds omit it.
- **Validator** gains a soft rule (warn) for DRs + T3/T4 specs missing the section.
- **Backfill** of the section into the 19 existing DRs is a separate, optional
  follow-up — not auto-done (avoids a 19-file churn commit); offered, not forced.
- No change to tier classification or to any existing required phase.

## Alternatives considered

- **Blanket — every doc, all tiers.** Rejected: violates tier-proportionality,
  adds ceremony to T1/T2 with low marginal value, contradicts "just enough spec".
- **DRs only.** Rejected as too narrow: T3/T4 specs carry the largest
  implementation blast radius and benefit most from explicit risk analysis.
- **Hard validator block.** Rejected: only the DR-012 approval gate blocks; every
  other MinSpec check is advisory. A blocking risks-rule would be inconsistent and
  punish drafts.
