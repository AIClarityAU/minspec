# DR-042: Outcome metrics before engagement — sequence the trust-measurement build

## Context

A review-telemetry audit (2026-06-26, 6-agent workflow, claims verified to
`file:line`) asked whether MinSpec can today (a) **prove** the value of SDD and
(b) **tune** the "just enough human" thesis (DR-029) — e.g. "this project has a
high error rate; can we point to the cursory reviews that were rubber-stamped?"

Verified state: **the outcome half of the trust-measurement layer (reviewer
identity, committed ground truth) has since landed (DR-034/SPEC-022, 2026-06-13);
the remaining unbuilt gap is the content snapshot + char-rework diff (SPEC-017
M1/M2) plus the engagement/outcome layers below.**

- Approval stores only a `sha256` hash, no content snapshot
  (`packages/minspec/src/lib/approval.ts:38-44`) → char-rework % cannot be
  computed at all.
- Reviewer identity ALREADY exists: `approval.ts` captures `git config
  user.email` and records it as `ApprovalRecord.approvedBy`.
- Ground truth is ALREADY committed: attributed, path-keyed sidecar records
  live under `.minspec/approvals/` (DR-034/SPEC-022, landed 2026-06-13) —
  gitignore only still excludes the legacy single-file `.minspec/approvals.json`.
- Zero engagement signal (no review-start, dwell, scroll, approve-latency).
- No outcome stream (bugs / rework) linked to approvals → the "high error rate"
  end of the correlation is uninstrumented.

The design that answers all of this already exists — SPEC-017 (Trust Dashboard:
M1 char-rework %, M2 superseded-chars, M3 time-to-approve), SPEC-014 (review
webview), SPEC-018 (spec custom editor), DR-029 (just-enough-human). The thinking
is done; the build is not. The founder's instinct — *own the review surface to
capture engagement* — is correct and already specified, but the **specific
tactic** ("intercept markdown-file opens and PR opens and redirect them") splits
into one path that is technically impossible and one already deliberately scoped
down:

- "Intercept markdown opens via Ctrl-P" — **no VS Code API exists** (SPEC-018
  researched and rejected this).
- "Redirect PR opens" — **impossible**: PRs live on github.com; an extension
  cannot redirect a browser. The realistic equivalent is a MinSpec PR-review
  queue (#182 signpost + #211 explorer), planned not built.
- A global `**/*.md` custom-editor hijack was already rejected as "the exact
  intrusive over-reach the just-enough-human thesis sells against," and collides
  with the keyboard-first preference, the never-wrong invariant, and the Tier-0
  air-gap (DR-004).

SPEC-017 also already forbids any **standalone** time/scroll verdict (its "proxy
trap"): a 2-minute approval of a co-authored doc is not a skim; a tab open at
lunch is not reading. Engagement may only ever be shown crossed with rework
outcome (FR-9). Over-indexing on "track engagement thoroughly" risks shipping the
exact false-accusation metric that destroys trust on first misfire.

The question is therefore not *what to build* (SPEC-017 et al. answer that) but
**in what order** — and the audit's load-bearing finding is that the order is
currently implicit and wrong-way-round (the visible, exciting work is the webview
and engagement capture; the load-bearing work is snapshot + outcome diff).

## Decision

**Outcome metrics are the moat; engagement is the garnish. Build the trust layer
in that order.**

1. **Outcome-first (no surface ownership needed).** Ship SPEC-017 **M1/M2** before
   any engagement or webview work: extend `ApprovalRecord` with a content
   **snapshot** (FR-1) and a char-level **rework diff** (FR-2). These are
   deterministic, recomputable from files, and are the actual proof mechanism for
   "rubber-stamp → rework." They require no engagement tracking and no intercept.

2. ~~**Ground truth must be committed and attributed.**~~ **Already satisfied** —
   reviewer identity (local git `user.email`, Tier-0, no network) and a
   committed, content-free approval ledger (path-keyed sidecars under
   `.minspec/approvals/`, hash + tier + approver + timestamps, never body text)
   landed via DR-034/SPEC-022 (2026-06-13), ahead of this DR. The
   no-committed-ground-truth premise no longer holds. Full body snapshots stay
   gitignored under `.minspec/snapshots/`. (→ #300 tracks any remaining cleanup)

3. **Engagement comes after M1, and only as the scoped, opt-in surface.** Do
   **not** build a global markdown/PR intercept (impossible and anti-thesis).
   The engagement source for M3 is the already-scoped, opt-in, **spec-paths-only**
   custom editor (SPEC-018), built **after** M1 exists to cross it against
   (FR-9). The PR equivalent of "owning the review" is a first-class PR-review
   queue (#182/#211, depends on SPEC-012), not interception.

4. **"No standalone engagement verdict" is a Tier-0 invariant.** Whatever
   engagement code lands must enforce in code (not just prose) that time/scroll is
   only ever displayed paired with rework outcome (SPEC-017 FR-9 proxy-trap).

5. **The tuning experiment must be two-armed.** DR-029's anticipation-rate study
   measures only the LLM half. A **human-side arm** — per-tier review-depth (M3,
   opt-in) joined to downstream error/rework-rate (M1 + RCDD bug tags) — is
   required before any claim that review can safely be tuned *down*. (→ #301)

This DR sets sequence and rejects the global-intercept tactic. It does not
re-specify SPEC-017/014/018, which stand.

## Rationale

- **The join is the product.** "Prove value" and "tune just-enough-human" both
  reduce to one join: (review act) → (downstream outcome). The review act today
  is a contentless, anonymous, gitignored hash — the weakest possible left side.
  Snapshot + identity + committed ledger fix the left side and need no UI at all.
- **Engagement without outcome is a liability, not an asset.** FR-9 already says
  engagement is meaningless — worse, defamatory — unless crossed with rework.
  Building capture first inverts the dependency and invites the trust-destroying
  false "you skimmed this" misfire.
- **The exciting surface is the cheap part; the boring schema is the moat.** The
  audit flagged the snapshot schema migration as "Costly to Refactor #1" — doing
  it first de-risks everything downstream. The webview is bounded UI on top.
- **The founder re-derived SPEC-017 from scratch.** That is itself a signal: the
  spec corpus answering the most strategic product question was not visible at the
  moment it was needed — a point for the next-task signpost (DR-019) and the
  Approvable queue (DR-041) to surface trust work by priority.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Snapshot storage bloats the repo | Body snapshots gzipped + gitignored under `.minspec/snapshots/`; only the content-free ledger is committed. |
| Reviewer identity reads as covert telemetry about a human | Identity is local git config the user already set; recorded in a visible, auditable, committed ledger; no network (DR-004). Invariant #8 (auditable visibility) satisfied. |
| Sequencing slips and engagement gets built first anyway | FR-9 dependency stated here as a hard ordering constraint: M3/SPEC-018 may not merge before M1. |
| "Outcome is the moat" read as "engagement is worthless" | It is not — engagement is the X-axis for tuning (point 5). It is *sequenced after*, not dropped. |
| Committed ledger leaks spec titles/structure | Ledger stores specId + hash + tier + approver + timestamps only — no body, no section text. Same exposure as existing spec filenames already in git. |

## Consequences

- SPEC-017 is promoted ahead of SPEC-014/018 in build order; M1/M2 become the
  next implementation slice of the trust layer.
- `ApprovalRecord` schema gains `approver` + (via SPEC-017 FR-1) a snapshot
  reference — a back-compat migration (records without `approver` stay valid).
- A new committed artifact appears: `.minspec/approval-ledger.jsonl`
  (content-free, append-only).
- The global markdown/PR-intercept idea is recorded as **considered and
  rejected**, so it is not re-derived a third time.
- DR-029's validation study is reframed from one-armed (LLM-only) to two-armed
  (LLM + human), gated on M1.
