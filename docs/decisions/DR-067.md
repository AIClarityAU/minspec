# DR-067: A 'planning' status for approved-but-pre-implementation specs

## Context

[`deriveStatus`](../../packages/minspec/src/lib/lifecycle.ts#L107) — the authoritative,
approval-aware SIGNPOST derivation ([DR-034](DR-034.md)) — returns `implementing` for
**any approved spec that is not all-done**:

```
if (approvalState !== 'approved') return 'specifying';  // INV-1
if (allRequiredDone(phases))      return 'done';
return 'implementing';                                   // approved + NOT all-done
```

Approval sets the first build-band phase (`plan`) to `in-progress` (`phasesForApproval`),
so a spec approved while still at **Plan/Tasks — implement phase pending, zero code** derives
to `implementing`. That is a **[DR-003](DR-003.md) false signpost** — the project's stated
worst defect: the signpost claims code is being written when none exists. The deterministic
literal-vs-derived validator cannot catch it (literal == derived); only the AI-review panel
did (PR [#879](https://github.com/AIClarityAU/minspec/pull/879), SPEC-040). The same trap is
latent for **SPEC-041** and every future approval, and it is aggravated by the #829/#460
requirement to add `implements:` past Clarify (that edit stales the prior approval, so
re-approving mid-Plan re-triggers the flip).

The code is also **broader than its own documented rule** (the FR-4 table says "approved +
**implement** in progress → implementing"); it fires on `plan`/`tasks` in-progress too. And
the status vocabulary (`new | specifying | implementing | done`) has **no value for
"approved, but implementation not yet started"** — `specifying` means *unapproved* (INV-1),
so it cannot be reused.

## Decision

**Add a fifth lifecycle status, `planning`: "approved; in Plan/Tasks; the implement phase has
not started." `deriveStatus` returns `implementing` only once the implement phase is actually
`in-progress`/`done`; otherwise an approved-but-incomplete spec derives `planning`.**

```
  if (allRequiredDone(phases)) return 'done';
+ if (phases.implement === 'in-progress' || phases.implement === 'done') return 'implementing';
+ return 'planning';   // #886 — approved, pre-implement: honest, not a false signpost
- return 'implementing';
```

The boundary is *implement-started vs not* — it preserves the genuine `implementing` signal
and keeps INV-1 (unapproved → specifying) and INV-6 (explicit terminal wins) untouched.

### 1. The `planning → implementing` resolver mapping (the blast-radius neutraliser)

The next-task resolver keys on its **own** `ResolverSpecStatus` enum, fed through
`SPEC_STATUS_MAP` (`satisfies Record<SpecStatus, ResolverSpecStatus>`). **`planning` maps to
the resolver's `implementing`.** Consequence: every resolver seam — answer-OQ surfacing,
`isAdvancing`, spec-ahead-of-epic, flooring — sees an approved-planning spec **exactly as
today**, so there is **zero change to next-task behaviour and zero risk of dropping an
open-question node or a gate-violation**. An approved-planning spec genuinely *is* in-flight;
treating it as `implementing` in *resolver* space is correct. `planning` is confined to the
human/UI/validator surface (a new tree lane + an honest `status:` line).

### 2. The literal-status writer moves in lockstep

The approval path stamps the literal `status:` via the phases-only `getSpecStatus`
(`= implementing`). To avoid a `status.mirror-drift` warning on every approved-planning spec,
the writer (`advanceSpecToImplementing`) switches to the approval-aware
`deriveStatus(phases, 'approved', …)` — so a freshly-approved pre-implement spec is written
`status: planning`. `getSpecStatus` itself is **unchanged** (it is a freeze-gate twin — see §3).

### 3. HARD SAFETY CONSTRAINT — do NOT touch the freeze-gate twins ([DR-012](DR-012.md))

The "unapproved spec blocks its impl code" freeze gate is **phase-based**, not status-based:
it uses `phase_intent_status` ([spec-gate.py](../../scripts/hooks/spec-gate.py)), a mirror of
`getSpecStatus`, which maps *any* current phase in {plan, tasks, implement} to `implementing`
and **never reads `deriveStatus` or the `SpecStatus` enum**. This fix is confined to
`deriveStatus` + the enum surface + the writer, so the gate is unaffected — an unapproved spec
at plan/tasks still lands in-range and still blocks its declared impl files.

**The three phase-position twins — `getSpecStatus`, `phaseIntentStatus` (migrate-approvals),
and `phase_intent_status` (spec-gate.py) — MUST keep plan/tasks in the `implementing` band.**
Narrowing any of them to match `deriveStatus`/`planning` would drop unapproved plan/tasks
specs out of the freeze range → their impl code becomes editable → **silently reopens the
spec-gate freeze hole** that [DR-012](DR-012.md)/[DR-031](DR-031.md) close. There is no parity test binding the twins, so the drift
would be silent. Each twin carries a code comment forbidding the alignment; a follow-up issue
adds the missing parity test.

### 4. Enum wiring (loud-forced, except two silent sites)

`planning` is threaded through: `SPEC_STATUSES`; `SPEC_STATUS_MAP` (`planning: 'implementing'`,
compile-forced); a new **Planning** tree lane (test-forced); `LIFECYCLE_RANK`
(`…specifying:1, planning:2, implementing:3, done:4` — **silent if missed**, so covered by a
test); `SPEC_STATUS_WORDS` (**silent if missed**); `findActiveSpec`; frontmatter-completion;
`statusIcon`. `SPEC_DONE_OK` stays `{implementing, done}` — deliberately: a "SPEC-X
implemented" prose claim against a `planning` spec now correctly warns.

## Consequences

**Positive:** the signpost stops lying — an approved, un-built spec reads `planning`, not
`implementing` (DR-003). SPEC-041 (and every future approval) approves honestly. The
`planning→implementing` resolver map means the fix is *behaviour-neutral* for the next-task
engine.

**Negative / costs:**
- A new status value fans out to ~10 sites (mostly compile-/test-forced) + a new tree lane.
- The ~6 already-approved-pre-implement specs (SPEC-022/026/027/031/034/040) show a
  **non-fatal** `status.mirror-drift` WARN (Validate-command + tree lane only, never CI) until
  their literal `status:` is re-mirrored. Their corpus migration is **deferred** (a body
  `**Status:**` edit is inside the canonical hash → would stale their approval; frontmatter
  `status:` is hash-stripped and safe, but Rule-11 parity ties the two) — tracked as a
  follow-up so the migration is done deliberately, not as a hash-staling side effect here.
- SPEC-022's FR-4 rules table gains a `planning` row — **deferred** likewise (editing an
  approved spec's body stales it); tracked.

## Alternatives considered

- **Reuse `specifying` for approved-pre-implement.** Rejected — overloads INV-1 (`specifying`
  *is* "unapproved"), swapping one false signpost for another, and mis-ranks in `LIFECYCLE_RANK`.
- **A deterministic gate on the literal only** (no `deriveStatus` change). Rejected — the lie
  is in the *derived* signpost (tree lane + resolver + validator all read `deriveStatus`); a
  literal-only gate leaves `deriveStatus` still returning `implementing`.
- **Narrow the phase-position derivation (`getSpecStatus`/gate twins) too.** Rejected —
  reopens the spec-gate freeze hole (§3).

## Costly to Refactor

Not <1-day reversible — this is why it is a DR. Once specs are stamped `status: planning` on
disk and approvals recorded against that lifecycle, reverting needs a corpus migration; the
new value is a cross-package vocabulary commitment (`SpecStatus` ⇄ `ResolverSpecStatus` map).
Recorded accordingly.
