# DR-064: The layering contract is machine-enforced, deterministically and offline

## Context

The 2026-07-13 malleability audit ([#690](https://github.com/AIClarityAU/minspec/issues/690), #689–#696, #742) found MinSpec's layered architecture is held **by convention alone**, and the convention has **already leaked** — verified on `main` at authoring time and re-verified for this record:

- **Two `lib→views` inversions** (pure logic depending on the UI layer): `packages/minspec/src/lib/active-spec.ts:6` *value*-imports `fromFrontmatter`/`computeProgress` from `../views/status-bar`; `lib/approval-diff.ts:21` *type*-imports `SpecNode` from `../views/spec-tree-provider`. Grep confirms these are the **only two** `lib→views` edges and there are **zero** `lib→commands` edges.
- **Three type-held import cycles** among `lib` modules, each currently held runtime-safe by exactly one `import type` edge — **zero runtime cycles today**, but each one careless `type`→value edit away from a real require cycle, ungated.
- **Seven `lib/` files import `vscode`** (`resolve-folder`, `diagnostics`, `active-spec`, `ai-usage-detector`, `approval-diff`, `bridge`, `active-adr`) with deep API use — so `lib` is not vscode-free today.
- **Core data access in a UI file:** the recursive `listSpecs()` (a pure fs scan) sits in `views/spec-tree-provider.ts` and is consumed by `approve`/`approve-active`/`validate` — "refactor the tree view" silently reaches the approval pipeline.

The enforcement surface is a **single root `eslint.config.mjs`** (flat config, `files: ['packages/*/src/**/*.ts']`) carrying only `tsPlugin.configs.recommended` + `no-unused-vars`. It has **no** `no-restricted-imports`, **no** cycle check, and — load-bearing for the choices below — **no** `parserOptions.project` (so it cannot see type-only imports today).

SPEC-040 turned this audit into requirements (FR-1..FR-6, AC-1..AC-7, OQ-1..OQ-4). The spec's own **Costly to Refactor** section names *"the layering contract encoded in the eslint config"* as an architectural artifact: once code depends on the tree passing it, loosening it later silently re-opens the door and cannot be re-audited in under a day. That is the DR-359 ADR filter (parent register, mmo-platform) tripping — a costly-to-reverse choice with no DR recording it. **This DR is that record**, and it resolves the three load-bearing open questions (OQ-1/OQ-2/OQ-3) so Clarify can close; the cheap-to-reverse one (OQ-4) is left to Plan by design.

The thesis is SPEC-038's, proven: **deterministic gates change behaviour; conventions do not** (the RCDD commit-msg gate forced `Root cause:` compliance up at install — DR-003). This is the validator-asymmetry insight (#137) applied to **code structure** — where SPEC-038/#460 made the **spec→code** contract enforceable, this makes the **code→code layering** contract enforceable.

## Decision

**Encode MinSpec's layer-import contract as machine-enforced gates — eslint `no-restricted-imports` for direction/depth rules and an in-repo, dependency-free cycle checker for acyclicity — running fully offline. Ship the `error`-level rules only after the small prerequisite refactors (SPEC-040 FR-4/FR-5) make the current tree pass them green; ship the vscode-purity rule at `warn` until the 7 coupled files relocate (#830).**

### 1. The cycle gate is an in-repo checker adapting `detectCycles`, not a dev-dependency (resolves OQ-1)

The acyclicity gate (FR-2) is a **small in-repo checker** that adapts the proven iterative three-color DFS at `packages/shared/src/next-task.ts:327` (`detectCycles` — explicit-stack, no recursion, O(V+E), deterministic neighbour order) from the `depends_on` artifact DAG to the **module import graph** of `packages/minspec/src`. It reads the value-import edges (a type-only edge is not a runtime edge) and fails on any back-edge.

- **Rejected — `madge --circular`:** pulls transitive dependencies (a new supply-chain surface) and network at install; violates INV-1 (offline) in spirit and DR-005's supply-chain posture.
- **Rejected — `dependency-cruiser`:** heavier still, same supply-chain objection, more config than the rule needs.
- **Chosen — in-repo checker:** zero new dep, no network, reuses code MinSpec already trusts and tests. This is the choice that best satisfies **INV-1 (constitution #1 — offline core)** and it is the reason a *cheap-to-reverse tool choice* is nonetheless recorded: the **principle** (a structural gate must not add supply-chain surface) is load-bearing and would otherwise be re-litigated every time someone reaches for madge.

> Corrects an earlier SPEC-040 draft claim that "the audit prototyped a checker" — it did not; `detectCycles` is the real reusable asset.

### 2. Ban type-only `lib→views`/`lib→commands` edges too, not just value edges (resolves OQ-2)

The direction rule is the simplest possible invariant: **`lib` imports *nothing* — value or type — from `../views` or `../commands`.** A type-only carve-out would be a standing exception a reader must remember and a `type`→value flip could later exploit silently. FR-5 already removes the one type edge (`SpecNode` in `approval-diff.ts`), so banning type costs nothing on the shipped tree while making the rule unambiguous. (The `@aiclarity/shared` deep-import ban — barrel-only, enforcing **DR-014** — is preventive: zero violations today.)

### 3. Use `@typescript-eslint/no-restricted-imports` with `allowTypeImports: false`, adding `parserOptions.project` (resolves OQ-3)

Banning type edges (§2) requires the rule to *see* type-only imports, which base eslint `no-restricted-imports` cannot. The mechanism is `@typescript-eslint/no-restricted-imports` with `allowTypeImports: false`, which needs `parserOptions.project` wired into the root config. **The `@typescript-eslint` plugin + parser are already dependencies** (the config imports them today) — so this adds type-aware linting cost, not a new dependency.

- **Accepted cost:** type-aware linting is slower and requires the tsconfig project reference. Measured at Plan (R3); if the cost proves unacceptable the documented fallback is value-only `no-restricted-imports` (i.e. fall back to OQ-2 "allow type-only" — a strictly weaker but still-directional rule), **not** abandoning the gate.
- **Rejected — `eslint-plugin-boundaries`:** a heavier new dependency for a rule two `no-restricted-imports` blocks express directly.

### 4. Ship `error` rules green — prerequisite refactors first; vscode-purity ships at `warn`

The direction + depth rules land at `error` **only after** FR-4 (extract the recursive `listSpecs` → `lib/spec-catalog.ts`) and FR-5 (reverse the two inversions) make the tree pass — so `main` is never red on landing (AC-5). The **vscode-purity** rule (FR-3) ships at `warn` because 7 `lib/` files legitimately violate it today; it flips to `error` when **[#830](https://github.com/AIClarityAU/minspec/issues/830)** relocates them. Asserting the warn count by test makes that flip a one-line, test-verified change (AC-3). Scope is `packages/minspec/src/{lib,views,commands}` + `extension.ts`; `src/test/**` and `src/__benchmarks__/**` stay exempt (they legitimately import views/vscode).

Dissolving the three type-held cycles (FR-6) is **fragility-reduction, not gate-required**: the cycle gate flags *runtime* cycles and there are zero, so it ships green without touching them. Recorded here so a future reader does not mistake "gate is green" for "no fragility remains."

### 5. Not resolved here — the home for relocated utilities (OQ-4) is a Plan choice

Where `fromFrontmatter`/`computeProgress`, the `SpecNode` field-subset, and (if FR-6 runs) `slugify` land — one `lib/util` module vs. beside their most-related lib module — is **cheap to reverse** and therefore deliberately **left to Plan**, not decided here. Leaning: co-locate beside the most-related lib module rather than a grab-bag `lib/util`. Recording only the costly-to-reverse choices keeps this DR honest to the ADR filter.

## Consequences

**Positive:**
- The layering contract becomes a deterministic, offline, no-LLM-in-the-decision-path gate — reproducible across editor/commit/CI/agent (**DR-054** G-6 determinism-as-moat).
- The two `lib→views` inversions and any future one are caught at `npm run lint`; a fresh runtime cycle is caught by the CI gate; `@aiclarity/shared`'s barrel boundary (**DR-014**) is finally enforced, not merely documented.
- `listSpecs` moves to Tier-0 `lib`, so the approval pipeline stops depending on a high-churn UI file (AC-6); the latent `spec-panel` flat-scan bug is surfaced for a decision, not silently carried.
- Same "gates not conventions" mechanism as SPEC-038/#460 and the RCDD gate — one coherent code-safety story.

**Negative:**
- Type-aware linting adds lint time and a tsconfig project reference (R3; fallback documented in §3).
- The `error` rules encode an allowed-edge set that a future legitimate edge could collide with; mitigated by shipping only after the tree is clean (AC-5) and reviewing the allowed edges at Plan.
- A maintainer could try to dodge a rule with a per-file `eslint-disable` — explicitly a violation the review must reject (SPEC-040 **INV-4**); the `warn` posture is by design, never by disable.

## Alternatives considered

- **Do nothing — keep the convention.** Rejected: the audit is the disproof — the convention already leaked (2 inversions, 3 one-edit-away cycles). This is the #137 asymmetry: the linter checks style but never asserts the layering rule.
- **A dependency-based cycle checker (madge / dependency-cruiser).** Rejected — supply-chain surface + offline (INV-1) cost for a graph walk MinSpec already implements (§1).
- **Value-only direction rule (allow type-only `lib→views`).** Rejected as the *default* — kept only as the measured fallback if type-aware linting is too slow (§3). A standing type-only exception is a rule a `type`→value flip can silently defeat.
- **Ship the vscode rule at `error` now.** Rejected — reds `main` on 7 known files; the relocation is #830's separate scope. `warn` + asserted count is the honest interim (§4).
- **Block the gate on first dissolving the three cycles (FR-6).** Rejected — the gate flags runtime cycles and there are zero; requiring a refactor the gate does not flag is scope the spec explicitly deferred.

## Costly to Refactor

**Costly:** the encoded layer-import contract itself (which layer may import which) — once code depends on the tree passing it, loosening it re-opens the boundary invisibly and cannot be re-audited in under a day. This is the artifact that earns the DR (DR-359 ADR filter — parent register, mmo-platform). **Cheaper to reverse:** the checker *implementation* (§1 — swappable), the type-vs-value strictness (§2/§3 — the documented value-only fallback), and whether FR-6's cleanup happens now or later. The utility-home choice (§5) is cheap and deferred to Plan.
