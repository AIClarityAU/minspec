## Context

Building VS Code extensions for two related but distinct problems:
1. SDD (Spec-Driven Development) productivity tooling — helps developers write just enough spec
2. LLM cost optimization proxy — reduces LLM API costs via middleware chain

Initial concept was one extension ("LLMProxy"). Market research showed SDD tooling is an underserved niche with strong SEO opportunity and no dominant player.

## Decision

Split into three separate extensions:
- **MinSpec** (`aiclarity.minspec`) — free SDD tool, top-of-funnel (Tier 0 / air-gapped)
- **ScroogeLLM** (`aiclarity.scroogellm`) — freemium proxy, monetization vehicle
- **Agent Execute** (`aiclarity.agent-execute`) — Tier 1 agent dispatch / execution substrate. Productized dispatch ships here, NOT inside MinSpec. See DR-015 (packaging), DR-004 (tier model), DR-008 (dispatch security), DR-017 (execution substrate).
- **MinSpec Pro** (`aiclarity.minspec-pro`) — extension pack, premium upsell

Use SDD as the primary development methodology for building both extensions (dogfooding).

## Rationale

- SDD tool has lower scope and faster path to v1 → ships first
- Two free tools create broader surface area for discovery
- Extension pack is natural upsell when both are valuable independently
- Dogfooding SDD builds credibility and surface issues early

## Consequences

- Three extension IDs, three marketplace listings (agent-execute may list without its own domain)
- ScroogeLLM specs not started until MinSpec ships or reaches stable v1
- Shared classification engine extracted to `packages/shared`
