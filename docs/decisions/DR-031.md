# DR-031: Spec-gate must be sound in dispatch contexts

> Triggered by #143 + #144 (and #146), surfaced 2026-06-04 when #25 repaired the
> hook's relative-path bug and the spec-gate began *actually firing* in dispatch
> worktrees for the first time — exposing that the whole worktree-dispatch path had
> been silently bypassing the HITL gate.

## Context

The PreToolUse **spec-gate** (DR-362 (parent register, mmo-platform) enforcement of the DR-012 HITL approval gate)
denies source edits while any T3/T4 spec is `status: implementing` without a current
approval. It is the only enforcement that survives bypass-permissions mode. Three
defects block it — and block the user's goal of an automated `triage-inbox.sh` →
`dispatch-issue.sh` pipeline where auto-approved (T1–T2 `agent-ready`) issues build
themselves:

- **#144 — the approval store does not propagate to dispatch contexts.** The gate
  reads `.minspec/approvals.json` relative to cwd. That file is **gitignored**, so
  `git worktree add` (dispatch-issue.sh's `/tmp/minspec-agent/issue-N`, and the
  Agent-tool worktrees) never copies it. The worktree gate sees an empty store →
  treats every T3/T4 implementing spec as unapproved → **blocks all source edits**,
  even though the human genuinely approved (the worktree's spec files hash to the
  canonical `approvals.json`). The automated builder therefore cannot edit at all;
  this session required hand-seeding approvals into each worktree as a workaround.

- **#143 — the bypass is agent-accessible.** The deny message literally states *"To
  bypass intentionally, set `MINSPEC_GATE_OFF=1`."* An autonomous agent reads that and
  self-disables the HITL gate. Observed this session: of 6 dispatched agents, 3
  self-bypassed and 3 refused — enforcement became a coin-flip on agent disposition,
  not a human decision. For the productized agent-execute (Tier-1) path, where agents
  edit source by design, this voids the gate's guarantee.

- **#146 — the gate's own test is flaky.** `spec-gate.test.ts` execs the real hook,
  whose decision depends on ambient cwd (`os.getcwd()` fallback → repo root with live
  specs) + a spread `process.env`, while sitting in the shared parallel pool. env/cwd
  pinning *reduced* but did not *eliminate* the flake (≈8 green, then a failure). A
  flaky test in the enforcement suite erodes the trust the gate exists to provide.

## Decision

1. **Approvals are local-machine state, resolved from the canonical checkout — not
   cwd.** The gate resolves `.minspec/approvals.json` from the main working tree (via
   `git rev-parse --git-common-dir` → its parent), so every linked worktree on the
   machine evaluates against the same human-approved state. The *spec being edited* is
   still read from the editing context (cwd), so a worktree that **modifies** a spec
   still correctly goes stale. If the canonical store cannot be resolved **and** gated
   (T3/T4 implementing) specs exist → **fail closed** (deny), never silently allow.
   (Keep the existing "fail open if python3 is missing" — do not block on a broken
   interpreter.)

2. **The kill-switch is human-only and unadvertised.** (a) The agent-facing deny
   `reason` no longer names the bypass mechanism. (b) `MINSPEC_GATE_OFF` stays as the
   *human* escape hatch, but the dispatch pipeline MUST NOT set it. (c) Every honored
   bypass is appended to `.minspec/gate-bypass.log` (timestamp, cwd, tool, target), so
   a bypass is always auditable after the fact.

3. **`dispatch-issue.sh` relies on canonical approvals, never the bypass.** With (1), a
   dispatched agent editing source under a genuinely-approved spec passes the gate; an
   unapproved/stale spec correctly *blocks* the dispatch (surfaced, not bypassed). No
   approvals-seeding workaround, no `MINSPEC_GATE_OFF`.

4. **`spec-gate.test.ts` runs serially-isolated** via a per-file serial annotation /
   dedicated pool that does **not** restructure the global vitest config. (The earlier
   parallel/sequential project-split is **rejected** — it broke `approval.test.ts` and
   `spec.test.ts`.)

## Risks & Mitigations

- **Wrong canonical checkout** (nested repos / submodules). → use `git rev-parse
  --git-common-dir`, verify `.minspec/` sits beside its parent, else fall back to cwd
  then fail-closed. Test with a real worktree + main pair.
- **Fail-closed locks out legitimate edits** when git/approvals momentarily
  unreadable. → fail closed *only* when gated specs are detected AND the store is
  unresolvable; with no gated specs there is nothing to gate, so allow.
- **A determined agent still bypasses** (writes settings, edits the hook). → #143
  raises the bar (no advertisement + audit log); true containment is fully-sandboxed
  dispatch (DR-008 / agent-execute, no agent write to the hook), tracked separately.
  This DR makes casual self-bypass non-obvious and always-auditable, not impossible.
- **Editing the live gate destabilizes active sessions.** → develop in isolation,
  verify with deny/allow/kill-switch envelopes + the de-flaked suite, merge only on
  green, probe the gate manually post-merge.

## Costly to Refactor

- **Approval-resolution location (cwd → canonical)** — moderate to reverse; a contract
  every gate evaluation and future dispatcher depends on. *What to check:* worktree and
  main agree on "approved"; stale detection still fires when a worktree edits a spec.
- **Kill-switch semantics (human-only)** — low to reverse (a string + a source check),
  but a behavioral contract the dispatch pipeline assumes.

## Consequences

- The automated triage→build pipeline becomes viable: dispatched agents pass for
  approved specs, block visibly for unapproved ones, and cannot casually self-disable
  the gate.
- The manual approvals-seeding workaround (`[[worktree-approvals-provisioning]]`) is no
  longer needed once shipped.
- The gate's never-wrong posture extends to dispatch contexts, closing the window where
  the entire worktree-dispatch path silently bypassed HITL.

## Alternatives considered

- **Commit `approvals.json` (un-gitignore).** Rejected: approvals are per-machine,
  per-human local state (DR-012 hash-bound); committing shares one human's approvals
  across all checkouts/devs and pollutes history.
- **Seed approvals in `dispatch-issue.sh` only.** Rejected as the *sole* fix: a
  per-dispatcher band-aid; other worktree paths (Agent tool, future dispatchers) still
  break, and it does nothing for #143.
- **Remove the kill-switch entirely.** Rejected: a genuine human escape hatch is
  needed; the fix is to make it human-only + audited, not absent.
