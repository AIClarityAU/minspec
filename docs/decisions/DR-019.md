# DR-019: Deterministic next-task resolution (DAG, not LLM)

## Context

A session asked for "a prioritised list of docs / specs / epics / DRs I need to
approve", then reframed: not a list ‚Äî **the single next task** the human dev must
do, with an optional expansion to sense the pipeline. The follow-up question was
the crux: **can priority be reliably assessed by programmatic means (a DAG)
instead of an LLM?**

Three approval/status mechanisms feed "what must I decide next":

- **specs** ‚Äî DR-012 content-hash approval (approve before implement),
- **epics** ‚Äî `status: proposed ‚Üí active` promotion,
- **ADRs** ‚Äî `status: proposed ‚Üí accepted`.

Plus [SPEC-010](../../specs/minspec/signpost-correctness/requirements.md)'s
within-feature phase signpost, whose Open Question #1 already asked how to order
*globally* when several items pend at once.

The signpost is EPIC-002's core promise: MinSpec is *always opinionated about
what happens next*, and **never wrong**. An LLM-derived next-task cannot meet
"never wrong" ‚Äî it is non-reproducible, untestable, and would make the product's
central pointer a probabilistic guess.

## Decision

**Next-task priority is computed deterministically ‚Äî a pure function over a
dependency DAG of structural state ‚Äî and never by an LLM.**

### 1. All inputs are structural

Epic status + `order`, spec status/tier/phase + approval-hash, ADR status, and
the membership edges (`spec.epic`, `adr.epic`) plus the fixed phase DAG
(`specify‚Üíclarify‚Üíplan‚Üítasks‚Üíimplement`) and the DR-012 approve‚Üíimplement gate.
Nothing requires semantic understanding.

### 2. Total order = (severity class, epic.order, artifact-id)

Severity classes, highest first: **gate-violation** (downstream work past an
unsatisfied gate) ‚ñ∏ **blocked-ready** (gate whose clearance unblocks the next
phase under an active epic) ‚ñ∏ **promote-parent** (proposed epic with waiting
members) ‚ñ∏ **pending** (remaining proposed/unapproved). The **next task** is the
minimum of this total order. Fully topological + deterministic; unit-testable.

### 3. Subjective weight lives in explicit fields, not inference

The one thing a DAG cannot derive ‚Äî relative importance between *independent*
branches (e.g. "ScroogeLLM is future") ‚Äî MUST be lifted into explicit frontmatter
(`epic.order`, a new `deferred:` / optional `priority:`), edited by the human.
Prose in CLAUDE.md is invisible to the engine until lifted. The engine computes
structure; the human sets weight; **the LLM infers neither at resolve time.**

### 4. Two queues stay separate

The next task is the **human's** decision queue (approve / promote / accept /
author). The **agent/LLM dispatch** queue (DR-015/017) is a separate substrate
and never appears as a human next task.

### 5. Coherence is a deterministic precondition

A child must not outrank its parent's status (spec `implementing` under a
`proposed` epic; ADR `accepted` under a `proposed` epic). Such breaches are
detected structurally and surface as the top gate-violation next task ‚Äî the same
class that caught, by hand, the two live inconsistencies in the triggering
session (stale epic INDEX; SPEC-004 implementing under proposed EPIC-004).

### 6. Tier 0, one engine

A single pure function in `packages/shared` (no `vscode`, no network; DR-004,
DR-014), consumed by the status-bar signpost, the explorer rollup, and CI alike ‚Äî
so every surface agrees on the one next task.

## Consequences

- New spec [SPEC-012 Next-Task Resolver](../../specs/minspec/next-task-resolver/requirements.md)
  carries the contract. Resolves SPEC-010 OQ#1.
- New optional frontmatter: `deferred:` (and possibly `priority:`) on epics/specs.
  Backward compatible ‚Äî absent = not deferred.
- The explorer pane gains a single next-task signpost + collapsible pipeline,
  rather than the rejected dedicated "Approvals list" view. May restructure the
  existing Specs/Decisions panes (downstream UX spec).
- The LLM's only sanctioned role in priority is *suggesting* `order`/`deferred`
  values for the human to accept ‚Äî never computing the live next task.

## Alternatives considered

- **LLM ranks pending items** ‚Äî rejected. Non-reproducible, untestable, violates
  EPIC-002 "never wrong" and DR-004 Tier-0. A probabilistic core signpost is a
  trust liability.
- **Render a flat prioritised list (no single next task)** ‚Äî rejected as the
  primary surface. A list is a backlog; collapsing it to one pointer is the
  product. List survives only as an opt-in expansion (SPEC-012 FR-6).
- **Dedicated unified "Approvals" tree view** ‚Äî rejected in-session as too
  list-heavy; superseded by the single-next-task signpost direction.
- **Infer branch priority from CLAUDE.md prose / edit recency** ‚Äî rejected as
  hidden, non-deterministic weight. Weight must be explicit data (decision ¬ß3).
