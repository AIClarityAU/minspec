# DR-018: Multi-license scheme — MPL-2.0 core, MIT apps, CC-BY-4.0 content

## Context

The monorepo ships open-source artifacts of three different kinds, and a single
repo-wide license is wrong for all of them at once. The packages are not peers:
`packages/shared` (`@aiclarity/shared`) is a reusable **library** — the T1–T4
complexity classifier engine plus the contract types — and it is the project's
core IP (the classifier is the differentiator over GitHub Spec Kit, and its
measurement direction is still an open research question, so improvements to it
have outsized value). The two VS Code extensions (`minspec`, `scroogellm`) and
the pack are **end-user applications** built on top of that library; the
whitepaper and site copy are **prose**, not code.

A permissive license everywhere (MIT) lets a competitor fork the classifier into
a closed product and contribute nothing back. A strong copyleft everywhere (GPL)
is a "poison pill" for commercial adoption and contradicts the free, frictionless,
Spec-Kit-compatible posture the extensions depend on for distribution. Neither
extreme fits a repo that is simultaneously a protectable library and adoption-led
apps.

Decision framework adopted from baryodev, "Why I'm choosing MPL-2.0 for my core
libraries and MIT for everything else": weak (file-level) copyleft on the core
engine you want to keep open, permissive on everything else to remove commercial
friction. Reviewed and confirmed by external licensing expert (2026-06-01).

## Decision

Multi-license the repo. Each package's own `LICENSE` file is authoritative; the
root `LICENSE` is a pointer/summary.

| Artifact | License | Rationale |
|---|---|---|
| `packages/shared` | **MPL-2.0** | The one reusable library + core IP. File-level copyleft: modifications to these source files stay open; commercial tools may still depend on/link it. |
| `packages/minspec` | **MIT** | End-user app. No library to protect; maximize install/adoption. |
| `packages/scroogellm` | **MIT** | End-user app. Its moat is the proxy measurement secret-sauce, protected by **not publishing it**, not by license. |
| `packages/extension-pack` | **MIT** | Trivial meta-pack referencing the two extensions. |
| `docs/`, `sites/` copy, whitepaper | **CC-BY-4.0** | Prose/content — CC is the right tool, not a code license. (`LICENSE-CONTENT`) |
| `scripts/`, root tooling | **MIT** | Dev-time monorepo tooling. |

`license` fields set per `package.json` (SPDX): `shared` → `MPL-2.0`,
`minspec`/`scroogellm`/`extension-pack` → `MIT`, root → `SEE LICENSE IN LICENSE`.

### Why this is safe across the dependency edge

`minspec`/`scroogellm` (MIT) depend on `shared` (MPL-2.0). MPL-2.0 is **file-level**
copyleft: the larger MIT-licensed work may combine with MPL files freely; only the
MPL'd source files themselves carry the share-modifications obligation. This is the
standard "MPL library inside a permissive app" pattern — no friction, no relicensing
of the extensions, no obligation on consumers beyond the shared files they modify.

## Consequences

- Forking the classifier into a proprietary product now requires publishing changes
  to the `shared` source files — the improvements-stay-open guarantee the core IP
  warrants — while Spec Kit or any commercial tool can still link `@aiclarity/shared`.
- The extensions keep a zero-friction permissive license; nothing about MPL on the
  dependency changes their MIT terms or their consumers' obligations.
- Liability: every LICENSE here (MIT, MPL-2.0, CC-BY-4.0) carries an "AS IS" warranty
  disclaimer for the *software/content*. This does **not** cover the marketing claims
  on the sites (e.g. ScroogeLLM savings claims) — a separate website Terms of Use is
  still needed for that surface (tracked separately).
- Site JSON-LD `SoftwareApplication.license` for both extensions now points at the MIT
  license; the earlier placeholder is resolved.
- Publishing `@aiclarity/shared` to npm under MPL-2.0 is unambiguous; the Exhibit-A
  header may be added to individual source files later but the LICENSE file at the
  package root is sufficient to license the package.

## Alternatives considered

- **MIT everywhere.** Simplest. Rejected: leaves the classifier (core IP) unprotected
  against closed forks that give nothing back.
- **MPL-2.0 everywhere.** Protects the engine but needlessly imposes file-level
  copyleft on the end-user apps, adding adoption friction with no library to guard.
- **GPL/AGPL on the core.** Strongest copyleft, but a commercial poison pill and
  incompatible with the frictionless, embed-anywhere posture of the extensions.
- **Apache-2.0 for the core** (instead of MPL). Permissive + explicit patent grant,
  but permissive — improvements can still disappear into proprietary forks. MPL's
  weak copyleft is the specific property wanted for the classifier.

## Status

Accepted (external licensing expert confirmed, 2026-06-01). Implemented: per-package
`LICENSE` files (`shared`=MPL-2.0; `minspec`/`scroogellm`/`extension-pack`=MIT),
root `LICENSE` summary, `LICENSE-CONTENT`=CC-BY-4.0, `package.json` SPDX `license`
fields, site footer License links, and site JSON-LD `license` fields. Follow-up: a
website Terms of Use disclaiming code-breakage/marketing-claim liability for both
sites is **not** covered by these licenses and remains open.

Triggered by: session request — "shouldn't we note the license on the minspec.dev
website? … i'm doing some research into which license (MIT/Apache/CC/etc) we should
use" — resolved against the baryodev MPL-2.0/MIT framework, expert-confirmed.
