# DR-048: Memory-poisoning + reality-check split by tier; promptfoo dev-time only

> **Born `proposed` per [DR-029](DR-029.md)** — acceptance is a separate human act.

## Context

Three founder asks arrived as "add to one of our vsix": (1) **memory-poisoning
defences + scanning**, (2) inherit useful **promptfoo** capabilities, (3) a
**reality-checker** that cross-checks a *response* — verify a URL exists *and*
contains what the response claims — delivered as a footer link or an opt-in
always-on mode.

"One vsix" is the wrong frame. The asks cross the established tier boundary
([DR-004](DR-004.md)): **MinSpec is Tier-0, air-gapped — no network**, so anything
that fetches a URL or runs a model cannot live in its core. Two of the three already
have lineage: untrusted-input handling is gated by the T0 invariant in
`packages/minspec/tests/scripts-untrusted-tools.test.ts` ([DR-008](DR-008.md):
*"an allowlist cannot sandbox a dev agent"*), and a spec/DR-scoped reality-check agent
already exists ([DR-030](DR-030.md), advisory, two-lens decorrelated per
[DR-029](DR-029.md)).

Material new fact: **promptfoo (MIT) agreed to be acquired by OpenAI (Mar 2026)**
([plugin catalog](https://www.promptfoo.dev/docs/red-team/plugins/)). `LICENSE` on
`main` is still MIT (verified 2026-06-30), but a vendor-neutral / frugality product
taking a hard runtime dependency on an OpenAI-owned tool is a positioning + lock-in
risk.

## Rationale

1. **Tier-0 air-gap is load-bearing** (Invariant #1 / DR-004). URL-content
   verification and red-team model calls are network ops; they belong to the Scrooge
   proxy (response path + network) and SealBox/agent-execute (agent runtime), never
   MinSpec core. A *static* pattern scan needs no network, so it — and only it — is
   legal in MinSpec.
2. **Defence ≠ attack-generator.** promptfoo's memory-poisoning plugin *generates
   attacks to test* defences; it is not itself a runtime guardrail. So promptfoo is a
   harness, and the defences are ours to build.
3. **Validator-asymmetry, both directions.** MinSpec already gates *egress* (leaking
   its own DR/SPEC refs) but not *ingress*. A static injection scanner is the missing
   ingress counterpart — same class, gate both ways.
4. **Advisory, never auto-block / auto-edit.** never-wrong signpost + DR-030's
   rejection of "sanitise injection text" mean every layer here surfaces and lets the
   human decide; nothing auto-mutates content or hard-blocks a response.
5. **Inherit patterns, not the vendor.** promptfoo's eval + red-team patterns are
   durable; the tool is acquisition-clouded. Use it dev-time, pinned, offline — and
   keep the value even if we drop it.

## Decision

1. **promptfoo = dev-time / CI harness only — never a shipped runtime dependency**
   (any vsix). Pin/vendor a known-MIT version; do not auto-track upstream
   post-acquisition; run offline. Uses: classifier (n=120, κ=0.80) + Scrooge-routing
   eval, a red-team pre-publish gate (sibling to Bumblebee, [DR-005](DR-005.md)), and
   indirect-prompt-injection tests for the untrusted-issue/spec threat. → **minspec #372**.
2. **Memory-poisoning defence splits by tier:**
   - **MinSpec (Tier-0):** a *static* injection/poison **scanner validator** over
     local artifacts (specs, DRs, memory files) — advisory, no network, no auto-edit.
     → **minspec #371**.
   - **SealBox / agent-execute (Tier-1):** *runtime* hardening (temporal/segregated
     memory, periodic state-vs-guardrail validation, no-credential isolation) stays
     under existing [DR-030](DR-030.md)/[DR-008](DR-008.md). **No new decision** — this
     DR only records that runtime defence is out of MinSpec's scope.
3. **Reality-checking splits by target:**
   - **Spec/DR content** → unchanged, already [DR-030](DR-030.md) (agent-execute,
     advisory, two-lens).
   - **Response-level** (URL exists + contains claimed content; checkable claims) →
     **Scrooge proxy** (response path + network + `X-Scrooge-Tag` measurement).
     **Mode A default:** a footer `🔍 N claims worth checking → validate` link/button
     (on-demand, keyboard-triggerable per the RSI rule) — keeps cost/latency off the
     default path. **Mode B opt-in:** always-validate when the claim-detector fires.
     → **scroogellm #40**.

## Consequences

- **No network enters MinSpec core** — the scanner is pure-static; all fetch/model
  work is Scrooge- or SealBox-side. Invariant #1 intact.
- **Marketing surface.** Response-level validation is the concrete mechanism behind
  the never-wrong signpost + the Stella Lorenzo validator-gate angle; it strengthens
  the "we catch fabricated claims" pitch with a shipped feature, not a slogan.
- **promptfoo eval** (#372) can regression-test both the static scanner's pattern
  precision and the Scrooge claim-detector's precision/recall — measurement-as-moat.
- **Reversible:** dropping promptfoo costs ~0 (dev-only, pinned); the ingress scanner
  and response validator are additive advisory layers.
