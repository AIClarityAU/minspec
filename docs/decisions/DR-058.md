# DR-058: Auto-merge low-blast requires affirmative evidence, not absence of a high signal

## Context

### The hole (#490 — mechanism, not restatement)

`classifyBlast(signals, touchesExportedSurface)`
([auto-merge.ts:188](../../packages/minspec/src/lib/auto-merge.ts#L188)) returns `'low'`
when `signals` is **empty** — the recognition loop simply never runs, and the function
falls through to `return 'low'`. Two more facts make that empty set fully *eligible*, not
merely low-blast:

1. `deriveTouchesExportedSurface([])` → `false`, so the `unmeasured-blast` gate
   ([auto-merge.ts:385](../../packages/minspec/src/lib/auto-merge.ts#L385) —
   `if (!reachKnownLow(signals) && touchesExportedSurface) failed.push('unmeasured-blast')`)
   **never fires** (its `touchesExportedSurface` conjunct is false).
2. `reachKnownLow` **always returns `false` in v1**
   ([auto-merge.ts:240](../../packages/minspec/src/lib/auto-merge.ts#L240) — "No
   affirmative low-reach signal type exists in v1") — so it can never *add* safety; it can
   only ever gate via conjunct (1).

So a **code** change that trips no analyzer — an internal logic edit, an auth check, a
money-path branch with no *exported-surface* touch and no manifest/boundary/sensitive-term
hit — emits an empty consequence-signal set and classifies **low-blast, eligible,
auto-merged with no human eyes**. The #414 (manifest) and #422 (boundary) injections cover
only *non-code* blind spots; nothing covers an analyzer **false-negative on code**.
Deny-by-default currently stops at *known high-signal names*, not at *absence of any
signal*. This is the same class as the standing classifier finding: **absence of a signal
is being read as evidence of safety.**

### Why the obvious fix is wrong

The tempting one-liner — "empty signal set on a code change ⇒ `high`" — is **safe but
useless**, and the reason is load-bearing: because `reachKnownLow` is always `false` and
there is **no affirmative low-blast signal type in v1**, the empty-set→`low` fall-through is
the *only* route to eligibility that exists today. A *genuinely* low-blast change (a typo
fix, a comment, a docs edit) also emits **zero** signals — the analyzers only speak up for
*dangerous* patterns. So "empty ⇒ high" would make auto-merge fire **≈ never** — functionally
identical to `pr-gate`-always. It would close #490 by disabling the feature, not by fixing
the measurement.

The issue names the real fork: *"a floor on unanalyzed/opaque code, **or** requiring an
affirmative low-blast signal (not mere absence)."* The first is the safe-but-useless path
above (and a size floor would also be the exact anti-pattern the classifier finding
warns against). The second is the answer.

## Decision

**Low-blast classification requires AFFIRMATIVE evidence. Absence of a signal is
`unmeasured`, and unmeasured is a hold — never `low`.**

1. **`classifyBlast` returns `low` ONLY when a positive low-blast signal is present.** An
   empty set, or a set carrying no affirmative low-blast certification, classifies as
   **unmeasured → `high`** (deny-by-default). Recognized high signals still return `high` as
   today; the change is that *silence no longer means safe*.

2. **Introduce affirmative low-blast analyzers (SPEC-023) that positively certify a change's
   CONSEQUENCE is low — categorically, never by diff size.** A starter catalog (small,
   provably-safe, expandable):
   - **docs/text-only** — every changed path is documentation/prose (`.md`, `docs/**`,
     `LICENSE`, etc.) and no code/manifest/config file is touched;
   - **test-only** — every changed path is a test file (already recognized by the hollow-
     findings scanner's `TEST_FILE_RE`), no product source touched;
   - *(future, each its own analyzer + T0 test:)* comment/whitespace-only diffs, generated-
     artifact-only diffs, an explicit repo-configured low-blast path allowlist.
   Each analyzer keys off the **real git diff** — the `changedFilePaths` authority added for
   [#489](https://github.com/AIClarityAU/minspec/issues/489) — so an affirmative low-blast
   claim is checked against what actually changed, never an agent's self-report.

3. **Consequence, not size (SPEC-004 guard).** An affirmative low-blast signal certifies that
   the *kind* of change is inherently low-consequence. A 500-line docs PR is low-blast; a
   3-line auth edit is high-blast. No analyzer added under this DR may gate on line count,
   file count, or churn magnitude — that is the classifier anti-pattern this project already
   rejected. If a change class cannot be *categorically* certified low-consequence, it does
   not get an affirmative signal, and it holds.

4. **Deny-by-default is preserved end-to-end (INV-1).** `eligible: true` still requires
   *every* condition; this DR makes "blast is low" itself require positive proof rather than
   the absence of a red flag. There is no path where "we couldn't tell" resolves to eligible.

## Consequences

- **#490 closes without disabling auto-merge.** Unrecognized code → no affirmative signal →
  hold (the hole shuts); docs-only / test-only changes → affirmative signal → still
  eligible (the feature stays useful for the change classes it can actually vouch for).
- **Consistent with SPEC-033 FR-7's `pr-gate` scaffold default.** Until the affirmative
  analyzers ship, the affirmative-signal set is empty, so *everything* holds — which is
  exactly the safe, `pr-gate`-equivalent posture FR-7 already ships. The two decisions
  compose: nothing auto-merges until **both** the four merge-holes close **and** affirmative
  low-blast analyzers make specific change-classes eligible.
- **Depends on #489 (landed).** The affirmative analyzers check the real `changedFilePaths`;
  without #489's diff authority they could be fed a fabricated file set. #489 merged
  2026-07-14, so this is unblocked.
- **Scope honesty.** This DR does not claim to solve "prove an arbitrary code change is
  low-blast" — that is the open classifier problem (SPEC-004), deliberately *not* attempted.
  It draws the safe line: certify the change *classes* we can, hold everything else.
- **A test T0 invariant falls out:** `classifyBlast([], _) === 'high'` (the exact inversion
  of the #490 bug) plus one eligibility test per affirmative analyzer.

## Follow-ups (tracked — DR-023)

- **Implement in SPEC-024 + SPEC-023** (the actual #490 fix, after this DR is accepted):
  invert `classifyBlast`'s empty-set default to `high`; add the docs-only + test-only
  affirmative analyzers keyed on `changedFilePaths`; T0 tests (`classifyBlast([]) === 'high'`;
  docs-only → eligible; internal-code-no-signal → hold). Filed as the #490 implementation
  sub-task. **#490 stays open until that lands** (this DR is design only).
- **Affirmative-analyzer catalog is expandable** — each new low-blast class (comment-only,
  generated-only, repo-allowlisted paths) is its own analyzer + T0 test + a line in this
  decision's catalog; none may gate on size.
- **Revisit the FR-7 flip criteria (SPEC-033 D-7-rev)** — the `pr-gate → consequence-hybrid`
  default flip should additionally require that at least the docs-only/test-only affirmative
  analyzers exist, else the flip changes nothing (everything still holds).
