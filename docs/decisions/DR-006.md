title: Auto-detect and offer MinSpec setup actions (replace manual Ctrl-Shift-P workflow)
status: accepted

## Context

Users were reporting confusion because MinSpec required them to manually run three palette commands ("Initialize SDD Structure", "Refresh Harness Files", "Classify Task Complexity") to bootstrap a project. Users would forget to run them, or wouldn't know they existed. The marketplace README was also cluttered describing each command, and there was no onboarding flow.

User feedback (2026-05-28):

> "wouldn't we really just want the minspec extension to detect what needs to happen and just do it? same for Initialize SDD Structure, Refresh Harness Files, and Classify Task Complexity. maybe this way we don't need an onboarding page, and we can remove some of these Ctrl-Shift-P commands and doco if they're now redundant."

The manual-command model violates the MinSpec invariant "ceremony proportional to complexity" — setup ceremony itself was disproportionate to the trivial bootstrap each command performs. A T1-style detect-and-offer flow eliminates the discovery problem without removing user control.

## Decision

On extension activation, MinSpec runs three detection checks and surfaces a non-blocking toast offering the corresponding setup action when needed:

1. **Missing `.minspec/`** → offer `minspec.init` ("Initialize SDD Structure").
2. **Stale harness templates** (template hash mismatch) → offer `minspec.initRefresh` ("Refresh Harness Files").
3. **No classification for current diff** → offer `minspec.classify` ("Classify Task Complexity").

For each toast the user can:

- **Accept** — runs the corresponding command.
- **Not Now** — defers; toast may re-appear next activation.
- **Don't ask again** — persists a dismissal flag to `.minspec/preferences.json`, scoped per-check.

### Command visibility

The three commands (`minspec.init`, `minspec.initRefresh`, `minspec.classify`) remain registered and callable but are hidden from the Command Palette via `menus.commandPalette: when: "false"` in `package.json`. They are invoked exclusively via toast actions or programmatically (`vscode.commands.executeCommand`).

### Master toggle

A setting `minspec.autoBootstrap.enabled` (default: `true`) disables the entire detect-and-offer system for power users who want manual control. When disabled, no toasts fire and detection logic is skipped on activation.

## Rationale

- Lower onboarding friction: users don't need to learn three commands or discover them via Ctrl-Shift-P.
- The marketplace README simplifies — no command catalog to maintain.
- Preserves invariant #5 ("user override always wins"): every toast is dismissible, every prompt is non-blocking, master toggle exits the flow entirely.
- Preserves invariant #1 ("no AI dependency"): detection is pure file-system checks (existence, hash compare, diff inspection) with zero AI involvement and zero network calls (Tier 0, DR-004).
- Aligns with invariant #4 ("ceremony proportional to complexity"): setup ceremony now scales to zero for projects that don't need it, and to one click for projects that do.

## Consequences

- **Positive**: Lower onboarding friction. Users don't need to learn three commands. README simpler. Aligns with "ceremony proportional to complexity" — including the ceremony of setup itself.
- **Negative**: Toasts are interruptive. Users with `autoBootstrap.enabled = false` lose discoverability of the commands (mitigation: docs mention they can still run via `vscode.commands.executeCommand` or rebinding).
- **Risk**: Detection logic for "stale harness" depends on template hashing — false positives could spam users. Mitigation: "Don't ask again" persistence + master toggle + per-check granularity in `preferences.json`.
- `package.json` settings schema gains `minspec.autoBootstrap.enabled`.
- `.minspec/preferences.json` becomes a recognised user-state file (added to project artifact storage conventions).
- Marketplace README loses the per-command sections; an "Auto-bootstrap" paragraph replaces them.

## Implementation

- `packages/minspec/src/lib/auto-bootstrap.ts` (new) — detection + toast orchestration.
- `packages/minspec/src/extension.ts` — activation hook invokes auto-bootstrap when `minspec.autoBootstrap.enabled` is true.
- `packages/minspec/package.json` — adds setting, hides three commands from palette via `menus.commandPalette` when-clauses.

## Invariant impact

Reinforces:

- **#1 (no AI dependency)** — detection is pure file-system, no AI calls.
- **#4 (ceremony proportional to complexity)** — setup ceremony now scales with project need rather than being a fixed three-step ritual.
- **#5 (user override always wins)** — "Don't ask again" per check + master toggle preserve full user control.

No invariants weakened.
