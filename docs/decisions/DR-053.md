# DR-053: Paragraph-addressable reference scheme

> **Born / stays `proposed` per DR-029** — acceptance is a separate human act
> (*MinSpec: Accept ADR*). **This is v2**: it widens the original cross-project
> reference decision ([#58](https://github.com/AIClarityAU/minspec/issues/58) /
> [#500](https://github.com/AIClarityAU/minspec/issues/500)) down to the
> **paragraph level** and changes the **token shape** for typing ergonomics. The
> **prefix table** and the paragraph-**type** set below are a *proposal to redline*
> at Accept-ADR — the founder explicitly invited "think up prefixes for the other
> para types." The v1 `MS-SPEC-019` code + `.minspec/project-prefixes.md` seed +
> the `@aiclarity/shared` `project-prefix` module already shipped; updating them to
> v2 and migrating the corpus are **tracked follow-ups, not applied here** (the
> founder's rule this session: *renumber / record now, reformat the corpus later,
> before creating many more specs*).

## Context

Two problems, one scheme.

**1. Cross-project id collision (v1, unchanged premise).** SDD ids are allocated
**per repo** — [DR-027](DR-027.md) split scrooge into its own private repo with its
own local register, and every minspec-enabled repo numbers `SPEC-`/`DR-`/`EPIC-`
from 1. So `SPEC-001` exists in minspec **and** scrooge **and** sealbox and means a
different artifact in each. Within a repo that is unambiguous; the ambiguity bites
only when a reference **leaves** its repo.

**2. Sub-document references are unscoped and inconsistently prefixed (new).**
Approvables are full of references to *items inside* other approvables — a risk row,
an FR, an open question — and today those are:

- **Unscoped.** A bare `FR-3` at the top of scrooge `DR-019` means "FR-3 *of the doc
  this ref lives in / the doc the reader has in mind*" — reader-resolved, ambiguous
  out of context. There is no consistent way to say "FR-3 *of* MS-SPEC-019."
- **Inconsistently prefixed.** The same *kind* of paragraph carries different
  prefixes across docs: open questions appear as `OQ-n` **and** `FR-OQ-n`; risks as
  `R1`…`R7`; costly-to-refactor items as unlabeled `Costly #N`; consequences,
  follow-ups, deviations, alternatives are frequently **unlabeled** entirely. (No
  `D-n` scheme exists — the founder's recollection of one was a false memory; an
  exhaustive corpus + git-history search found none.)

There is **no addressable, uniform way to point at "a paragraph / bullet / row in
any approvable."** That address is the precondition for two downstream features:
a **trace graph** (fine-grained links between items, not just between files —
narrower than [DR-038](DR-038.md)'s artifact-level next-task graph) and the
**lozenge / hover-card renderer** (SPEC-035) that turns a ref into a live
title/status/summary chip. Both need a machine-findable id on each item.

**3. The v1 token shape is slow to type (new, ergonomic).** The founder types these
refs constantly and reports that **dashes and hashes slow him down**, the padding
zero is dead weight, `SPEC` is longer than it needs to be, and a 2-letter repo code
(`MS`) is too terse to read at a glance. `MS-SPEC-019-FR-3` — the v1-consistent way
to name one paragraph — is five tokens and three dashes. (RSI / keyboard-ergonomics
is a standing constraint here, not a nicety.)

## Decision

**Every referenceable item — approvable *and* the typed paragraphs/bullets/rows
inside it — has a short, slash-joined, zero-padding-free id of up to three
segments: `PROJECT / APPROVABLE / PARAGRAPH`. Segments elide left-to-right when
implied by context: drop `PROJECT` for an intra-project ref, drop `APPROVABLE` too
for an intra-document ref.**

```
Full (cross-project):     MIN/SP19/FR3      ← FR3 of minspec SPEC-19
Intra-project:            SP19/FR3          ← same repo, name the doc
Intra-document:           FR3               ← same doc, name only the item
Whole approvable:         MIN/SP19          ← the spec itself (2 segments)
Cross-project issue:      SCR/IS204         ← scrooge issue 204
```

### 1. Segment 1 — PROJECT (3 uppercase letters)

- **3 letters**, uppercase, unique. 2 was too terse to read; 3 is "meaningful" (the
  founder's word). **Default = first 3 letters of the repo name**, hand-overridable
  in the table for a nicer form.
- Seed: `MIN` (minspec), `SCR` (scrooge), `SEA` (sealbox), `MMO` (mmo-platform).
  *(v1 was `MS`/`SC`/`SB` — superseded.)*
- Elided for **intra-project** references.

### 2. Segment 2 — APPROVABLE (letter code + number, no pad)

- **Code:** `SP` spec · `DR` decision · `EP` epic · `PR` pull request · `IS` issue.
  (`SPEC`→`SP`, `EPIC`→`EP`: shorter; `DR`/`PR` already short.)
- **Number:** no leading zeros — `SP19`, `DR18`, `SP1` (not `SP019`/`SP01`).
- Elided for **intra-document** references (the item's home doc is the doc it's in).

### 3. Segment 3 — PARAGRAPH (type code + number, no pad)

Every *typed* item carries its type code + a per-document sequence number. **Proposed
type set (redline at Accept-ADR):**

| Paragraph / row type | Code | Today in the corpus |
|---|---|---|
| Functional requirement | `FR` | `FR` (kept) |
| Open question | `OQ` | `OQ` **and** `FR-OQ` — unified to `OQ`; an OQ that is *about* an FR is written `SP19/FR3` in its body, not a compound prefix |
| Risk (row of the Risks table) | `R` | `R1`…`Rn` (kept) |
| Acceptance criterion | `AC` | `AC` (kept) |
| Invariant | `INV` | `INV` (kept — iconic, exempt from the shorten rule) |
| Alternative considered | `AL` | usually unlabeled |
| Costly-to-refactor item | `CR` | `Costly #N` (unlabeled) |
| Consequence (± bullet) | `CQ` | unlabeled |
| Follow-up item | `FU` | unlabeled |
| Metric | `M` | `M1`…`Mn` (kept) |
| Goal (constitution / DR-039) | `G` | `G-1` (kept) |
| Resolved design decision | `RD` | `RD-1` (kept) |
| Methodology-fit deviation | `DV` | unlabeled (numbered rows) |
| Plain prose paragraph (untyped) | **OPEN — see OQ3** | none today |

Numbers restart per document, per type: `FR1, FR2, …`; `R1, R2, …`.

### 4. Grammar must stay machine-findable (for the trace graph + SPEC-035 renderer)

The scheme exists to be **regex-detected** in rendered markdown and turned into
links/lozenges. The token grammar is therefore fixed and anchored on the known
**code vocabulary** (project codes from the table; approvable codes `SP|DR|EP|PR|IS`;
paragraph codes from §3), roughly:

```
(PROJ/)?(APPR/)?PARA     |  (PROJ/)?APPR     |  PROJ
PROJ = one of the table's 3-letter codes
APPR = (SP|DR|EP|PR|IS)\d+
PARA = (FR|OQ|R|AC|INV|AL|CR|CQ|FU|M|G|RD|DV)\d+
```

Anchoring on a **closed vocabulary** (not "any slash-separated token") is what keeps
the detector from firing on real file paths / URLs — see **R1** and **OQ1**, the
central risk of the `/` joiner.

### 5. Not fail loud (unchanged from v1)

An unknown PROJECT code is **advisory**, never a hard error: the Tier-0 resolver
returns an `unknown-prefix` result; the Tier-1 seam may SUGGEST a code
(`suggestPrefixDeterministic`, now emitting 3 letters) and offer to add a table row.
Same for an unknown APPROVABLE/PARAGRAPH code — resolve to prose, optionally advise.

### 6. Tier split (unchanged from v1)

Read + resolve + format + deterministic-suggest is **Tier-0**
(`@aiclarity/shared` `project-prefix`, pure — no `fs`/`vscode`/network). The LLM
suggestion + toast + validate advisory + the SPEC-035 lozenge/hover render are the
**Tier-1** layer.

### 7. Elision is display-only; storage is canonical

A stored/emitted **cross-artifact** ref should be written in its **fullest unambiguous
form for its scope** (intra-doc `FR3`; intra-project `SP19/FR3`; cross-project
`MIN/SP19/FR3`). Elision never *removes* information the reader needs — it only drops
the segment the surrounding context already pins. The renderer may *display* a
shortened chip while the source carries the full form (parallels the SPEC-021 /
DR-032 egress rule: never leak an ambiguous internal ref).

## Consequences

- One uniform, typeable address for **any** referenceable item in the workspace —
  the precondition for the trace graph and SPEC-035's live lozenges/hover-cards.
- The inconsistent-prefix mess (`OQ` vs `FR-OQ`, unlabeled costly/consequence/
  follow-up rows) is resolved by a single type table.
- **Breaking vs v1** (all still `proposed`, nothing accepted): the shipped v1
  artifacts must be updated — `.minspec/project-prefixes.md` (2→3-letter), the
  CLAUDE.md *Cross-project references* convention text, and the `project-prefix.ts`
  module (new grammar: `/` joiner, 3-letter codes, approvable `SP|DR|EP|PR|IS`,
  paragraph third segment, no-pad numbers). The module change is a **tracked
  follow-up** (below), not done here.
- **Corpus migration is deferred and large.** Re-tokenising existing refs across all
  approvables (bare `FR-3`→`FR3`, `Costly #N`→`CR`n, adding ids to unlabeled
  consequence/follow-up/deviation rows) is a mechanical but wide sweep — a tracked
  follow-up, explicitly **not** applied this session (founder: record now, reformat
  later). Until it runs, old and new forms coexist; the resolver must read both.
- **GitHub-native `#N` vs `IS`/`PR` codes** now diverge (see OQ2): `MIN/IS500`
  no longer auto-links on GitHub the way `#500` does. A real trade recorded, not hidden.
- Orthogonal, as in v1, to within-repo id **uniqueness**
  ([#396](https://github.com/AIClarityAU/minspec/issues/396) duplicate-id gate) and
  to the egress guard that strips MinSpec's own internal refs from user output
  (DR-032 / SPEC-021).

## Alternatives considered

- **Keep v1 (`MS-SPEC-019`, dash-joined, cross-project only).** Rejected: does not
  reach the paragraph level (the actual need), and the founder rejects the ergonomics.
- **Globally-unique ids** (renumber into disjoint blocks). Rejected in v1 and still:
  breaks DR-027 separate registers, re-opens the cross-repo allocation race.
- **`.` or `:` as the joiner** (`MIN.SP19.FR3` / `MIN:SP19:FR3`). Live alternatives to
  `/` — both dodge the path/URL-collision risk `/` carries (R1/OQ1). `/` is the
  founder's stated preference for typing; the collision risk is mitigated by the
  closed-vocabulary regex (§4). Held open at **OQ1**.
- **Compound paragraph prefixes** (`FR-OQ` = an OQ about an FR). Rejected: that is
  what created the inconsistency; an OQ is one type (`OQ`), and its FR linkage is
  expressed by *referencing* the FR in its body (`SP19/FR3`), not by a compound code.
- **Address only typed items, leave prose un-addressable.** Held open at **OQ3** —
  the trace-graph vision ("ids on *all* paragraphs") wants prose addressable too,
  which needs a generic `P`-type; the cost is noise. Undecided.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | **`/` joiner collides with file paths / URLs** — the SPEC-035 detector could fire on `src/foo/bar` or a link, or miss a real ref. | High · Med | Anchor the regex on the **closed code vocabulary** (§4), not on "slashes": require a known 3-letter PROJECT and/or `SP\|DR\|EP\|PR\|IS`+digits and/or a known paragraph code. Still the central open risk → **OQ1** keeps `.`/`:` joiners live. |
| R2 | **Type-set churn** — 13 proposed codes, several net-new (`AL`/`CR`/`CQ`/`FU`/`DV`); authors won't adopt uniformly. | Med · Med | The set is redlined at Accept-ADR; the resolver treats an unknown paragraph code as advisory (§5), never a hard fail; migration tooling applies them mechanically. |
| R3 | **Migration is wide and lands during active authoring** — a mass re-tokenise races other in-flight spec edits. | Med · Med | Deferred to a dedicated follow-up run on a clean tree; resolver reads old+new during the interim so nothing breaks mid-migration. |
| R4 | **Elision reintroduces ambiguity** — a bare `FR3` copied out of its doc loses its home. | Med · Low | §7: storage is canonical/fullest-for-scope; elision is display-only; the egress guard (DR-032/SPEC-021) already resists leaking bare internal refs. |
| R5 | **`IS`/`PR` codes lose GitHub auto-linking** vs `#N`. | Med · Low | OQ2 decides whether GitHub refs keep `#N` (native) or take codes (uniform). Recorded trade. |

## Costly to Refactor

**Not undoable in <1 day — correctly a DR.** The expensive, hard-to-reverse parts:

- **The token grammar itself** once the corpus is migrated to it and the
  `project-prefix` module + SPEC-035 detector encode it — reverting means a second
  mass re-tokenise. (The migration is deferred precisely to keep this cheap *until*
  the scheme is accepted.)
- **The paragraph-type vocabulary** — once ids are stamped and cross-referenced, a
  code rename (`CR`→`CX`) ripples to every citing item and the trace graph.
- **Cheap to reverse (ADR-filter "yes" parts):** the 3-letter *project* codes in the
  table (one-row edits); the joiner choice *before* migration (OQ1); the SPEC-035
  render styling. These are same-day knobs, deliberately isolated from the grammar.

## Open Questions

- **OQ1 — joiner.** `/` (founder's typing preference, but path/URL-collision-prone,
  R1) vs `.` vs `:`. Decide **before** migration runs — it is cheap now, expensive after.
- **OQ2 — GitHub issue/PR form.** Uniform codes (`MIN/IS500`, `SCR/PR61`) vs keep
  GitHub-native `#500` intra-project (auto-links) and only code them when cross-project.
- **OQ3 — untyped prose.** Does the trace-graph "id on *every* paragraph" apply to
  plain prose (needs a generic `P`n type + auto-numbering), or only to the typed set
  in §3? Scope + noise trade.
- **OQ4 — number scope.** Per-document-per-type sequence (assumed above) vs a single
  per-document sequence across types. Per-type reads better; confirm.
- **OQ5 — `INV` / project-wide items.** Invariants and constitution goals are
  *project-wide*, not per-doc. How is a constitution item addressed — `MIN/INV3`
  (project + item, no approvable) or a reserved `CON` pseudo-approvable
  (`MIN/CON/INV3`)? Same question for the term glossary ([#672](https://github.com/AIClarityAU/minspec/issues/672)).
