# DR-053: Cross-project reference prefixes

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records the reference-scheme decision from [#58](https://github.com/AIClarityAU/minspec/issues/58)
> / [#500](https://github.com/AIClarityAU/minspec/issues/500) and ships its Tier-0 core.
> The Tier-1 advisory UI is tracked separately ([#614](https://github.com/AIClarityAU/minspec/issues/614)).

## Context

SDD ids are allocated **per repo** — [DR-027](DR-027.md) split scrooge into its own
private repo with its **own** local register, and every minspec-enabled repo numbers
`SPEC-`/`DR-`/`EPIC-` from 1. So `SPEC-001` exists in minspec **and** scrooge **and**
sealbox and means a different artifact in each. Within a repo that is unambiguous and
fine ([#58](https://github.com/AIClarityAU/minspec/issues/58) is a non-issue *inside* a
repo). The ambiguity bites only when a reference **leaves** its repo: a minspec DR that
cites a scrooge spec, a commit or issue body that points at another repo's issue. A bare
`SPEC-001` there is project-dependent, and any cross-project tooling (search, link
resolution, impact) can mis-resolve it.

Two rejected alternatives:

- **Globally-unique ids** (renumber scrooge into a disjoint block). Breaks DR-027's
  separate-register model, forces cross-repo coordination on every new id, and re-opens
  the exact allocation race the duplicate-id gate ([#396](https://github.com/AIClarityAU/minspec/issues/396))
  exists to close — but now across repos, where no single validator can see both.
- **Always repo-qualify** (`AIClarityAU/scroogellm#26`, full paths for specs). Correct but
  verbose, and there is no natural full-path form for an SDD id the way there is for a
  GitHub issue.

## Decision

**Per-repo-local ids stay unprefixed. A reference that spans projects carries the target
project's short prefix**, drawn from a committed mapping table.

- **SDD refs:** `<PREFIX>-<ID>` — `MS-SPEC-019`, `SC-DR-007`, `SB-EPIC-002`.
- **Issue / PR refs:** `<PREFIX>#<N>` — `MS#500`, `SC#26`. The prefix abuts the `#`
  (no dash); the SDD form needs a dash because the id starts with a letter, the issue form
  does not because `#` already separates. (The issue/PR half was added at the founder's
  request, 2026-07-10 — the same prefix scheme, one rule for both reference kinds.)
- **Mapping table:** `.minspec/project-prefixes.md`, a committed, human-editable pipe
  table (`| Prefix | Project | Repo |`). Prefixes are uppercase, 2 letters where possible,
  unique.
- **Not fail loud.** An unknown prefix is **advisory**, never a hard error: the
  deterministic core resolves it to a `'unknown-prefix'` result; the Tier-1 seam (assistant
  / agent-execute) may then SUGGEST a prefix and offer to add a row. `suggestPrefixDeterministic`
  is the offline Tier-0 default so the core still degrades gracefully with no LLM.
- **Tier split.** The read + resolve + format + deterministic-suggest is Tier-0
  (`@aiclarity/shared` `project-prefix` module — pure, no `fs`/`vscode`/network, mirroring
  `canonical.ts`). The LLM suggestion + toast + validate advisory are the Tier-1 layer.

## Consequences

- A cross-project reference is now unambiguous out of context; cross-project search / link
  resolution / impact can key off the prefix deterministically.
- One more committed file per workspace (`.minspec/project-prefixes.md`); adding a project
  is a one-row edit, reviewable in a diff.
- This is the *reference-display* axis. It is orthogonal to within-repo id **uniqueness**
  ([#396](https://github.com/AIClarityAU/minspec/issues/396) duplicate-id gate) and to the
  egress guard that strips MinSpec's **own** internal refs from user-facing output
  (DR-032 / SPEC-021) — those remain as-is.
- Shipped in this DR's PR: the Tier-0 module + tests, the seed table, and the
  CLAUDE.md *Cross-project references* convention. Not shipped here: the validate advisory
  + toast + init offer (Tier-1) — tracked in [#614](https://github.com/AIClarityAU/minspec/issues/614),
  deferred to keep the core PR focused (same resolver-before-UI split as SPEC-012).
