# DR-053: Paragraph-addressable reference scheme

> **Born / stays `proposed` per DR-029** — acceptance is a separate human act
> (*MinSpec: Accept ADR*). **v2**: widens the cross-project reference decision
> ([#58](https://github.com/AIClarityAU/minspec/issues/58) /
> [#500](https://github.com/AIClarityAU/minspec/issues/500)) to the **paragraph
> level** and re-shapes the token for typing ergonomics. The prefix table + the
> paragraph-**type** set are a **proposal to redline** at Accept-ADR. This body was
> **reworked after a 6-lens adversarial design review** (2026-07-12) — see the
> **Addendum** for what the review changed and the **⚑ review-recommended** markers
> where it revises the founder's original proposal. Nothing is applied to the corpus
> yet (**record now, reformat later**).

## Context

Two problems, one scheme.

**1. Cross-project id collision (v1 premise, unchanged).** SDD ids are allocated
**per repo** ([DR-027](DR-027.md)); `SPEC-001` exists in minspec **and** scrooge
**and** sealbox and means a different artifact in each. The ambiguity bites only when
a reference **leaves** its repo.

**2. Sub-document references are unscoped and inconsistently prefixed.** Approvables
constantly cite items *inside* other approvables — a risk row, an FR, an open
question — and today those are:

- **Unscoped.** A bare `FR-3` atop scrooge `DR-019` means "FR-3 of the doc the reader
  has in mind" — ambiguous out of context. There is no way to say "FR-3 *of* SP19."
- **Inconsistently prefixed.** The same *kind* of item carries different prefixes
  across docs: open questions as `OQ-n` **and** `FR-OQ-n`; costly-to-refactor items as
  unlabeled `Costly #N`; consequences, follow-ups, deviations, alternatives frequently
  **unlabeled**. Several *decision-family* schemes coexist with no lifecycle rule:
  `OQ-n` (open question), **`D-n` (decision item — live in SPEC-033 at HEAD, D-1..D-9)**,
  `CL-n` (clarification — SPEC-019, DR-046), `RD-n` (resolved design decision). *(An
  earlier draft of this DR wrongly claimed no `D-n` scheme existed — corrected: it does,
  and the founder's recollection was right. See Addendum.)*

There is **no uniform, addressable way to point at "an item in any approvable."** That
address is the precondition for a **trace graph** (item-level links, narrower than
[DR-038](DR-038.md)'s artifact-level graph) and the **lozenge/hover-card renderer**
([SPEC-035](../../specs/minspec/SPEC-035-approvable-ref-lozenges/requirements.md)).

**3. The v1 token shape is slow to type (ergonomic).** The founder types these refs
constantly: dashes/hashes and padding zeros slow him down, `SPEC` is over-long, a
2-letter code reads poorly. `MS-SPEC-019-FR-3` is five tokens, three dashes. (RSI is a
standing constraint here.) *(Review nuance: the real RSI cost is **shift-presses**, not
keystrokes — see §8 and the ergonomics finding.)*

## Decision

**Every referenceable item — approvable *and* the typed items/rows inside it — has a
short id of up to three segments: `PROJECT / APPROVABLE / PARAGRAPH`. Segments elide
left-to-right when context implies them.**

```
Full (cross-project):     MIN/SP19/FR3      ← FR3 of minspec SPEC-19
Intra-project:            SP19/FR3          ← same repo, name the doc
Intra-document:           FR3               ← same doc, name only the item
Whole approvable:         MIN/SP19          ← the spec itself
Cross-project issue:      SCR#204           ← scrooge issue 204 (hash-native, ⚑)
```

### 1. Segment 1 — PROJECT (3 uppercase letters)

- **3 letters**, uppercase for **render**, **case-insensitive for input** (⚑ §8):
  typing `min/sp19` renders `MIN/SP19`. Default = first 3 letters of the repo name.
- Seed: `MIN`/`SCR`/`SEA`/`MMO`. *(v1 `MS`/`SC`/`SB` superseded.)*
- Elided for **intra-project** refs. **A bare PROJECT alone is not a reference** (⚑ §4).

### 2. Segment 2 — APPROVABLE (letter code + number, no pad)

- **SDD codes:** `SP` spec · `DR` decision · `EP` epic, + number, no leading zeros
  (`SP19`, not `SP019`).
- **Issues / PRs keep GitHub-native `#N`** (⚑ review-recommended, reverses the original
  `IS`/`PR` proposal): `#500` intra-project (GitHub auto-links it; SPEC-035 already
  detects it; the shipped module emits `<PREFIX>#<N>`), and **`SCR#204`** cross-project
  (prefix abuts `#`). Coding issues as `IS500`/`PR412` was rejected: +keystrokes, loses
  GitHub autolinking in every non-webview surface, and adds an issue-vs-PR mislabel mode.

### 3. Segment 3 — PARAGRAPH (type code + number)

set (redline at Accept-ADR — the review expanded it against the real corpus):**

**Decision-family lifecycle (redline).** `OQ` (open) → when answered, the resolution is
recorded; whether it *becomes* a `D`/`RD` **keeping the same number** or stays `OQ-n`
marked resolved is **OQ4** — pick one so migration never mints two ids for one logical
item.

### 3.1 ID stability — a number is a **handle, not an ordinal** (DECIDED, ⚑ was undefined)

Ids are **allocate-once, monotonic per document per type, never reused**:

- **Delete** leaves a **gap** (`FR1, FR3`, no `FR2`) — the number is retired, never
  recycled.
- **Insert** takes the **next-highest free number**, regardless of textual position.
- **Reorder** never touches ids.

This is mandatory: sequential/positional numbering would silently renumber every item
after an edit, so a stored `SP19/FR3` would resolve to a *different* requirement with no
error — exactly the silent-lying signpost DR-003 / never-wrong exist to prevent, and the
reason [SPEC-014](../../specs/minspec/SPEC-014-review-webview/requirements.md) FR-OQ1
rejected positional ids for comment pins (see Alternatives). The
[#396](https://github.com/AIClarityAU/minspec/issues/396) uniqueness gate must tolerate
the gaps.

**Corroborated by the code-side trace-graph work ([#643](https://github.com/AIClarityAU/minspec/issues/643)).**
That issue's core lesson — *line/paragraph citations rot* (the #362 backfill's #627 found
**20 files** with stale line-citations) — is exactly why an approvable→code link there
anchors to a **symbol name + test** (resolved via CodeGraph), never a line. The same logic
applies one level up: a paragraph **id** here must be a stable *handle*, not a position.
This DR provides the **"standard linking nomenclature"** #643's design explicitly said it
would require; #643 (approvable↔**code**) and DR-038 (artifact-level graph) are its siblings
on the code and graph-viz axes (see Follow-ups).

### 4. Grammar — detection contract (⚑ the review's central fix)

The scheme exists to be **regex-detected** and turned into lozenges. The detector fires
on a **closed vocabulary** (project codes from the table; `SP|DR|EP`; the paragraph
codes), which reliably skips file paths, URLs, dates, and fractions. **But a *bare*
paragraph code (`R1`, `M1`, `G7`, `FR3`) collides catastrophically with prose** (corpus:
`R1`×51, `M1`×82; "Apple M1", "Windows SP1", "Lexus IS500", `G7`, "fly into SEA"). So:

- **Auto-lozenge only a ref carrying at least an APPROVABLE segment** — `SP19/FR3`,
  `MIN/SP19`, `DR53`, `#500`, `SCR#204`. A **bare** intra-doc `FR3`/`R1` is **authoring
  shorthand**: it stays human-readable and in-doc-navigable but is **not** auto-detected
  (avoids the prose flood). **⚑ Optional sigil** `[[FR3]]` may force-lozenge a bare ref
  where you want one. *(This reconciles §3.1/canonical storage with detectability — the
  review's blocker-A; the exact contract is redline-able but must be decided, not left an
  OQ.)*
- **Longest-match ordering** in any published grammar: `RD` before `R`, `NFR` before
  `FR`/`R`, etc. — `RD1` must not tokenise as `R`+junk (they co-occur, e.g. SPEC-013).
- **Invariants/goals are named**, not numbered: the grammar admits `INV-<slug>` and
  `G-<n>` (a `INV\d+`-only grammar false-negatives every real invariant, incl. this DR's
  consumer's own).
- **A bare PROJECT token alone (`MIN`, `SEA`) is not a reference** — the minimum
  machine-detected ref carries ≥ an approvable segment.

### 5. Not fail loud (unchanged). 
An unknown PROJECT / APPROVABLE / PARAGRAPH code is **advisory** — resolve to prose,
optionally suggest a table row; never a hard error, never a silent drop.

### 6. Tier split (unchanged). 
Read + resolve + format + deterministic-suggest is **Tier-0** (`@aiclarity/shared`
`project-prefix`, pure). LLM suggestion + toast + the SPEC-035 render are **Tier-1**.

### 7. Elision is display-only; storage is canonical. 
Store a cross-artifact ref in its **fullest unambiguous form for its scope**; elision
only drops the segment context already pins. Never leak an ambiguous internal ref
(parallels the DR-032/SPEC-021 egress rule — but note that guard is **not** orthogonal;
Consequences).

### 8. Ergonomics — the real RSI lever is case, not the joiner (⚑ review)

Keystroke-wise v2 wins (`SPEC-019-FR-3`→`SP19/FR3`, 16→12), but the **shift-press** win
(the actual RSI cost) is near-zero: `/` and `-` are both unshifted, so the joiner swap is
ergonomically neutral; the win comes from `SP`+de-padding+segment-merge. The **biggest
untapped lever — accept case-insensitive lowercase input, render canonical uppercase —**
zeroes every shift-cycle and is adopted (§1). The "3 letters is meaningful" rationale is a
**render/disambiguation** choice, not a typing one.

## Consequences

- One uniform, typeable, **machine-detectable** address for any referenceable item —
  the precondition for the trace graph + SPEC-035.
- The inconsistent-prefix mess is resolved by one type table + a decision-family
  lifecycle rule.
- **The egress guard is NOT orthogonal (⚑ review).** DR-032/SPEC-021's stripper that
  removes MinSpec's own internal refs from user output is a **ref detector of the same
  class**. If only `project-prefix` and SPEC-035 learn v2, new-form internal refs
  (`SP19`, `MIN/SP19`) slip past the egress guard into user product output — a
  never-wrong-class leak. → tracked follow-up to teach the egress detector v2.
- **Breaking vs v1** (all `proposed`): update `.minspec/project-prefixes.md` (done, 3-letter),
  the CLAUDE.md convention (done), and the `project-prefix` module (follow-up: `/` grammar,
  3-letter codes, `SP|DR|EP`, paragraph segment, named-invariant forms, dual-read).
- **Interim-state caveat (⚑ review).** The seed table is now 3-letter while the module
  still emits the v1 dash join — so `formatCrossRef` would emit `MIN-SPEC-019`, a form in
  **neither** register. This is **latent only** (no runtime loader wires the table today);
  the module update (#679) is a **hard predecessor** to any runtime use, and must add a
  version-aware gate (dash⇒2-letter, slash⇒3-letter) so the chimera is rejected.
- **Corpus migration deferred + large** — a wide sweep, run post-acceptance on a clean
  tree (follow-up). Old/new coexist until then; the resolver must read both.

## Alternatives considered

- **Keep v1** (cross-project only, dash-joined). Rejected: doesn't reach the paragraph
  level; ergonomics rejected.
- **Sequential/positional paragraph numbers.** Rejected (§3.1): silent renumber-on-edit;
  the never-wrong violation SPEC-014 FR-OQ1 already rejected for pins. We choose an
  immutable handle rather than SPEC-014's content-re-anchoring because an explicit stamped
  id survives the *machine* trace-graph use better than fuzzy re-match — but it inherits
  SPEC-014's lesson that position ≠ identity.
- **Code issues/PRs as `IS`/`PR`.** Rejected (§2): loses GitHub autolinking, +keystrokes,
  mislabel mode; keep `#N`.
- **Auto-lozenge bare paragraph codes.** Rejected (§4): prose-collision flood.
- **`.`/`:` joiner instead of `/`.** Moot for detection (closed-vocab regex handles
  paths/URLs) and shift-neutral for typing; `/` kept. OQ1 downgraded from "central risk"
  to aesthetic.
- **Globally-unique ids.** Rejected (v1): breaks DR-027 separate registers.

## Costly to Refactor

- **The detection contract (§4) + id-stability policy (§3.1)** — shared with the module,
  the egress guard, and the trace graph; changing them post-migration is a re-tokenise.
- **The paragraph-type vocabulary** — once ids are stamped and cross-referenced, a code
  rename ripples to every citing item.
- **Cheap to reverse:** the 3-letter *project* codes (one-row edits); the `/` joiner
  *before* migration; the SPEC-035 render styling; the case-insensitive input toggle.

## Addendum — 6-lens adversarial design review (2026-07-12)

The v2 first draft was reviewed by 6 independent lenses (regex/path-collision, code
collision + completeness, id-stability, migration/back-compat, ergonomics/RSI, dogfood/
consistency) + synthesis. Verdict: *rework before human redline*. What it changed:

1. **Factual correction (evidence discipline).** The draft asserted "no `D-n` scheme
   exists — false memory." **Wrong** — `D-1..D-9` are committed in SPEC-033 at HEAD; the
   founder's memory was right. Corrected in Context; `D`/`CL` added to §3.
2. **Detection contract (§4, blocker).** Bare paragraph codes (`M1`×82, `R1`×51 in prose)
   would flood false positives — the closed-vocab anchor only defends `/`-paths. Now:
   auto-lozenge only refs with ≥ an approvable segment; bare = shorthand; opt-in sigil.
   OQ1 (joiner) downgraded — it was the *wrong* worry.
3. **Id stability (§3.1, blocker).** Was undefined → could silently renumber. Now a
   decided allocate-once monotonic-handle policy, reconciled with SPEC-014.
4. **Issue form (§2, OQ2 resolved).** `IS`/`PR` codes → back to GitHub-native `#N` /
   `SCR#204`.
5. **Type set expanded** against the real corpus (`NFR AS CT DP TV FM D CL`) + longest-
   match + named-invariant grammar.
6. **Egress guard not orthogonal** (Consequences + new follow-up) — prevents a
   never-wrong-class internal-ref leak into user output.
7. **Ergonomics (§8):** the RSI win is case-insensitive input + uppercase render, not the
   `/` joiner. Adopted.
8. **Interim chimera** (`MIN-SPEC-019`) flagged; module-update #679 made a hard predecessor.
9. **SPEC-035** amended: FR4 `summary:` ↔ DR-012 hash interaction (canonical.ts strips only
   status/phases, so a regenerated summary voids approval — resolve in OQ1), FR7 degrade,
   FR9 named invariants, negative ACs.
