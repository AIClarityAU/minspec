# DR-057: Generated approvables take the review lane; the drain actions phase-advance by enqueue

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records context + decision + follow-ups; no code ships with this DR. Extends
> [DR-051](DR-051.md) §4(b) (trivial-vs-review-needing branch policy) with a
> **provenance** axis it does not yet name. Builds on [DR-051] and [DR-047](DR-047.md),
> both still `proposed` — see Risks.

## Context

The background piggyback loop (`scripts/drain-inbox.sh`, fired from the session-start
hook) has ONE input source: **GitHub issues** (`inbox` → triage → `agent-ready` →
`dispatch-issue.sh` → worktree → PR). It never reads `.minspec/approvals/` sidecars or
spec `status:` frontmatter. Meanwhile SDD phase-advance (specify→plan→tasks→implement)
lives entirely in the VS Code extension as a human Command-Palette action. So an
**approved plan stalled with no `tasks.md`** just sits there — nothing in the background
loop notices or generates the next phase. The founder asked to close that gap: let the
drain **action** approved-doc phase-advance (primarily plan→tasks), and weighed three
mechanisms — an Alt-A "advance?" toast, a CC-session job, and/or **excluding the detailed
generated docs from DR-051's main-direct lane** so the drain's worktree→PR pipeline can
process them ("tasks.md and the detailed specs are too detailed for me to ever read, but
other devs might want to").

Three facts constrain the answer (each verified against the code this session):

1. **`ai-review.yml` fires ONLY on `pull_request`** (`.github/workflows/ai-review.yml`).
   A generated `tasks.md` landed **direct on `main`** therefore gets the structural CI
   checks plus a single human keystroke — and the founder never reads `tasks.md`, so that
   keystroke is inherently a rubber-stamp. That is exactly the [DR-047](DR-047.md) /
   #344-349 failure surface, reopened for the artifact class that MOST needs a second
   adversarial read. **The only built independent reviewer is PR-gated.**
2. **The MinSpec free extension is Tier-0 / air-gapped — no LLM** ([DR-004](DR-004.md),
   [DR-015](DR-015.md)). It cannot run `/tasks`. LLM generation legitimately runs in a
   **CC-session drain consumer** (`scripts/`) or the separate Tier-1 **agent-execute**
   extension — never in the free extension.
3. **DR-051 §4(b) already routes review-needing approvables through worktree→PR** and
   reserves main-direct for *trivial* edits. It just does not yet name that
   **machine-generated == review-needing**. The `spec-gate` never blocks a `.md` write, so
   generating a next-phase doc is not gate-blocked; phase-status governs implement.

## Decision

**Route approvables by PROVENANCE (who authored them), and make the drain a
detect-and-enqueue producer whose LLM generation runs outside the Tier-0 core.** Five
coupled rulings:

### 1. Generated next-phase derivatives take the review (worktree→PR) lane

- **`tasks.md`** (always) and **`design.md`** (opt-in, §4) — LLM-generated next-phase
  derivatives — are **generated in a worktree and land via a review branch → PR**, so
  `ai-review.yml` fires and gives the doc an independent adversarial read BEFORE any human
  sign-off. They are **review-needing by construction**, not by per-edit judgment.
- **Human-authored approvables** (`requirements.md`, DRs, constitution, epics, domain)
  **stay main-direct** per DR-051 §1 — read-in-place, IDE-visible, low ceremony, where a
  human actually authors/reviews. This DR does not change that.
- Provenance is **not path-encoded** (`tasks.md` and `requirements.md` both live under
  `specs/**`), so the classifier keys off **MinSpec's fixed filename convention**
  (`tasks.md`/`design.md` = generated). This is the one new distinction the DR-051 corpus
  constant must learn.

### 2. The drain DETECTS + ENQUEUES; generation runs in a CC-session/dispatch consumer

- The **Tier-0 extension** and the **drain sweep** only DETECT a spec ready to advance and
  **write an LLM-free request** to a local, gitignored **`.minspec/queue/`**. This write is
  identical in nature to the existing `recordApproval` sidecar write — so the never-call-an-
  LLM invariant holds **by construction**, not by discipline.
- The **CC-session drain consumer** (reusing `dispatch-issue.sh`'s credential-free worktree
  pipeline) or **agent-execute** dequeues and runs the generation. It NEVER runs in the free
  extension (Fact 2).

### 3. Two triggers, one request: Alt-A toast + drain-sweep backstop

- A non-modal **Alt-A follow-up toast** ("Advance to next phase", with an **"Always"**
  button = a global `minspec.advancePhaseOnApprove` pref, default off) enqueues at the
  approval moment.
- A **background drain-sweep** (a deterministic scan reusing the **SPEC-012 next-task
  resolver**, honoring DAG priority — sibling to #420) enqueues the SAME request for
  approvals made via direct git or after "Always" was set, so nothing slips un-actioned.

### 4. `design.md` lane is a per-repo pref — DEFAULT authored, PROMOTE-ONLY

`design.md` is provenance-ambiguous (LLM-drafted via `/plan`, yet carries human design
intent). It **defaults to authored / main-direct**. A **per-repo** setting, surfaced as a
**demotion banner in the spec webview** (hidden once acted on), may **promote it into the
review lane**. This pref is **one-directional-safe: it may only move a doc toward MORE
review, never route a generated doc to main-direct.** A free lane-selector that could send
generated docs main-direct is **rejected** — it is a knob that can silently void the only
independent reviewer (the DR-047 surface).

### 5. Approval ground truth is unchanged; the lane only relocates authoring

The [SPEC-022] committed, hash-matched approval sidecar remains the sole truth. This DR
only relocates **where `tasks.md`/`design.md` are authored** (a worktree PR vs `main`), and
the **terminal**: the generated-doc PR **auto-merges on `ai-review:pass`** → lands
`status:specifying` (un-gated, editable-in-place) → the human's **Alt-A afterward** flips
status and writes the sidecar via the existing commit-on-approve. Two gates in series:
independent AI review, then human sign-off. **Implement stays blocked by phase-status (not
spec-gate, which never blocks `.md`) until that sidecar exists.**

## Rationale

The disqualifying fact settles it: the only BUILT independent reviewer, `ai-review.yml`,
is `pull_request`-only. Any lane that lands an LLM-generated `tasks.md` direct on `main`
therefore hands the artifact-class-that-most-needs-a-second-read to a lone rubber-stamp
keystroke on a doc the founder never reads — the DR-047/#344-349 hole, reopened. Routing
generated docs through worktree→PR is not a strain against DR-051; it is DR-051 §4(b)'s
OWN endorsed path ("a review-needing approvable is authored in a worktree on a review
branch → PR — literally this DR's own workflow"). The only extension is making
"machine-generated == review-needing" **structural** rather than a per-edit judgment call.
Keeping generation OUT of the free extension preserves the Tier-0 air-gap by construction.
And generated derivatives **mint no new `SPEC-NNN`**, so branching them carries **zero** of
the id-collision risk DR-051 §3 worries about for main-direct new specs — the provenance
cut is cleaner on numbering, not messier.

## Consequences

**Positive:**
- The highest-risk approvable class (LLM-authored, no human wrote it, founder never reads
  it) finally gets the independent AI review — closing the DR-047 rubber-stamp surface for
  exactly the case it was most open.
- The founder never has to read `tasks.md`: the bot reviews what he won't, and his one
  human act (Alt-A) still gates implement.
- Detection stops being dead weight — the enqueue → consumer path actions it.
- Air-gap and concurrency stay clean; approval truth (SPEC-022) is untouched.

**Negative:**
- A net-new **queue primitive** + **doc-phase role** + **drain consumer** to build (the
  worktree/PR pipeline itself is near-total reuse).
- The **provenance distinction** (filename-keyed) must be taught to the DR-051 corpus
  constant and any consumer that keys off it (pre-commit main-pin gate, SPEC-026
  worktree-steer, the #575 bypass hook) — a multi-guard touch to bound carefully.
- A **post-merge window**: an AI-reviewed but human-UNapproved `tasks.md` sits on `main` at
  `status:specifying` before Alt-A. Implement must stay phase-status-blocked until sign-off.
- Async decouple depends on a **live consumer**; with none running, enqueued requests rot
  silently unless a queue-depth indicator surfaces them (#737).

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| A generated doc silently skips the only independent reviewer | A lane pref routes generated `tasks.md`/`design.md` to main-direct | Ruling §4 makes the pref **promote-only** (toward more review); `tasks.md` is never configurable; a free lane-selector is explicitly rejected. |
| Provenance distinction widens blast radius | Filename-keyed cut must be learned by every corpus-keyed guard | Pin it in the ONE named corpus constant DR-051 §"Costly to Refactor" already isolates; enumerate the guards in Specify. |
| Implement starts on an un-approved generated `tasks.md` | Post-merge window at `status:specifying` before Alt-A | Implement is gated by **phase-status**, not spec-gate (which never blocks `.md`); block until the sidecar exists (#735). |
| Enqueued phase-advance rots unseen | No live drain/agent-execute consumer | Queue-depth visibility indicator, keyboard-reachable (#737). |
| Stale generated doc after an upstream edit | Human edits `requirements.md` after `tasks.md` approved | Dedup/detector keys on `spec-path + upstream specHash`; a changed upstream hash re-enqueues regeneration (#738). |
| Builds on unratified policy | DR-051 and DR-047 are both `status:proposed` | Called out here; if either changes under this DR, the provenance cut needs rework (accepted, tracked). |

## Alternatives Considered

- **Generated docs land direct on `main` (a distinct main-direct dispatch lane).**
  Rejected: skips `ai-review.yml` (PR-only) → zero independent review for the
  most-needs-review class; also leans on a standing admin main-write credential. Cheapest
  to build, structurally unsafe for machine-authored content.
- **A per-repo lane-selector pref (main-direct vs PR) for generated docs.** Rejected as a
  free knob: a wrong value silently voids the only independent reviewer. Survives ONLY as
  the **promote-only** `design.md` variant in §4.
- **Alt-A toast runs `/tasks` inline.** Rejected: puts an LLM inside the Tier-0 free
  extension (breaks the air-gap) and blocks the UI thread on generation. The toast may
  only **enqueue**.
- **Toast-only trigger (no drain sweep).** Rejected: misses approvals made outside the IDE
  (direct git) or after "Always" was set. The sweep is the backstop.

## Costly to Refactor

Medium. Approval truth (SPEC-022 sidecar+hash) is **unchanged** — no data migration. The
provenance distinction is a filename predicate added to one named corpus constant; the
queue primitive + doc-phase role + consumer are net-new but bounded (the worktree/PR
pipeline is reused). Reversing the policy is a convention + scope change, not a data
migration. The genuinely costly mistake to avoid is shipping the main-direct-for-generated
lane and only later discovering generated docs went un-reviewed — which is why this is a
recorded decision.
