## Context

MinSpec and ScroogeLLM ship as VS Code extensions to a public marketplace. A supply-chain compromise in any bundled dependency — a malicious post-install in a transitive npm package, a hijacked publisher account, or a typosquat slipping into `package-lock.json` — would propagate directly to every install. Recent npm worms (shai-hulud, polyfill.io takeover, etc.) demonstrate this is an active threat for any VS Code extension publisher.

The existing pre-commit hook (`check-pii.sh` pattern from `mmo-platform`) catches secrets but does nothing for supply-chain exposure. `npm audit` covers CVEs but misses package-name takeovers and is noisy. We needed a check that:

1. Runs read-only (no `npm ls`, no source-file reads, no network calls in the scanner).
2. Produces a deterministic NDJSON inventory we can diff over time.
3. Cross-references the inventory against an exposure catalog the moment an advisory drops.
4. Blocks `vsce package` / `vsce publish` when a match is found, without slowing day-to-day work.

## Decision

Add a pre-publish supply-chain gate built on `perplexityai/bumblebee` v0.1.1+.

### Components

- **`packages/minspec/scripts/check-supply-chain.sh`** (single repo, strict, blocking) — wired into `package.json` as `prepackage` and `prepublish` lifecycle scripts. Scans the whole MinSpec monorepo via `bumblebee scan --profile project`, fails the build if any record_type=finding emitted.
- **`~/code/mmo-platform/scripts/bumblebee-scan.sh`** (cross-project, non-blocking) — iterates every `~/code/*` repo with a recognised manifest, dumps NDJSON per repo into `~/.cache/bumblebee/scans/<date>/`. Intended for nightly cron / on-demand advisory response.
- **Exposure catalog dir: `~/.cache/bumblebee/catalogs/*.json`** — schema_version 0.1.0 catalogs. Empty by default = inventory-only mode. When an advisory drops, drop the JSON in the catalog dir and re-run the scan; matches show up as findings.
- **`SKIP_SUPPLY_CHAIN_CHECK=1`** env bypass for emergency cuts only. Use sparingly; the bypass itself is logged to stderr.

### Catalog update mechanism (closes the "empty catalog" gap)

The open question in the original decision — "presumably a separate process" — is
resolved as follows:

- **`~/code/mmo-platform/scripts/update-bumblebee-catalog.sh`** — uses bumblebee's
  own upstream maintainer-side tool, `osvcatalog` (added upstream in v0.1.2,
  `tools/osvcatalog/main.go`), rather than a hand-rolled OSV fetch. Flow: shallow
  `git pull` (or clone) of [`ossf/malicious-packages`](https://github.com/ossf/malicious-packages)
  into `~/.cache/bumblebee/sources/malicious-packages/`, then
  `osvcatalog -o ~/.cache/bumblebee/catalogs/osv-malicious.json osv/malicious/`.
  `osvcatalog` emits only malicious-package (`MAL-`) records at `severity: critical`,
  already shaped to `exposure-catalog.schema.json` v0.1.0 — no bespoke mapping code
  to maintain. (CVE-style vulnerability advisories, as opposed to known-malicious
  packages, stay out of scope — same boundary bumblebee itself draws; `npm audit`
  remains the complementary CVE check per Alternatives Considered.) Idempotent —
  re-running with no new upstream commits is a no-op diff.
- **Cadence: daily cron**, `systemctl --user` timer (`bumblebee-catalog-update.timer`
  → `.service` runs `update-bumblebee-catalog.sh`), not baked into the per-package
  `check-supply-chain.sh` call — the gate script stays read-only/network-free per
  Scope boundaries; only the separate cron path touches the network.
- Cron run logs to `~/.cache/bumblebee/catalog-update.log`; a failed `git pull` or
  `osvcatalog` run does **not** delete the existing catalog (fail-open on the
  fetch, fail-closed on the gate — stale catalog beats no catalog).
- **Bumblebee install pinned**: `check-supply-chain.sh`, `bumblebee-scan.sh`, and
  the new `update-bumblebee-catalog.sh` now `go install .../cmd/bumblebee@v0.1.2`
  (and `.../tools/osvcatalog@v0.1.2`) instead of `@latest`. The Go toolchain
  verifies the module against the public checksum database (`GOSUMDB=sum.golang.org`,
  on by default) before it's cached, so the pin is checksum-backed without a
  separate manual verification step. Bumping the pin is a manual, reviewed step —
  not automatic — same as any other dependency version bump.
- Any catalog entry can still be added by hand (drop JSON in the same dir) for a
  same-day disclosure not yet in `ossf/malicious-packages` — the manual path from
  the original decision remains available, cron is additive not exclusive.
- **Reported built and verified 2026-07-06** (external to this repo — the
  scripts, cache, and counts below live in `mmo-platform`/`~/.cache`, not in
  MinSpecPro, so this bullet is not independently verifiable from here):
  `update-bumblebee-catalog.sh` self-installs
  `osvcatalog` pinned to v0.1.2, clones `ossf/malicious-packages`, and produced a
  first catalog of 23,736 malicious-package entries (npm 17,688 / pypi 5,056 /
  rubygems 972 / editor-extension 19 / packagist 1). Re-run is a verified no-op
  when upstream hasn't moved. `check-supply-chain.sh` scanned this repo against
  that catalog end-to-end: 0 findings, 32,581 packages catalogued. Unit files at
  `mmo-platform/scripts/systemd/bumblebee-catalog-update.{service,timer}`; a
  transient `systemd-run --user` timer is active now (see that dir's README —
  transient because this dev container doesn't mount `~/.config/systemd/user`,
  so it won't survive a user-manager restart; install the file-based unit on a
  host where that path is writable for a durable schedule).

### Scope boundaries

- Bumblebee is package-presence detection, **not** an EDR / IOC feed. It does not detect compromised binaries by hash, runtime behavior, or network IOCs. That layer is separate.
- v0.1 of bumblebee matches on exact `(ecosystem, normalized_name, version)`. Version-range / fuzzy matching is not yet supported upstream.
- Tier 0 invariant (DR-004) is preserved: bumblebee is a build-time / dev-tooling concern, not a runtime extension dependency. Nothing here changes the shipped `.vsix` payload.

## Rationale

- Bumblebee is Go-static-binary, stdlib-only, read-only — minimal supply-chain risk in the tool itself.
- 300ms scan time on MinSpec (2,300 packages catalogued) is well under any reasonable build-step budget.
- NDJSON inventory means catalogs can be applied retroactively: when an advisory drops next month against a package that's in today's `.vsix`, we know immediately by re-scanning the existing artifact without rebuilding.
- Same tool covers all `~/code/*` projects — supply-chain hygiene compounds across the portfolio rather than being a per-repo bespoke check.

## Costly to Refactor

ADR filter — **not** undoable in <1 day. Two adopted commitments raise the blast radius:

- **Dependency on bumblebee's `(ecosystem, normalized_name, version)` match model + catalog schema** (`exposure-catalog.schema.json` v0.1.0; the script greps `"record_type":"finding"` in NDJSON — `check-supply-chain.sh:59`) — switching scanners rewrites both `scripts/check-supply-chain.sh` and the cross-project `mmo-platform/scripts/bumblebee-scan.sh`, plus any accumulated catalog JSON in `~/.cache/bumblebee/catalogs/`.
- **`prepackage`/`prepublish` lifecycle wiring + `SKIP_SUPPLY_CHAIN_CHECK` bypass** — the integration seam at `packages/minspec/package.json:475-478` (`supply-chain:check` → `../../scripts/check-supply-chain.sh`); moving it touches the release flow for both MinSpec and (later) ScroogeLLM.

The `SKIP_SUPPLY_CHAIN_CHECK=1` env var (`check-supply-chain.sh:18`) keeps an emergency exit so a bad gate can't hard-block a release, but unwinding the scanner/catalog dependency portfolio-wide is multi-repo work, not a same-day revert.

## Consequences

**Positive:**

- NDJSON inventory means catalogs can be applied retroactively: when an advisory drops next month against a package in today's `.vsix`, re-scanning the existing artifact (no rebuild) surfaces the finding immediately.
- `.cache/supply-chain/` is gitignored at repo root, so per-scan NDJSON never pollutes the tree.
- Same `bumblebee` binary covers every `~/code/*` repo via `mmo-platform/scripts/bumblebee-scan.sh`, so supply-chain hygiene compounds across the portfolio instead of being a per-repo bespoke check.
- The `auto-publish vsix` standing rule (feedback_auto_publish memory) is now strict-mode gated through the `prepackage`/`prepublish` hooks — a compromised dependency blocks the auto-publish before the `.vsix` reaches the marketplace.

**Negative:**

- `npm run package` and `npm run publish` in `packages/minspec` now require `bumblebee` to be on disk; the script self-installs via `go install` on first run, which needs Go 1.25+ present — a new toolchain prerequisite on the release machine.
- An open follow-up remains: the cross-project scan (`bumblebee-scan.sh`) is not yet wired to a daily cron — it fires nightly only once an advisory feed is identified (see Follow-ups).
- ScroogeLLM (Tier 2, future Specify phase) does not yet have the hook; it must be added when that package is scaffolded, so the gate is not portfolio-complete on day one.

## Alternatives Considered

- **`npm audit` alone** — rejected as the gate. It covers published CVEs but, per Context, misses package-name takeovers/typosquats and is noisy; it has no deterministic NDJSON inventory to diff over time or to re-scan an existing `.vsix` against a later advisory. Kept as a complementary check, not the supply-chain gate.
- **Extending the existing `check-pii.sh` pre-commit hook** — rejected. Per Context it catches secrets but does nothing for supply-chain exposure, and a commit-time hook can't gate `vsce package`/`vsce publish`; the `prepackage`/`prepublish` lifecycle seam is the correct boundary (DR-004 Tier 1, dev-tool only).
- **A full EDR / IOC feed** — rejected for v0.1 as disproportionate to a two-extension publisher and outside the read-only/no-network constraint; bumblebee's package-presence model meets the stated need at 300ms/2,300 packages (Rationale) without that operational weight.

## Amendment (2026-07-11): trust the vendor + use `@latest`; un-pin the scanner

Reconsidered (founder). Bumblebee is a **dev-time-only** tool — it runs at
`package`/`publish` + in CI, **never shipped in the `.vsix`** — the same trust class
as every other dev tool we run at `@latest` (eslint, tsc, vitest, gh). Under that
already-accepted **vendor-trust** posture, **pinning a security *scanner* is
net-negative**: it freezes malicious-package detections + fixes on a stale version
(worse coverage — the opposite of what a scanner should do), while the vendor-trust
exposure is identical either way, and it breeds false "pinned = safe" confidence.
GOSUMDB already gives tamper-evidence against retroactive content swaps regardless of
the tag.

Therefore `scripts/check-supply-chain.sh` is un-pinned back to `@latest` (the two CI
workflows were already `@latest`). The earlier Risks-table row *"Closed 2026-07-06:
pinned to v0.1.2"* and the matching Follow-up **overclaimed a reproducibility change
as a security closure** — corrected here: the actual control is vendor-trust +
not-shipped + GOSUMDB, not the pin. If we ever move OFF vendor-trust, the lever is
**auditing each version** (a pin would then be the enabler for that) — not the current
posture. (mmo-platform's `bumblebee-scan.sh` / `update-bumblebee-catalog.sh` carry the
same `v0.1.2` pin — un-pin them there too, separate repo.)
